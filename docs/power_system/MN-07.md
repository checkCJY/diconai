## 2. 요구사항 정의서

### MN-07-01 · MN-07-02 · MN-07-03 · MN-07-04 종합

#### 2.1 임계치 기준 (Phase A 고정값)

| 단계 | 범위 (W) | 차트 배경색 | 텍스트 색상 |
|:--|:--|:--|:--|
| 안전 | 0 ~ 2,200 | 없음 | `--safe (#3fb950)` |
| 주의 | 2,200 ~ 2,860 | 황색 반투명 `rgba(227,179,65,0.20)` | `--caution (#e3b341)` |
| 위험 | 2,860 W 이상 | 적색 반투명 `rgba(248,81,73,0.20)` | `--danger (#f85149)` |

> 위험 하한 = 주의 하한 × 1.3 = 2,200 × 1.3 = **2,860 W**
> Phase B: 서버 페이로드에 `threshold_warning_w`, `threshold_danger_w` 키 추가 시 동적 전환

#### 2.2 상태 판단 기준
- 클라이언트에서 `getStatus(watt)` 함수로 THRESHOLD 기준 재계산
- 서버의 `risk_level` 필드를 **신뢰하지 않음** (서버·클라이언트 임계치 불일치 가능성)

#### 2.3 데이터 Staleness (신선도) 처리
- FastAPI: `power_latest["updated_at"]` 기준 **8초** 이상 경과 시 `data_stale = True`
- data_stale 시 `equipment = []`, `power_loading: True` 페이로드 전송
- 더미 발신기 주기 3초 기준 2.5 사이클 대기

#### 2.4 설비 테이블 컬럼 정의

| 컬럼 | 데이터 소스 | 산출 방식 |
|:--|:--|:--|
| 설비명 | `eq.name` | 직접 표시 |
| 전력량 | `eq.watt` | `watt / 1000` → kW (소수 1자리) |
| 부하율 | `eq.watt` | `watt / THRESHOLD.danger × 100` (%) |
| 사용 상태 | `eq.onoff`, `eq.sensor_status` | ON / OFF / 수신 오류 |
| 위험도 | `eq.watt` + `THRESHOLD` | 위험 / 주의 / 정상 뱃지 |

## 8. 생성/처리 조건

### 8.1 차트 Y축 동적 스케일
- 기본: `THRESHOLD.maxY = 3500 W`
- 수신 equipment 중 최대 watt > 3500 W 이면: `Math.ceil(maxWatt × 1.1 / 100) × 100` 으로 동적 확장
- 임계치 annotation 라인은 항상 고정값(2200/2860 W) 유지

### 8.2 data_stale 판단 조건
```python
# fastapi-server/websocket_CJY.py
DATA_STALE_SEC = 8
data_age_sec = (datetime.now(timezone.utc) - last_dt).total_seconds()
data_stale = (data_age_sec is None) or (data_age_sec > DATA_STALE_SEC)
```
- stale 시: `equipment = []`, `power_loading: True` 페이로드 전송

### 8.3 위험도 판단 우선순위
1. 클라이언트 `getStatus(watt)` — THRESHOLD 기준 (최우선)
2. 서버 `risk_level` — 참고용으로만 보존, UI 색상에는 미사용

### 8.4 ON/OFF/수신오류 분기
```javascript
isComm → "수신 오류" (danger badge)
eq.onoff === true  → "ON"  (safe badge)
eq.onoff === false → "OFF" (gray badge)
```

### 8.5 AI 탭 (미연동 상태)
- `_lastEquipCache` 에 마지막 수신 실시간 데이터 캐싱
- AI 탭 클릭 시 캐시 데이터로 그리드 재렌더 + 공지 배너 표시
- **4차 프로젝트에서 AI 모델 API 연동으로 교체 예정**

### 10.5 개선사항 항목

| 구분 | 내용 | 비고 |
|:--|:--|:--|
| 🔵 데이터 | `value == -1` 저장만 하는 중 → 특정 장비가 **3회 이상 연속 -1** 을 보낼 경우 이벤트(알람) 생성 로직 추가 필요 | DRF events 앱 연동 |
| 🔵 아키텍처 | 임계치 계산을 DRF Serializer 가 아닌 **FastAPI 수신 즉시** 계산하여 `risk_level` 결정 후 전달 → 서버 부하 분산에 유리 | Phase B 전환 시 함께 적용 권장 |
| 🔵 UX | 부하율(%) 계산 기준이 현재 `watt / THRESHOLD.danger × 100` (위험 임계치 대비) → 정확한 부하율은 **설비 정격 용량 대비** 이므로, 설비별 rated_capacity 필드를 페이로드에 추가 필요 | 설비 마스터 데이터 연동 후 교체 |
| 🔵 성능 | 1초 주기로 `grid.innerHTML = ''` 전체 초기화 후 재생성 → 대규모 설비 수 증가 시 렌더링 버벅임 발생 가능 → DOM diff 방식 또는 카드별 부분 업데이트 구조로 개선 | 차트 destroy 이슈와 함께 개선 |
| 🔵 유지보수 | `ui-exception.js` 가 `static/js/detail/` 에 위치 → 공통 로직이므로 팀 머지 완료 후 `static/js/refactors/` 로 이동, 타 페이지에도 적용 | 팀 머지 후 진행 |
| 🔵 환경 | WebSocket URL 하드코딩 (`127.0.0.1:8002`) → `{% if debug %}` 분기 또는 Django settings 기반 context variable 주입 방식으로 전환 | 운영 배포 전 필수 |
