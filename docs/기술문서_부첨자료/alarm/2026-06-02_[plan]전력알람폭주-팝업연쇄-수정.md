## Context — 왜 하는가

전력 DANGER/WARNING 알람이 폭주하고, 팝업 "확인 완료"를 누르면 다음 알람이 0초·비프음과 함께 연쇄로 뜬다 (팀원 피드백). 진단 결과 **한 뿌리**:

- **(A) 백엔드 churn:** `fire_power_clear_task` 가 `auto_resolve_active_events(event_type_prefix="power", power_device_id)` — **채널 무관, 디바이스 전체** Event 를 RESOLVE ([tasks.py:638-640](../../drf-server/apps/alerts/tasks.py#L638-L640)). 한 채널이 잠깐 정상 복귀하면 디바이스 공유 Event 가 닫히고, 다음 위험이 **새 event_id** 로 생성돼 프론트 60초 dedup(event_id 키)을 통과 → 폭주. **가스는 `cleared_gases` 로 이미 고친 race** ([tasks.py:425-429](../../drf-server/apps/alerts/tasks.py#L425-L429)) 인데 전력만 무방비.
- **(B) 프론트 cadence:** 수동 ack → `close()` 가 동기로 `_process()` 호출 → 0ms·디바운스 없이 다음 큐 항목 렌더 + 비프 ([alarm-popup.js:700-718](../../drf-server/static/js/shared/alarm-popup.js#L700-L718)). (자동닫힘 경로엔 이미 15/30s 간격이 있어 **수동 ack 한정** 문제.)

**의도한 결과:** 전력 알람이 가스처럼 디바이스 단위로 안정 수렴(폭주 제거) + 운영자가 버튼을 눌러도 다음 팝업이 잠깐 텀을 두고 1회 비프로 등장.

### ⚠ 핵심 설계 결정
- **A 는 "채널별 Event 분리"가 아니다.** 채널을 병합 키에 넣으면 디바이스당 Event 가 16개로 늘어 **오히려 팝업이 증가**한다. 정반대. 올바른 방향은 **마지막 활성 채널이 정상화될 때만 resolve** — 디바이스 단위 Event 를 유지하되 일찍 닫지 않기. (가스 `cleared_gases` 와 같은 의도, 전력 호출 cadence 에 맞춘 변형)
- **B 는 자동닫힘이 아니라 수동 ack 경로만 손댄다.** 자동닫힘은 이미 spacing 보유.

### 2. `drf-server/apps/alerts/tasks.py` — `fire_power_clear_task` resolve 게이팅

현재 ([606-649](../../drf-server/apps/alerts/tasks.py#L606-L649)): 정상 토스트 push → 무조건 `auto_resolve_active_events(prefix="power", device)`.

변경:
- 정상 토스트 push 는 **그대로** (채널별 정상 복귀 안내는 유지).
- `auto_resolve_active_events` 호출을 **게이팅**: lazy import `PowerDevice` + `has_other_active_channel`. 다른 채널이 아직 위험하면 resolve **skip**(`resolved=0`), 전부 정상이면 기존대로 resolve.
- 로그에 게이팅 사유 1줄 추가 (`resolved=0 (other channels active)` 식) — 운영 관찰성.

```python
# (의사코드 — 구현 시 기존 try 블록 안)
from apps.facilities.models import PowerDevice
from apps.monitoring.services.power_alarm import has_other_active_channel

device = PowerDevice.objects.filter(pk=device_id).first()
channel_count = device.channel_count if device else 16
if has_other_active_channel(device_id, channel, channel_count):
    resolved = 0  # 다른 채널 위험 지속 → Event 유지 (마지막 채널이 닫는다)
else:
    resolved = auto_resolve_active_events(event_type_prefix="power", power_device_id=device_id)
```

- `auto_resolve_active_events` 시그니처 **변경 없음** (가스 cleared_gases 와 달리 전력은 호출부에서 게이팅 — clear 가 채널 1개씩 호출되는 cadence 라 subset 방식이 안 맞음. 진단 §A-1 참조).

### 3. `drf-server/static/js/shared/alarm-popup.js` — 수동 ack 간격 + 비프 throttle

| 위치 | 변경 |
|---|---|
| 상수 (신규) | `_PROCESS_GAP_MS = 700` (다음 팝업까지 텀), `_BEEP_THROTTLE_MS = 1500`, 인스턴스 `_lastBeepAt = 0`, `_gapTimer = null` |
| `close()` [717](../../drf-server/static/js/shared/alarm-popup.js#L717) | 동기 `this._process()` → `this._scheduleProcess()` 로 교체. 단 `immediate=true` 면 기존처럼 즉시 `_process()` |
| (신규) `_scheduleProcess()` | `this.isOpen = true`(갭 동안 reserve) → `clearTimeout(_gapTimer)` → `setTimeout(()=>{ this.isOpen=false; this._process(); }, _PROCESS_GAP_MS)` |
| `_process()` 비프 [592](../../drf-server/static/js/shared/alarm-popup.js#L592) | `_playAlarmSound(level)` 를 throttle 가드로 감쌈 (`Date.now()-_lastBeepAt >= _BEEP_THROTTLE_MS` 일 때만 재생 + `_lastBeepAt` 갱신) |
| 격상 분기 [489-498](../../drf-server/static/js/shared/alarm-popup.js#L489-L498) | `this.close({acknowledged:false})` → `this.close({acknowledged:false, immediate:true})` (warning→danger 격상은 즉시 표출 유지) |
| `_goDetail()` [788-806](../../drf-server/static/js/shared/alarm-popup.js#L788-L806) | `clearTimeout(this._gapTimer)` 추가 (페이지 이동 시 잔여 타이머 정리) |

- **collectstatic 필요:** 소스는 `static/`, 서빙은 `staticfiles/`. 빌드 후 반영.

## 구현 엣지케이스 / 주의 (literal 실행자용)

- **⚠ state 직렬화 parity** — `try_transition` 은 raw `_redis().eval` + `pickle.dumps` + `cache.make_key` 로 저장. `cache.get_many([_state_key(...)])` 가 같은 make_key/unpickle 경로인지 확인. 불일치 시 채널별 `get_state(...)` 루프(≤15회, 검증된 경로)로 대체. **추측 금지 — 한 채널 테스트로 확인 후 채택.**
- **⚠ state TTL 60s** — 센서 dropout 으로 위험 채널 state 가 만료되면 "정상"으로 보여 조기 resolve 가능. 더미 1초 주기라 실사용 영향 작음. 안전망: 12h 병합 윈도우 + 다음 발화 재생성.
- **⚠ resolve scope 불변** — prefix `"power"` 라 `power_overload`·`power_anomaly_ai` 둘 다 resolve(현행 유지). 게이팅 판정은 **룰 per-channel state** 기준. AI 이벤트 수명을 룰에서 분리하는 건 범위 밖.
- **⚠ 동시 다채널 clear** — task 실행 시점 state 를 읽으므로(clear_state 후), 마지막 task 가 "타 채널 없음" 보고 resolve. 드물게 둘 다 "상대 활성"으로 보면 event 가 잠깐 잔존(다음 clear/만료 시 정리) — 폭주 아님, 허용.
- **⚠ B 갭 중 신규 알람** — `show()` 가 `isOpen=true`(갭) 라 큐에만 push, 700ms 후 픽업. 정상.
- **⚠ B `_currentLevel` reset** — 갭 동안 `_currentLevel=null`(close 가 reset) 이라 격상 매칭 안 됨 → 들어온 danger 는 일반 큐로. 격상 즉시성은 D6(immediate)로 보장되는 "현재 떠있는 모달" 케이스에 한정.
- **프론트 테스트 하네스 없음** — `alarm-popup.js` 는 자동 테스트 0(verify 확인). 수동 검증 필수.

## Rollout / Rollback

- **Rollout:** drf 이미지 재빌드 + 재시작(web + celery worker). 프론트는 `collectstatic` + 브라우저 캐시 무효화. 신규 인프라/compose/k8s **변경 없음**.
- **Rollback:** A 는 게이팅 추가(가산적) → revert 안전. B 는 프론트 revert + collectstatic. 데이터 마이그레이션 없음.

## 범위 밖 (명시)

- **DANGER 표시값 불일치** (aggregate=다른 축인데 watt 값 표시) — 별도 sprint (진단 §부가발견 1).
- **AI rate-limit 프로세스 메모리·축별 키** — 알람 스트림 이니셔티브.
- **단일채널 고속 oscillation 잔여 churn** — A 는 멀티채널 churn 을 잡음. 단일채널이 60s 내 danger↔normal 반복하면 여전히 새 event 생성(정당한 별개 incident일 수도). 필요 시 per-channel re-fire cooldown 후속 검토.
- **AI 이벤트 수명을 룰 clear 에서 분리** — 현행 prefix `"power"` 동시 resolve 유지.
- **더미 single 모드 매 틱 `enter_scenario` 재진입** — 테스트 하네스.
