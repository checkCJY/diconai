# diconai 알람 백엔드 신뢰성 개선 — Phase 1

## Context

현재 알람 시스템에서 다음 증상이 모두 보고되었습니다 — **silent drop / 지연 / 중복 / SQLite locked**. 면밀한 코드 검토 결과 원인이 단일 버그가 아니라 다층에 걸친 설계 결함임이 확인되었습니다.

| 층위 | 근본 원인 |
|---|---|
| **WebSocket** | `active_alarms`(메모리 list) + `alarm_signal`(asyncio.Event) 조합의 set/clear race, `is_new_event` 필터링 silent drop, `[:5]` 슬라이스 cap, FastAPI 재시작 시 휘발 |
| **DB** | SQLite WAL 미설정 → 동시 쓰기 경합 ("database is locked"), PowerData `bulk_create(ignore_conflicts=True)` 후 원본 `objs`로 알람 트리거 → 미저장 행도 발화 |
| **Dedupe** | `cache.get → fire_task.delay → cache.set` 비원자 패턴 → 동시 다중 수신 시 중복 알람 |
| **푸시** | Celery → FastAPI 푸시 timeout 3초·실패 silent fail → DB는 있고 알림 없음 |

목표: **DB는 SQLite 유지**(향후 PostgreSQL 이관 예정 — 이관 시 자동으로 비활성화되는 형태로 격리)하면서 알람 신뢰성 회복. **백엔드 신뢰성만**(UI는 Phase 2). 단일 PR / 커밋 단위 5단계.

UI 검토에서 발견된 색상 매핑 2가지 공존(`LevelMapper` vs 하드코딩 `caution-text`), 비대시보드 페이지 WS 연결 배너 부재, 알람 그룹 카운트 가시성, 모바일 반응형 부재 등은 Phase 2 별도 계획.

## 합의된 결정

1. **WAL 활성화**: `connection.vendor == "sqlite"` 가드. PostgreSQL 이관 시 자동 noop.
2. **`active_alarms` → Redis LIST**: `diconai:ws:alarms`, `LPUSH`/`BRPOP` FIFO, `LTRIM`으로 10k건 cap.
3. **`alarm_signal` 제거**: BRPOP이 자체적으로 신호화.
4. **WS 페이로드 호환 모드 유지**: 프론트 변경 없이 기존 `{"alarms": [payload], ...}` 형식 그대로. Phase 2에서 `type` 분리 검토.
5. **단일 PR / 커밋 5단계로 분류**:
   - C1: SQLite WAL pragmas
   - C2: 알람 dedupe 원자화 (Redis Lua)
   - C3: PowerData bulk_create 정합성
   - C4: `active_alarms` → Redis LIST 전환 (FastAPI 영역)
   - C5: `_push_to_ws` 신뢰성 보강 (timeout↑ + retry)

### C1 — SQLite WAL pragmas (DRF 단독)

**신규**: [drf-server/apps/core/sqlite_pragmas.py](../../drf-server/apps/core/sqlite_pragmas.py)

```python
from django.db.backends.signals import connection_created
from django.dispatch import receiver

@receiver(connection_created)
def apply_sqlite_pragmas(sender, connection, **_):
    if connection.vendor != "sqlite":
        return
    with connection.cursor() as cur:
        cur.execute("PRAGMA journal_mode=WAL;")
        cur.execute("PRAGMA busy_timeout=5000;")
        cur.execute("PRAGMA synchronous=NORMAL;")
        cur.execute("PRAGMA foreign_keys=ON;")
```

**수정**: [drf-server/apps/core/apps.py](../../drf-server/apps/core/apps.py) — `CoreConfig.ready()` 에서 `from . import sqlite_pragmas  # noqa` import.

**이관 안전성**: `vendor != "sqlite"` 가드로 PostgreSQL에서 즉시 return → noop. 파일 자체를 남겨도 무해.

### C2 — 알람 dedupe 원자화 (DRF 단독)

**신규**: [drf-server/apps/alerts/services/alarm_dedupe.py](../../drf-server/apps/alerts/services/alarm_dedupe.py)

Redis Lua로 GET→CMP→SET을 단일 원자 명령으로 실행하는 헬퍼.

```python
_TRANSITION_LUA = """
local cur = redis.call('GET', KEYS[1])
if cur == ARGV[1] then return 0 end
redis.call('SET', KEYS[1], ARGV[1], 'EX', ARGV[2])
return 1
"""

def try_transition(state_key: str, new_state: str, ttl: int = 3600) -> bool:
    """현재 상태가 new_state와 다를 때만 원자적으로 SET. 천이 성공 시 True."""
    client = cache.client.get_client(write=True)
    return bool(client.eval(_TRANSITION_LUA, 1, state_key, new_state, ttl))

def clear_state(state_key: str) -> None: ...
def get_state(state_key: str) -> str: ...
```

**수정**: [drf-server/apps/monitoring/services/gas_alarm.py:73-105](../../drf-server/apps/monitoring/services/gas_alarm.py#L73), [drf-server/apps/monitoring/services/power_alarm.py](../../drf-server/apps/monitoring/services/power_alarm.py) (동일 구조)

각 분기 패턴:
- DANGER: `if try_transition(state_key, "danger"): fire_danger_alarm_task.delay(...)` — fire 호출과 상태 갱신이 한 연산.
- WARNING 타이머: `cache.add(task_key, task_id, ttl)`로 setnx 가드 (`cache.add`는 Django Redis에서 SETNX 동작). add가 True일 때만 `apply_async(countdown=...)`.
- NORMAL: `pending = cache.get(task_key); cache.delete(task_key); if pending: _revoke(pending)`, `prev = get_state(state_key); if prev in (...): clear_state(state_key); fire_clear_*_task.delay(...)`.

**이관 안전성**: Django Redis 캐시 사용 — DB 무관, 그대로 유지.

```python
import json
import logging
from core.redis_client import get_redis

ALARM_QUEUE_KEY = "diconai:ws:alarms"
MAX_QUEUE_LEN = 10_000
logger = logging.getLogger(__name__)

async def push_alarm(payload: dict) -> None:
    r = get_redis()
    await r.lpush(ALARM_QUEUE_KEY, json.dumps(payload, ensure_ascii=False))
    await r.ltrim(ALARM_QUEUE_KEY, 0, MAX_QUEUE_LEN - 1)

async def pop_alarm_blocking(timeout: int = 0) -> dict | None:
    r = get_redis()
    try:
        result = await r.brpop(ALARM_QUEUE_KEY, timeout=timeout)
    except Exception as exc:
        logger.warning(f"[alarm_queue] brpop error: {exc!r}")
        return None
    if result is None:
        return None
    _, raw = result
    return json.loads(raw)

async def queue_len() -> int: ...
```

### C5 — `_push_to_ws` 신뢰성 보강 (DRF 단독)

**수정**: [drf-server/apps/alerts/tasks.py:43-89](../../drf-server/apps/alerts/tasks.py#L43)

```python
def _push_to_ws(alarm_data: dict, *, raise_on_failure: bool = True) -> None:
    alarm_data.setdefault("created_at", datetime.now(timezone.utc).isoformat())
    headers = {}
    token = getattr(settings, "INTERNAL_SERVICE_TOKEN", "") or ""
    if token:
        headers["Authorization"] = f"Bearer {token}"
    base = getattr(settings, "FASTAPI_INTERNAL_URL", "") or "http://127.0.0.1:8001"
    url = base.rstrip("/") + _ALARM_PUSH_PATH

    result = "success"
    pushed = True
    try:
        resp = httpx.post(url, json=alarm_data, headers=headers, timeout=10.0)
        if resp.status_code >= 500:
            raise httpx.HTTPStatusError("5xx", request=resp.request, response=resp)
    except (httpx.RequestError, httpx.HTTPStatusError) as e:
        logger.warning("FastAPI WS 알람 푸시 실패: %s", e)
        result = "failure"
        pushed = False

    # IntegrationLog 비동기 기록 (기존 로직 유지)
    try:
        from apps.operations.tasks.integration_log_task import integration_log_create_task
        integration_log_create_task.delay(
            integration_type="transmit",
            target_system="DRF→FastAPI",
            result=result,
            description=f"alarm_type={alarm_data.get('alarm_type', '')}",
        )
    except Exception:
        pass

    if not pushed and raise_on_failure:
        raise RuntimeError("FastAPI WS push failed")
```

호출자 측 변경 — `fire_danger_alarm_task` 등 `fire_*` 태스크에서 `_push_to_ws(payload)` 호출 시 기본 `raise_on_failure=True`로 작동 → 호출자의 `except Exception as exc: raise self.retry(exc=exc)`가 흡수.

예외 — `fire_clear_notification_task`, `fire_power_clear_task`는 `raise_on_failure=False`로 호출 (정상화 알림 손실은 critical 아님).

**중복 알람 트레이드오프**: DB는 commit됐고 WS push만 실패한 케이스에서 retry → 동일 페이로드 2회 전송 가능. Phase 1에선 허용(현재의 silent drop이 더 큰 문제). 완전한 idempotency는 Phase 2 (event_id 기반 클라 dedupe).

**이관 안전성**: DB 무관.

## 변경하지 않는 것

- **`event_service.py` `select_for_update()`**: SQLite에선 noop, PostgreSQL 이관 시 자동으로 row lock 동작. WAL + `@transaction.atomic`로 SQLite 동시성도 충분. 건드리지 않음.
- **WS 페이로드 shape**: `{"alarms": [...]}` 그대로 유지 → 프론트 코드 무수정.
- **Celery `max_retries=3, default_retry_delay=5`**: 그대로.

## 향후 plan 호환성

Phase 1이 만드는 인프라는 [`power-threshold-roadmap.md`](power-threshold-roadmap.md), [`gas-threshold-roadmap.md`](gas-threshold-roadmap.md), [`if-integration-guide.md`](if-integration-guide.md)의 기반이 된다. 다음 계약을 후속 plan이 그대로 따른다:

1. **`try_transition` 호출 패턴 유지**
   - `gas_alarm.py`/`power_alarm.py`의 `from apps.alerts.services.alarm_dedupe import try_transition` 호출 위치·시그니처는 변경 금지.
   - 전력 §2-3 (W·A·V 확장): `_evaluate(watt)` 호출만 W·A·V 3호출로 확장. `try_transition(state_key, "danger")` 분기는 종합 위험도(세 위험도 중 최대)에 그대로 적용.
   - IF §2-3 (combined_risk 결합): `try_transition`의 `new_state` 인자만 영문 매핑으로 normalize (`{정상→normal, 주의→warning, 예측경고→anomaly, 위험→danger}`).

2. **Redis 키 네임스페이스 분리**
   - Phase 1 점유: `diconai:ws:alarms` (WS 큐), `alarm:{gas|power}:state:*` (dedupe), `alarm:{gas|power}:task:*` (WARNING 타이머)
   - 후속 점유: `alarm:ratelimit:*` (IF rate limit, 15분 TTL), `alarm:{gas|power}:state:*:threshold|:anomaly` (IF AlarmType별 독립 dedupe 시 접미사)
   - Celery 큐: `default`(Phase 1 알람 푸시) vs `ml`(IF 추론) 분리 — Phase 1의 `_push_to_ws`는 `default` 큐이므로 ML 추론 task의 `queue="ml"` 분리는 알람 푸시 지연과 무관.

3. **`_push_to_ws` retry 중복 알람**
   - Phase 1 단독으로는 허용된 트레이드오프 (현재의 silent drop이 더 큰 문제).
   - IF §2 결합 매트릭스 도입 시 `_push_to_ws` 페이로드에 `event_id` idempotency key 추가 → FastAPI `alarm_queue`에서 동일 `event_id` 알람은 LPUSH 전 검사로 dedupe. Phase 2(UI) 또는 IF §2-3에서 보강.

4. **PowerData/GasData 모델 신규 필드와 무관**
   - 후속 §3 더미 보강에서 추가될 `is_anomaly` / `anomaly_type` 필드는 Phase 1의 [power_data.py](../../drf-server/apps/monitoring/serializers/power_data.py) `filter()` 재조회(`power_device`/`measured_at`/`data_type`/`channel`) 컬럼과 겹치지 않음 → 별도 마이그레이션만 추가하면 됨.

### D. [skill/plan/if-integration-guide.md](if-integration-guide.md) — §2-3 안에 신규 하위 항목 추가

> #### 2-3-a. Phase 1 인프라와의 결합
>
> Phase 1이 도입한 `try_transition`과 `alarm_queue`를 그대로 재사용하면서 `combined_risk` 4단계 분류를 얹는다.
>
> 1. **상태 키 분리**: 임계치 알람과 IF 알람을 독립 dedupe하기 위해 `state_key`에 `:type` 접미사 도입.
>    - `alarm:{gas|power}:state:{device_id}:{channel|gas}:threshold` ← 임계치 (Phase 1 키에서 접미사만 추가)
>    - `alarm:{gas|power}:state:{device_id}:{channel|gas}:anomaly`   ← IF
>    - 마이그레이션 시점: IF §2 머지일에 Redis `FLUSHDB` 또는 점진적 TTL 만료로 처리. 둘 다 1시간 TTL이라 자연 만료 권장.
>
> 2. **`combined_risk` → `try_transition` 인자 매핑**: 결합 매트릭스 결과 한글 라벨을 영문 상태로 normalize.
>    ```python
>    _COMBINED_TO_STATE = {"정상": "normal", "주의": "warning", "예측경고": "anomaly", "위험": "danger"}
>    new_state = _COMBINED_TO_STATE[combined_risk]
>    if try_transition(f"{base_key}:{'anomaly' if new_state == 'anomaly' else 'threshold'}", new_state):
>        fire_*_task.delay(...)
>    ```
>
> 3. **`AlarmType.ANOMALY` 추가**: `AlarmType` enum에 `ANOMALY = "anomaly"` 추가. `_push_to_ws` 페이로드의 `alarm_type` 필드에 그대로 전달.
>
> 4. **`event_id` idempotency**: Phase 1의 `_push_to_ws` retry로 인한 중복 알람 보강.
>    - `_push_to_ws(payload)` 호출 시 `payload["event_id"]`가 이미 있다면 (Phase 1에서 이미 전달) 그대로 사용.
>    - FastAPI [`alarm_queue.py:push_alarm()`](../../fastapi-server/websocket/services/alarm_queue.py)에 다음 가드 추가:
>      ```python
>      async def push_alarm(payload: dict) -> None:
>          r = get_redis()
>          eid = payload.get("event_id")
>          if eid:
>              recent_key = f"diconai:ws:alarms:recent:{eid}"
>              if await r.set(recent_key, "1", ex=60, nx=True) is None:
>                  return  # 1분 내 동일 event_id 중복 push 차단
>          await r.lpush(ALARM_QUEUE_KEY, json.dumps(payload, ensure_ascii=False))
>          await r.ltrim(ALARM_QUEUE_KEY, 0, MAX_QUEUE_LEN - 1)
>      ```
>    - 정상화 알림(`alarm_type ∈ {gas_clear, power_clear}`)은 `event_id` 없으므로 그대로 통과 → 영향 없음.
>
> **Phase 1 단독 머지 직후엔 위 보강이 미적용 상태**라 retry 시 중복 발송 가능 — IF §2 머지 전까지는 운영자 알람 피로가 일시적으로 증가할 수 있다. 모니터링 필요.
