6. `batch_size` 만큼 모이면 `PowerData.objects.bulk_create(...)` (`ignore_conflicts=False` — 백필이므로 충돌 없음)
7. 진행률 로그 (`logging`, 1만 row마다)

**가스 백필 참고 사항** (다른 트랙 작업자용):

### 목표
훈련된 IF 모델의 추론 결과를 **AlarmRecord/Event로 발화**시키고, **threshold 평가
결과와 결합한 `combined_risk`** 를 산출해 운영자에게 의미 있는 단일 알람으로 표시한다.

### 결합 매트릭스 후보 비교

`MLAnomalyResult.risk_classified` 필드는 이미 NORMAL/CAUTION/PREDICT_WARN/DANGER
4단계로 정의되어 있음 ([ml_anomaly_result.py:58-64](../../drf-server/apps/ml/models/ml_anomaly_result.py#L58-L64)).
이 필드를 어떻게 채울지가 결합 매트릭스 정의.

#### 후보 A — `max(threshold_risk, anomaly_risk)` 기반
- **로직**: 둘 중 더 위험한 단계 채택
- **장점**: 단순, false negative 최소
- **단점**: ① threshold가 NORMAL이고 IF만 이상인 케이스(예측 경보의 핵심 가치)가
  단순히 "WARNING 또는 DANGER"로 묻혀 운영자가 IF의 가치를 체감 불가.
  ② `PREDICT_WARN` 단계가 사실상 사용되지 않음
- **적합성**: IF 도입 의의 희석 → ✗

#### 후보 B — 매트릭스 표 기반 (3×2 또는 3×3)
- **로직** (추천 매트릭스):

  | threshold \ IF | normal | anomaly |
  |---|---|---|
  | NORMAL | NORMAL | **PREDICT_WARN** |
  | WARNING | WARNING | DANGER |
  | DANGER | DANGER | DANGER |

  (확장: IF 결과를 anomaly_score 분포에서 CAUTION/ANOMALY 2단계로 세분화하면 3×3.
  1차 구현은 3×2로 단순화 권장)

- **장점**: ① `MLAnomalyResult.risk_classified` 4단계 설계 의도와 정합.
  ② `PREDICT_WARN`이 "threshold는 정상인데 IF가 미세 이상 감지" 시그널로
  분명한 의미를 가짐 → IF 도입 ROI를 운영자가 체감 가능.
  ③ [if-integration-guide.md](./if-integration-guide.md) §2-2의 원래 설계와 일치
- **단점**: ① 매트릭스 정의가 자의적일 수 있음 (운영 경험 누적 후 보정 필요).
  ② IF false positive가 PREDICT_WARN 알람으로 이어져 알람 피로도 증가 가능 →
  rate limit (T1-5)로 완화 필요
- **적합성**: ✓✓ — 본 plan의 구조 선택

#### 후보 C — IF를 완전 분리된 알람 트랙으로 운영
- **로직**: threshold 알람과 IF 알람을 합치지 않고 `AlarmType.ANOMALY`로 별도 발화
- **장점**: 단순, IF false positive가 기존 알람 신뢰도 훼손하지 않음
- **단점**: ① 같은 사건에 대해 알람이 2종 발화 가능 → 운영자 혼란.
  ② `risk_classified` 필드가 사실상 의미 없는 mirror가 됨.
  ③ 통합 위험도 표시 불가
- **적합성**: 운영자 혼란 비용 > 단순성 이익 → ✗

### **추천: 후보 B (매트릭스 표 기반, 3×2 시작)**

**이유**:
1. `MLAnomalyResult.risk_classified` 4단계 설계가 이미 매트릭스 결합을 전제 → 가장 적은 모델 변경으로 합치
2. `PREDICT_WARN`이라는 "threshold 정상 + IF 이상" 케이스를 살려야 IF 도입의 핵심 가치(예측·조기경보)가 살아남음
3. 향후 운영 경험 축적 시 매트릭스 셀 값을 조정하기만 하면 되므로 **확장 비용이 가장 낮음** (구조 변경 없이 함수 내부 dict 수정)
4. rate limit (T1-5)으로 IF false positive의 알람 피로도 문제는 별도 완화 가능
