# 알람 서브시스템 E2E 트레이스

**한 줄:** 4개 진입원(가스·전력·AI·지오펜스)이 → 채널별 dedup/confirm 게이트를 통과해 Celery 태스크로 → Event 병합 + AlarmRecord 적재 → **Redis Stream**으로 → 3채널(관리자 브라우저 / 작업자 인앱 / Discord)에 전달되는, 단일 파이프라인이 아닌 **다-진입원 fan-out** 서브시스템.

가스·전력·포지셔닝 트레이스가 각자 ⑥ 알람에서 이 문서로 합류한다. 8개 영역으로 본다:
```
① 진입점(4종) → ② Celery 태스크 → ③ Event/Record 병합 → ④ 데이터 모델
⑤ dedup·confirm 방어 → ⑥ Redis Stream 큐 → ⑦ 3채널 전달 → ⑧ 계측
```
- 짝 문서: [06-alarm.md](06-alarm.md) (가스 알람 + `create_alarm_and_event` 상세 SoT), [07-screen.md](07-screen.md) (WS 전달).
- 최근 변경 (코드가 옛 문서 우선): **LIST→Redis Stream 전환**, **작업자 DANGER 인앱 수신** (PR #107), **Discord 연동** (PR #107), **전력 폭주 해소 — 채널-aware clear + danger 2틱 confirm** (PR #110).

| 진입원 | 트리거 함수 | alarm_type | 발생원 FK | 게이트 | Celery 우회 |
|---|---|---|---|---|---|
| 가스 | `trigger_gas_alarms` | `gas_threshold` | sensor (GasSensor) | 2틱 confirm + gas AI mute | 아니오 (Celery) |
| 전력 | `trigger_power_alarms` | `power_overload` | power_device | 2틱 confirm(watt축) + AI mute | 아니오 (Celery) |
| 전력 AI | `forward_inference_e2e` | `power_anomaly_ai` | power_device + ml_anomaly_result | RATE_LIMIT 60s + decide_alarm | **예** (FastAPI 직접 push) |
| 지오펜스 | `handle_position_receive` | `geofence_intrusion` | geofence + worker | 근접 30px + 영역 위험센서 | 아니오 (Celery) |

**핵심 차이:**
- 가스·전력·지오펜스는 **DRF → Celery → DB → push** 경로. AI는 **FastAPI에서 ML forward + AlarmRecord 직접 생성 + push**로 Celery를 우회 (event_id 없는 시점이라 fingerprint도 anomaly_meta 기반).
- 가스/전력 DANGER는 `try_transition`으로 채널/가스별 1회만 fire (race-safe). AI는 `decide_alarm` 매트릭스가 source(`ai`/`static_cover_*`)와 alarm_type을 결정.
- 가스 진입은 [06-alarm.md](06-alarm.md), 전력은 [power-e2e-trace.md](power-e2e-trace.md) ⑥, 지오펜스는 [positioning-e2e-trace.md](positioning-e2e-trace.md) ⑥, AI는 [../ai-pipeline-실제코드-2026-05-27.md](../ai-pipeline-실제코드-2026-05-27.md) SoT.

| 태스크 | 호출원 | 발화 방식 | 정상화 |
|---|---|---|---|
| `fire_danger_alarm_task` | gas_alarm | 즉시 (`.delay`) | — |
| `fire_warning_alarm_task` | gas_alarm | `countdown=WARNING_DURATION_SEC`(5s) 지연 | task_key 정리 |
| `fire_clear_notification_task` | gas_alarm | 즉시 (9가스 묶음 1메시지) | `auto_resolve_active_events` |
| `fire_power_danger_task` | power_alarm | 즉시 | — |
| `fire_power_warning_task` | power_alarm | `countdown=5s` | task_key 정리 |
| `fire_power_clear_task` | power_alarm | 즉시 | 채널-aware `auto_resolve` (⑤) |
| `fire_geofence_alarm_task` | position_service | 즉시 | — |

공통 골격 — **DB 쓰기 / WS 푸시 try 블록 분리** (`fire_danger_alarm_task` tasks.py:168-226):

```python
try:
    event, alarm = create_alarm_and_event(...)   # ③ — 실패 시 retry (DB 1회 보장)
except Exception as exc:
    raise self.retry(exc=exc)
if event and alarm:
    ALARM_FIRED_TOTAL.labels(...).inc()          # ⑧ 계측
    try:
        _push_to_ws({...})                       # ⑥/⑦ — 실패해도 retry 안 함
    except Exception:
        logger.warning("WS 푸시 실패 (DB 저장은 완료)"); return
```

**왜 2단 try:** Celery retry가 `create_alarm_and_event`를 재실행하면 AlarmRecord 중복. DB는 1회만, WS 푸시 실패는 fingerprint dedup + 다음 tick으로 자연 회복.

### `_push_to_ws` — push + Discord 미러 (tasks.py:69-145)

```python
resp = httpx.post(url + "/internal/alarms/push/", json=alarm_data, headers=headers, timeout=10.0)
if resp.status_code >= 500: raise httpx.HTTPStatusError(...)   # 5xx만 retry 가치
...
if pushed:                                        # WS 성공 시에만 Discord (중복 방지)
    send_alarm_to_discord(alarm_data)
if not pushed and raise_on_failure:
    raise RuntimeError("FastAPI WS push failed")  # → 호출자 self.retry
```

**왜:** ⑴ 5xx만 retry (4xx는 페이로드 오류라 무의미). ⑵ Discord는 dedup이 없어 retry 시 중복 → **WS 성공 1회에만** 미러. ⑶ `raise_on_failure=False`는 clear 태스크 전용 (정상화는 손실 허용). ⑷ IntegrationLog는 `.delay`로 비동기 INSERT.

## ③ Event/AlarmRecord 병합 (create_alarm_and_event)

`event_service.py`의 `create_alarm_and_event`가 모든 진입원의 공통 종착 — 상세 3분기는 [06-alarm.md](06-alarm.md) ③ SoT. 요약:

```python
event_qs = Event.objects.select_for_update().filter(           # 행 잠금 (race 차단)
    facility_id=..., event_type=alarm_type, status__in=[ACTIVE, ACKNOWLEDGED, IN_PROGRESS])
# + source_sensor / source_power_device / source_geofence 중 해당 FK 필터
active_event = event_qs.first()
```

- **분기 ① 시간초과**: `is_mergeable_time_window`(12h) 초과 → 옛 Event 강제 RESOLVED + 새 Event.
- **분기 ② 병합**: 시간 윈도우 안 → AlarmRecord만 추가, Event 갱신. **위험도 격상은 cooldown 무시 재알림** (`last_notified_at` 비교, `ALARM_REPOPUP_COOLDOWN_SEC`=60s).
- **분기 ③ 신규**: 활성 Event 없음 → `match_policy`로 `Event.policy` 매칭 + Event + AlarmRecord + EventLog(CREATED) + `transaction.on_commit`(Notification).

**병합 키**: `(발생원 FK, event_type)`. 전력은 발생원이 `source_power_device`(디바이스 단위) — 채널별이 아니라 **디바이스 공유 Event** → ⑤의 채널-aware clear가 필요한 이유.

## ⑤ dedup·race·confirm 방어

### 원자 상태 천이 — `try_transition` (alarm_dedupe.py:37-45)

```python
_TRANSITION_LUA = """
local cur = redis.call('GET', KEYS[1])
if cur == ARGV[1] then return 0 end          -- 이미 같은 상태 → skip
redis.call('SET', KEYS[1], ARGV[1], 'EX', ARGV[2]); return 1
"""
def try_transition(state_key, new_state, ttl=3600) -> bool:
    return bool(_redis().eval(_TRANSITION_LUA, 1, cache.make_key(state_key), pickle.dumps(new_state), ttl))
```

**왜:** `GET→CMP→SET`을 단일 Lua로 — 동시 다중 수신 중 정확히 1개만 True. `cache.make_key`/`pickle`로 Django RedisCache와 같은 키 공간 공유 (`get_state`/`cache.add`가 같은 값을 읽음).

### danger 2틱 confirm — `confirm_consecutive` (alarm_dedupe.py:59-70)

```python
def confirm_consecutive(count_key, threshold, ttl) -> bool:
    count = (cache.get(count_key) or 0) + 1
    cache.set(count_key, count, ttl)
    return count >= threshold      # threshold<=1 이면 첫 틱 True (즉시 발화)
```

**왜:** "N틱 연속 초과해야 발화" — 단일 틱 센서 스파이크/인러시가 즉시 경보 되는 것을 차단 (PR #107). 가스·전력 DANGER 공통. 스트릭 끊기면 호출자가 `cache.delete`로 리셋. 전력은 watt축에서만 카운트 (축별 호출이라). 계측: `POWER_DANGER_CONFIRM_TOTAL{held/confirmed/reset}`.

### AI 우선순위 mute — `is_ai_mute_active` / `mark_ai_recent` (alarm_dedupe.py:95-144)

```python
_AI_FIRED_KEY_TEMPLATE = "ai_fired:{device_id}:{channel}:{rule_level}"
# mark: 발화 level '이하' 키를 모두 set → 격상 bypass 자연 표현
_LEVELS_AT_OR_BELOW = {"warning": ["normal", "warning"], "danger": ["normal", "warning", "danger"], ...}
def is_ai_mute_active(device_id, channel, rule_level) -> bool:
    try: return bool(_redis().exists(key))
    except Exception: return False     # fail-open
```

**왜:**
- AI가 발화하면 같은 (device, channel)의 룰 알람을 60s mute (중복 팝업 방지). FastAPI `services/ai_mute.py`가 같은 raw 키 공간에 마킹, DRF가 읽음 → **`device_id`는 반드시 IoT raw id**(PK 아님, 키 mismatch 방지).
- **격상 bypass**: AI가 warning만 set하면 danger 키 부재 → 룰 danger 자유 통과 (AI=경고인데 실제 danger면 막지 않음). level별 키 분리로 분기 로직 없이 표현.
- **fail-open**: Redis 장애 시 False — mute 가드 실패가 알람 발화를 막으면 안 됨. 가스는 `is_gas_ai_mute_active`(`ai_fired_gas:*`, co/h2s/co2만).

### 채널-aware clear (전력 폭주 해소, power_alarm.py:153 / tasks.py:645)

`has_other_active_channel`로 "마지막 활성 채널이 정상화될 때만" Event resolve — 1채널 복귀가 디바이스 공유 Event를 닫아 다음 발화가 새 event_id로 폭주하는 것 차단 (PR #110). 상세 [power-e2e-trace.md](power-e2e-trace.md) ⑥, `[[power_alarm_flood_diagnosis_2026_06_02]]`.

## ⑥ Redis Stream 큐 (LIST→Stream 전환)

키 `diconai:ws:alarms` (Django RedisCache와 분리). **Consumer Group 미사용** — 경쟁 분배(BRPOP)가 아니라 replica별 독립 XREAD로 모든 replica가 모든 알람을 받는 **fan-out** (멀티레플리카 대비).

### `push_alarm` — fingerprint dedup → XADD (alarm_queue.py:108-146)

```python
fp = _payload_fingerprint(payload)
if fp is not None:
    if not await r.set(DEDUP_KEY_PREFIX + fp, "1", nx=True, ex=dedup_ttl):  # SET NX EX 30s
        push_alarm_dedup_hits.inc(); return                                  # 중복 silently drop
await r.xadd(ALARM_QUEUE_KEY, {"data": json.dumps(payload)}, maxlen=MAX_QUEUE_LEN, approximate=True)
```

fingerprint 분기 (`_payload_fingerprint`):
- 룰 알람: `event:{event_id}:{risk_level}` (Celery retry는 같은 event_id) / RESOLVED 신호는 `event:{id}:resolved`
- AI 알람: `ai:{alarm_type}:{device_id}:{channel}:{risk_level}` (event_id 없는 시점)
- 정상화: `clear:{alarm_type}:{source_label}` (9가스 동시 정상화 → 9 push → 1)
- 전력 cover: `cover:{source}:{source_label}:{risk_level}` (매초 평가 폭주 차단)

**왜 TTL 30s:** Celery retry 총합(5s×3=15s)보다 길고, 재알림 cooldown(60s)보다 짧게 — 1분 후 정상 재발화는 안 막음. **XADD가 dedup보다 부분실패 가능하나** retry 중복(빈번) 차단 효과가 누락 위험(희소)보다 크다 (docstring 명시).

### `read_alarms_blocking` — XREAD 배치 + 커서 (alarm_queue.py:149-198)

```python
result = await r.xread({ALARM_QUEUE_KEY: last_id}, block=timeout * 1000)
# 빈 결과/예외면 (last_id, []) — 커서 미전진 (다음 iteration 재시도)
for entry_id, fields in entries:
    new_last_id = entry_id                      # 배치 마지막 ID로 커서 전진
    payloads.append(json.loads(fields.get("data")))
return new_last_id, payloads
```

**왜:** BRPOP(1건 경쟁 소비)과 달리 커서 이후 누적 N건을 배치로. 커서는 호출자(`alarm_flush_loop`)가 메모리 보유 — 부팅 시 `"$"`(신규만)에서 시작해 과거 무한 replay 방지. 예외 시 커서 동결 → slow-retry. `reset_stream_if_wrongtype`이 startup에 잔존 LIST 키만 1회 DEL (WRONGTYPE 방지).

## ⑦ 3채널 전달

### 채널 1 — 관리자 브라우저 (`alarm_flush_loop`, ws_router.py:55-107)

```python
last_id = "$"
while True:
    if not sensor_clients: await asyncio.sleep(1); continue        # M-1: 클라 없으면 커서 동결(보존)
    new_last_id, payloads = await read_alarms_blocking(last_id, timeout=1)
    if payloads:
        state = await fetch_broadcast_state()
        for payload in payloads:
            ingress_ts = payload.pop("ingress_ts", None)
            if ingress_ts: E2E_ALARM_LATENCY.labels(risk_level=...).observe(time.time() - ingress_ts)
            base = build_broadcast_payload(state, include_alarms=False); base["alarms"] = [payload]
            await _send_to_all(base)                                # sensor_clients 전체
    last_id = new_last_id
    ALARM_STREAM_LAG.set((_id_ms(tail) - _id_ms(last_id)) / 1000)   # stream lag 계측
```

### 채널 2 — 작업자 인앱 (`alarm_router.py:152-172`)

```python
# /internal/alarms/push/ 핸들러 안 — 스트림 XADD 후
if alarm.alarm_type == "geofence_intrusion" and alarm.worker_id is not None:
    ws = worker_clients.get(alarm.worker_id)                       # 위험구역 진입 → 개인
    if ws: await ws.send_json({"type": "worker_alert", **payload})
elif alarm.alarm_type in ("gas_threshold", "power_overload") and alarm.risk_level == "danger":
    for worker_id in alarm.target_worker_ids:                      # DANGER → 소속 시설 작업자
        ws = worker_clients.get(worker_id)
        if ws: await ws.send_json({"type": "worker_alert", **payload})
```

**왜:** ⑴ 지오펜스는 진입한 그 작업자에게만 (`worker_id`). ⑵ 가스/전력 **DANGER**는 DRF가 시설 기준 계산한 `target_worker_ids` 중 접속 작업자에게 대피 알림 (전체 broadcast 아님, WARNING은 미전송). ⑶ `target_worker_ids`는 분배 후 **스트림 push 전 pop** — 브라우저 broadcast엔 안 실림 (민감·bloat). 동일 event_id는 관리자 대시보드에선 프론트 dedup이 흡수.
