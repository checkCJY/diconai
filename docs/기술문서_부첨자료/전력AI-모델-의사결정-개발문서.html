1. 초록

ABSTRACT

전력 IF의 channel 단위 architecture는 옵션 B(채널 3-D raw) → D(하이브리드) → E(디바이스 16-D) → 커리큘럼 정합 회귀의 4단계 의식 흐름을 거쳐 채택했음.

처음 3단계는 학습 프레임워크(STEP 1~5)를 충분히 참조하지 않은 추상적 설계 사고였음. 4단계에서 학습 문서 재정독으로 옵션 B~E가 임의 설계였음을 인지하고 a247fb3 commit(STEP 3 + STEP F 정합 구현체)으로 회귀했음.

회귀 직후 강사 multivariate IF 외부 제안(05-29)에 대해 즉시 구조 변경 대신 Pearson + DTW + null baseline PoC로 정량 응답했음. motor(ch1/3/8/13)·panel(ch11/16) cluster 후보를 식별했으나 더미 tautology·N=1·6일치 한계로 architecture 결정은 D+30 운영 데이터 재측정 + D+45~D+90 3 architecture F1 비교로 보류했음.

핵심 설계 동기는 “학습 프레임워크 우선 · 피드백 자의적 해석 차단 · 정량 측정 응답 · 정직한 한계 인정”임.

3. 설계 철학

본 의사결정의 축은 다음 세 원칙임.

  1. 학습 프레임워크가 추상 설계보다 우선
  2. 외부 입력에 정량 측정으로 응답
  3. 도메인 의존 결정 인지

3.1 학습 프레임워크가 추상 설계보다 우선

초기 옵션 B~E 검토는 추상적 직관에 의존했음. “채널마다 다른 장비, 다변량이 단변량보다 강함” 정도의 가정만 깔려 있었음.

학습 문서(STEP 1~5)는 (device, channel, data_type) × engineered features(value/roll_mean/roll_std/diff)를 명시했고 a247fb3가 이를 구현했음. 학습 프레임워크가 있으면 1순위 참조하고, 추상 설계는 그 안에서 변형하는 것이 올바른 순서임.

3.2 외부 입력에 정량 측정으로 응답

강사 외부 multivariate IF 제안(05-29)에 대해 즉시 구조 변경 대신 PoC로 정량 응답했음.

외부 권위가 있는 제안이라도 베이스라인 유지, 측정, 결정 보류가 정답 패턴임을 4단계에서 학습했고 5단계 PoC에서 실제로 적용했음.

결정 근거 05-29 작성. 근거: 2026-05-29 channel-correlation-poc.md §5 — “본 PoC의 가치는 정답 도출이 아니라 분석 파이프라인 + 비교 baseline + null 기준선 확보임”.

3.3 도메인 의존 결정

architecture 결정은 도메인 특성에 따라 다른 답이 나옴. 가스(분 단위 즉시 위험)와 전력(시간~일 단위 점진 변화)이 다른 ARIMA 처리(가스 = 격하 / 전력 = Un-격하)를 채택한 것과 동일 원칙임.

도메인 사고 시간 척도 architecture 결정
가스 분 단위 즉시 위험 다변량(CO+H2S+CO2 등) 자동 정당. 동시 패턴이 진짜 신호.
전력 시간~일 단위 점진 변화 channel 단위 분리가 자동 정당하지 않음. cluster 구조에 따라 달라짐. 본 부첨 §6 PoC가 답하려는 질문.

원칙 도메인이 결정 변수면 일반 해결책을 추구하지 말 것. 가스의 채널 통합이 전력에 그대로 적용되지 않음. 전력의 channel 단위 결정도 cluster 측정으로 다시 열려 있음.

결정 근거 ai-model-study-2026-05-17.md §2.4 격하 vs Un-격하 매트릭스. “가스 격하는 도메인에 맞춘 최적화이지 일반 해결책이 아님” 원칙을 architecture 결정에 확장 적용했음.

4. 의사결정 흐름 (시간순·결정 중심)

옵션 B → D → E → 커리큘럼 회귀 → PoC 순서로 4단계 의식 흐름과 1회 정량 측정으로 진행했음.

4.1 옵션 B — 채널 × 3-D raw

전 후
모델 단위·입력 결정 없음. 채널마다 다른 장비 가정만 있음. → 모델 단위 = (device, channel). 입력 = 3-D [current, voltage, watt] raw. 디바이스당 16모델. 운영(9가스 + 3전력) 합계 57모델.

근거 (당시)

채널마다 다른 장비가 연결된다는 직관. 3-D가 1-D보다 다변량 신호를 더 잡을 거라는 가정.

한계 (다음 단계에서 정량 확인) V·I·W collinearity 미검토. 채널 간 패턴 학습 불가.

증빙 학습 문서 미참조 상태의 추상 직관. 구체 측정 없음.

4.2 옵션 D — 채널 1-D × 16 + 디바이스 16-D × 1 (하이브리드)

전 후
옵션 B (57모델, 실효 1.5-D) → 채널 1-D current × 16 + 디바이스 16-D current × 1 = 17모델/디바이스. 운영 60모델.

근거 (collinearity 정량)

  • voltage ≈ 규제 220V. 분산이 극소라 정보량이 거의 0.
  • W = V × I × cos(φ). V가 일정한 환경에서 W와 I의 선형 상관이 0.99에 가까움.
  • 결과: 옵션 B의 3-D raw 실효 차원이 약 1.5-D 수준.

전환 의도

  • 단일 채널 이상은 1-D 모델이 잡음.
  • 채널 간 패턴은 16-D 모델이 잡음.
  • voltage·watt를 베이스라인에서 제외해 collinearity 회피.

한계 (다음 단계에서 인지) 모델 수 12 → 60 (5배 증가). 운영 부담 측면 미검토.

4.3 옵션 E — 디바이스 16-D current 단일

전 후
옵션 D (60모델, 채널 + 디바이스 분리) → 디바이스 16-D current 단일 모델. 디바이스당 1모델. 운영 12모델.

근거

강사 피드백 액면 “모델이 많아지면 공장환경에서의 가동이 문제”. current 외 voltage·watt 제외 (collinearity로 정보 미미).

근거 정량화 (강사 피드백이 무엇을 지칭했나)

“공장 가동 문제”의 구체 비용은 도입 사전 확인(ai-model-study §3.1)에서 3개 리스크로 정리됐음.

리스크 단계별 정량
DB 폭주 1 device × 16ch × 3 type 기준 일 138K row, 1년 약 50M row. 10 device 확장 시 1년 약 500M row. SQLite 운용 불가.
모델 관리 운영 부담 모델 파일 수가 1~2개에서 수십~수백 개 단위로 증가. 학습 batch가 30초에서 수시간 단위로 증가.
추론 latency asyncio가 sklearn predict (CPU bound)를 serial 처리. 다수 sensor 동시 송출 시 backpressure 발생.

연결 옵션 E의 12모델 축소는 위 3개 리스크 전면 회피를 의도했음. 단 학습 프레임워크 미참조 위에서 회피했기 때문에 회귀 단계(§4.4)에서 정정됨.

부록 증빙 설계 (강사 “변경하려면 증빙” 요구 대응)

번호 내용
D-1 V·I·W collinearity 정량화
D-2 옵션 B/D/E 탐지 성능 비교
D-3 자원 부담 측정
D-4 합성 vs 실데이터 비교
D-5 max_samples=256 동작 문서화

잘못 (다음 단계에서 인지) 강사 피드백 자의적 확대 해석. 실제 의도는 “운영 부담 인지”였으나 받은 해석은 “구조 바꿔라”였음. 옵션 B → D → E 모두 학습 프레임워크 미참조 위에 쌓였음.

4.4 커리큘럼 정합 회귀 — a247fb3 채택

전 후
옵션 E (디바이스 16-D 스냅샷, 시간성·engineered features 없음) → (device, channel, data_type) × 4-D engineered features [value, roll_mean, roll_std, diff]. window 30, contamination 0.01. a247fb3 commit 그대로.

근거 (커리큘럼 직접 인용)

커리큘럼 위치 인용
STEP 3 예제 FEATURES = ["value", "roll_mean", "roll_std", "diff"]
STEP 3 예제 IsolationForest(contamination=0.01, random_state=42)
STEP 5 STEP C “device_id별 + 가스 항목별 최근 N개”
STEP 5 STEP F “value, rolling mean, rolling std, diff 같은 feature를 구성”

a247fb3 ↔︎ 커리큘럼 1:1 매핑

위 4개 인용 모두 코드와 일치 (apps/ml/services/feature_service.py, train_anomaly_model 디폴트값).

4 features의 의미 (STEP 3 → a247fb3 매핑 상세)

각 feature가 잡는 이상 패턴은 도입 사전 확인(ai-model-study §1.3)에서 정리됐음.

feature 계산 잡는 이상 패턴
value arr[-1] (현재값) 학습 분포 밖의 큰 값. 갑작스러운 부하.
roll_mean arr.mean() (30틱 평균) 점진적 상승·하강 trend. 모터 부하 증가.
roll_std arr.std(ddof=0) (30틱 표준편차) 변동 폭 증가. 베어링 마모, 회로 노이즈.
diff arr[-1] - arr[-2] (1차 차분) 단발 spike. 전력 surge.

위치(value) + trend(roll_mean) + 분산(roll_std) + 변화율(diff)이 시간 시계열 기본 통계 요약을 구성함. STEP 3 학습의 4피처 선정이 자의적이지 않은 도메인 정당성을 가짐.

인지한 점

학습 프레임워크 미참조 → 추상 설계 → 임의 회귀의 연쇄였음. 정답은 이미 구현된 a247fb3였음.

증빙 학습 문서 7건 재정독 결과. 근거: STEP 1~5 학습 자료, a247fb3 commit (apps/ml/).

4.5 채널 상관성 PoC — 외부 multivariate IF 제안 대응 (2026-05-29)

전 후
강사 “디바이스 기준 multivariate IF가 표준” 외부 제안. Phase 1~3 reflex라면 즉시 구조 변경. → 베이스라인(per-channel) 유지. Pearson + DTW + null baseline PoC 당일 실행. 결정 보류.

측정 결과 (§6 상술)

  • 실측 DTW 0.184 / null 0.266. 비율 0.69.
  • motor-panel DTW 0.241 = null의 91%. 카테고리 간 무관.
  • cluster 후보: 모터 ch1/3/8/13, 패널 ch11/16.

채택안

per-channel 유지. architecture 결정은 D+30 운영 데이터 재측정 후로 보류.

고려한 대안 · 포기한 것 per-cluster(2~3모델/디바이스) 즉시 채택을 포기했음. 더미 tautology, N=1, 6일치 한계로 정당성 부족. 운영 데이터 재측정 + 3 architecture F1 비교(D+45~D+90) trigger로 등록했음.

7. 트레이드오프

옵션 B → D → E → 커리큘럼 회귀 → PoC 각 단계의 “채택 / 포기한 것 / 정당화”를 한 자리에 정리했음.

7.1 트레이드오프 매트릭스

결정 채택 포기한 것 얻은 것 · 정당화
옵션 B 채널 × 3-D raw 모델 단순 통합 직관적 다변량 가정. 사후 정량 근거 부재.
옵션 D 채널 1-D + 디바이스 16-D 하이브리드 옵션 B의 단순 구조 collinearity 회피 + 채널 간 패턴 학습. 단 모델 수 5배 증가.
옵션 E 디바이스 16-D current 단일 채널별 sensitivity 모델 수 12개로 운영 부담 감소. 단 강사 피드백 자의적 해석 위에 구축됨.
커리큘럼 회귀 (device, channel, data_type) × 4-D engineered (a247fb3) 옵션 B~E 임의 설계 학습 프레임워크 정합. STEP 3·F 인용 가능.

관통하는 원칙 각 단계의 “포기”가 다음 단계에서 “왜 포기했어야 했는가”로 정당화됨. 옵션 B의 collinearity 미검토 포기가 D에서 정량 확인됐고, D의 운영 부담 포기가 E의 피드백을 받았고, E의 임의 설계 포기가 커리큘럼 회귀로 학습 프레임워크를 회복시켰고, PoC의 즉시 채택 포기가 운영 데이터 재측정으로 정량 결정을 준비시킴.

7.2 자체 식별한 학습 (P1~P4)

번호 학습 적용

P2 피드백 자의적 확대 해석. “운영 부담 인지”가 “구조 바꿔라”로 변질. 액면 그대로 받기로 원칙 등록

8.1 정량 의사결정 trigger

조건 채택안
cluster 재현됨 + F1(B) > F1(C) + Δ ≥ 0.05 per-cluster 채택
cluster 재현됨 + F1(B) ≈ F1(C) per-channel 유지 (단순함 우선)
cluster 재현 안 됨 per-channel 유지
F1(A) > F1(B) 및 F1(A) > F1(C) per-device 재검토 (가능성 낮음)

8.2 단계별 모델 수 스케일 — per-cluster 채택의 운영 효과

PoC 결과 per-cluster 채택 시 모델 수 17N에서 (2~3)N + 16N으로 압축 가능. 도입 사전 확인 §3.4의 단계별 스케일에 본 부첨 결과를 더한 표.

단계 범위 per-channel (Un-격하 + IF) per-cluster (PoC 결과 시) IF 압축률
1 1채널 PoC (현재) 1 1 —
2 1 device, 16 채널 17 (1 IF + 16 ARIMA) 2~3 IF + 16 ARIMA = 18~19 IF 1 → 2~3
3 N devices (5~) 17N (2~3)N + 16N = 18~19N IF만 약 85% 감소

해석 ARIMA는 단일 시계열 specific이라 채널별 분리가 강제됨 (16N 고정). per-cluster 채택의 압축 효과는 IF 모델 수에 한정됨. 다만 IF는 모델 파일, 학습 batch, 캐시 관리의 주력이라 운영 효과가 큼.

연결 §4.3 옵션 E의 정량 비용 표(DB 폭주·모델 관리·latency)가 17N의 부담을 가리켰고, 본 §8.2가 per-cluster 채택 시 IF 부담을 어떻게 줄일 수 있는지 정량화했음. 단 PoC 결과는 더미 한정이라 §8.1 trigger 만족 시에만 채택.

증빙 ai-model-study §3.4 (전력 단계별 모델 수), §7.1 (단계별 작업).

본 부첨은 시연 자료가 아니라 의사결정 trace의 부첨 자료임. 인용 시 “현재 채택 = per-channel, 변경 trigger = D+45 F1 비교”로 사용할 것.
