## 1. watt 단독 추론 (A/V 는 저장만)

### 결정
IF + ARIMA 추론은 `watt` 한 채널만. `current`, `voltage` 는 raw 저장만 하고 추론 대상에서 제외.

### 장단점

| 장점 | 단점 |
|---|---|
| P = V × I — watt 한 축으로 부하 이상의 종합 신호 확보 | 전류 파형 이상 (절연 열화·역률 이상) 의 조기 검출 불가 |
| 학습·추론 모델 수 1/3 — 1ch PoC 단계의 운영 부담 최소 | A/V 의 도메인 특성 (방향성·위상) 활용 못 함 |
| 정격 % 임계가 watt 기준이라 통계·도메인 임계의 단위 일관 | 다변량 IF (W+A+V) 의 잠재 검출력 미활용 |

### 선택 이유
- **시연 가치 우선:** 시연 핵심 메시지는 "AI 가 부하 이상을 잡는다" — watt 만으로 충분히 시연 가능.
- **PoC 단계의 모델 N 최소화:** A/V 까지 학습하면 3×N 채널 학습·운영 부담. 1ch PoC 의 검증 목적과 비례하지 않음.
- **A/V 도메인 가치는 4차 본격 도입 시점에 맞음:** [4차_향후확장방향_문서.md](../4차_향후확장방향_문서.md) §4 — 실 게이트웨이 도입 + 라벨링 데이터 누적 후가 적절.

### 후속 plan
- 시연 후 D+30 운영 데이터 누적 → A 도 watt 와 함께 다변량 IF feature 로 묶는 PoC.
- 4차 단계에서 SARIMAX + W/A/V 다변량 정식 도입 검토.

## 2. 16채널 중 4채널만 활성

### 결정
디바이스 `63200c3afd12` 의 ch1 (압연기) / ch9 (메인 전력반) / ch14 (공조) / ch15 (조명) **4채널만** IF + ARIMA 추론 분기 진입. ch2~8 / ch10~13 은 정적 임계만 작동.

### 장단점

| 장점 | 단점 |
|---|---|
| 부하 종류 다양성 확보 (모터·전력반·공조·조명) — 모델 일반화 1차 검증 가능 | 16채널 중 25% 만 5축 풍부 시그널 적용 — 나머지 12채널은 정적 임계 의존 |
| ch2~8 모터 동질군 제외 → 학습 데이터 중복 최소 | 다채널 운영 부담 (LRU 캐시·메모리 폭증) 실측 미진행 |
| ARIMA 학습 cost 4× 만 발생 (4채널 = 4 모델) | auto-arima·SARIMA 미적용 상태로 16채널 확장 시 (p,d,q) 튜닝 부담 ↑ |

### 선택 이유
- **1ch → 4ch 는 모델 일반화 검증의 첫 단계:** [../../docs/codereviews/2026_05_21/power-ai-multichannel-activation.md](../../docs/codereviews/2026_05_21/power-ai-multichannel-activation.md) — 부하 종류별 IF + ARIMA 일관 동작 확인이 핵심 목표.
- **D+30 un-downgrade 정식 적용 근거 데이터 수집:** 4채널 × 4주 운영 데이터로 ARIMA / IF 의 도메인 적합성 정량 측정 → 16채널 확장 의사결정 근거.
- **시연까지 운영 안정성 우선:** 16채널 한 번에 활성화 시 false positive / 자원 폭증 위험. 4채널로 incremental 진행.

### 후속 plan
- D+30: 운영 데이터 분석 → 16채널 확장 가능성 평가.
- D+30 ~ D+90: Lazy + LRU 캐시 도입 (`_cache` 에 cap 추가), auto-arima 튜닝.
- 디바이스 N 폭증 대비 클러스터링 PoC ([IF_ARIMA_적용현황 §3-D](IF_ARIMA_적용현황_2026_05_19.md)).

## 3. 윈도우 30 (1Hz × 30초)

### 결정
IF · ARIMA 입력 윈도우 `maxlen=30`. Change Point 만 `maxlen=60`.

### 장단점

| 장점 | 단점 |
|---|---|
| 1Hz dummy 기준 30초 — 추론 latency 짧음 (수신 즉시 분기) | ARIMA(1,1,1) 의 1-step forecast 가 국소 노이즈에 민감 |
| 가스와 동일 윈도우 → 도메인 간 추론 코드 통일 | 부하 사이클 (펌프·공조 가동 주기 1~5분) 보다 짧음 |
| 슬라이딩 윈도우 메모리 부담 최소 — 다채널 확장 시 유리 | trend break 검출이 짧은 윈도우라 늦거나 false positive 가능 |

### 선택 이유
- **추론 latency 우선:** 알람 발화 latency p95 < 2초 목표 ([4차_향후확장방향 §6.3](../4차_향후확장방향_문서.md)). 윈도우 길수록 누적 대기 ↑.
- **Change Point 가 윈도우 길이 보완:** CP 만 maxlen=60 으로 운영 — trend 변화 시점은 60초 윈도우의 two-window 비교로 잡음.
- **튜닝 cost 분리:** 시연 전 윈도우 변경 = 회귀 위험. 시연 후 false positive 빈도 측정 후 조정이 안전.

### 후속 plan
- 시연 리허설 (D-7 ~ D-3): dummy 1주 가동 후 false positive 빈도 측정 → Z=3.0 / window=30 튜닝 근거 확보.
- D+30 sprint: 부하 사이클 길이 분석 (FFT / ACF) → 윈도우 60~120 실험.

## 4. night_abnormal 시각 격상 (KST 22-05 + 정격 30%)

### 결정
KST 야간 (22:00 ~ 05:00) + watt > 정격 × 0.30 시 위험도 1단계 격상.

### 장단점

| 장점 | 단점 |
|---|---|
| SARIMA 의 계절성 학습 cost 없이 도메인 휴리스틱으로 야간 가동 검출 | 상시 가동 장비 (펌프·냉장설비) 에서 오탐 누적 가능 |
| 운영자가 이해하기 쉬운 룰 — "야간 + 정격 30%" 한 줄 설명 | 정격 30% cutoff 의 데이터 근거 미공식 (도메인 직관) |
| 시연 시 시각 컨텍스트 검출 능력 명시 가능 | 채널별 야간 baseline 학습 미적용 — 일률 30% |

### 선택 이유
- **SARIMA 도입은 4차 본격 단계의 비용:** [IF_ARIMA_적용현황 §Part 6-A 문제 #1](IF_ARIMA_적용현황_2026_05_19.md) — SARIMA(p,d,q,P,D,Q,m) 학습 명령 추가 + 일·주 단위 데이터 누적이 선행 필요.
- **시연 가치 vs 학습 cost trade-off:** 학습 자료의 5축 정책 엔진 권고 (시각 컨텍스트 명시) 는 휴리스틱으로도 시연 가능.
- **운영 데이터 누적 후 baseline 학습이 자연스러움:** D+30 운영 후 채널별 야간 정상 부하 분포 측정 → baseline 대비 격상으로 전환.

### 후속 plan
- D+30 ~ D+90: 채널별 야간 baseline 학습 → 정격 % 일률 대신 baseline 대비 격상.
- 데이터 누적 충분 시 SARIMAX 도입 검토.

## 5. ARIMA 1-step forecast + 95% CI

### 결정
`get_forecast(steps=1)` + `conf_int(alpha=0.05)` 의 CI 위반 여부로 ARIMA 축 판정. multi-step / CUSUM 미적용.

### 장단점

| 장점 | 단점 |
|---|---|
| forecast cost O(1) — 1Hz 스트림에 실시간 가능 | **단발 spike 한계 명시** — forecast 가 actual 근처로 빠르게 적응해 CI 안 들어옴 |
| 신뢰구간 위반 = 직관적 신호 (`is_violation` boolean) | 점진 drift 누적은 매 step 모델 갱신으로 따라가 못 잡을 수 있음 |
| 95% CI 의 통계적 의미 명확 — 운영자 설명 용이 | n-step ahead 의 운영 가치 (predictive maintenance) 미활용 |

### 선택 이유
- **단발 spike 한계는 의도된 동작, 다른 축 (IF + threshold) 이 잡음:** [../troubleshooting/0519_arima-single-spike-limit.md](../troubleshooting/0519_arima-single-spike-limit.md) — 4축 결합으로 보완 검증 완료.
- **ARIMA 의 강점은 trend break / 점진 상승, 1-step 으로 충분:** 단발 spike 잡으려고 multi-step 으로 늘리면 false positive ↑.
- **Change Point 가 drift 누적 보완:** STEP E CP (two-window 비교) 가 CUSUM 의 누적 drift 검출 역할 대체.

### 후속 plan
- D+30: ARIMA multi-step forecast (steps=5~10) PoC → 누적 위험 점수 (PREDICTIVE_ALERT 권고).
- D+30 ~ D+90: SARIMAX (계절성 추가) → 1-step forecast 의 시각 사이클 적응 정확도 향상.
- 장기: Online ARIMA (Kalman filter / RecursiveLS) 점진 update.

## 6. ARIMA 와 IF 동등 가중치 (un-downgrade)

### 결정
전력 5축 결합에서 ARIMA 는 IF 와 동등한 algorithm. `combine_risk_3axis(threshold, IF, ARIMA)` 의 12 cell 매트릭스로 base 결정. 가스는 ARIMA 잔차를 IF 의 15-피처 중 3개로 격하.

### 장단점

| 장점 | 단점 |
|---|---|
| ARIMA 의 4가지 본질 가치 (CI 위반 / trend break / multi-step forecast / seasonal) 살릴 수 있음 | ARIMA 가 non-stationary 데이터에서 false positive ↑ — IF 와 동급 가중치의 정당성은 운영 데이터로 확인 필요 |
| 운영자가 "어느 축이 격상에 기여했는지" 추적 가능 (`algorithm_source` 라벨) | 가스와 비대칭 — 도메인 간 architecture 통일 X |
| 예측 정비 (predictive maintenance) framing — 비즈니스 가치 명확 | un-downgrade 인프라 (sensor_identifier 매칭 / MLModel.algorithm enum) 의 운영 복잡도 ↑ |

### 선택 이유
- **도메인 의존 결정 — 전력 = un-downgrade 필수, 가스 = 격하 충분:** [ai-model-study-2026-05-17.md](ai-model-study-2026-05-17.md) §2.4 / §3.4.
  - 전력: 점진 부하 증가·시간대 베이스라인·선행 경보 모두 비즈니스 핵심 → ARIMA 본질 가치 살려야 함.
  - 가스: 분 단위 즉시 위험 도메인 → ARIMA 의 trend / seasonal 가치 거의 없음 → IF 피처로 흡수.
- **5축 직교 결합의 해석가능성:** ARIMA 격하 시 ARIMA 기여를 추적 불가. un-downgrade + `algorithm_source` 라벨로 운영자 디버깅 가능.
- **시연 메시지 일관성:** "AI 가 어느 신호로 어떻게 잡았는지" 시연 시나리오와 직결.

### 후속 plan
- D+30: 운영 데이터로 ARIMA confusion matrix (precision / recall) 측정 → 동급 가중치 정당성 검증.
- 정당성 부족 시: ARIMA 격하 vs 가중치 조정 vs auto-arima 튜닝 옵션 비교.
- 가스도 un-downgrade 인프라 시점에 sensor_id 1개 가정 회피 정리.
