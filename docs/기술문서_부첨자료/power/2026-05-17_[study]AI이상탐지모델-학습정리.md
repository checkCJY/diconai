## 1. 현재 IF 구조

### 1.1 sensor_type 단일 매칭

```python
# fastapi-server/ai/router.py
async def _get_or_load(sensor_type: str):  # "power" 또는 "gas"

# fastapi-server/power/services/power_service.py
entry = await _get_or_load("power")  # ← hardcoded
```

**모든 device·channel·data_type 의 추론이 "power" 활성 모델 1개로 처리**. sensor_identifier (`power:device_1:ch1:watt`) 는 로깅·forward 라벨일 뿐 모델 선택에 안 쓰임.

`MLModel.is_active=True` 인 sensor_type 별 row 1건이 universal 모델.

### 1.2 가스 vs 전력 — feature engineering 만 다름

| 항목 | 전력 (현재) | 가스 (현재) |
|---|---|---|
| 입력 차원 | 1D univariate (단일 채널) | 3D multivariate (CO + H2S + CO2) |
| feature 수 | 4 | **15** (3 가스 × 5) ← 5번째는 `arima_resid` |
| 모델 input shape | (1, 4) | (1, 15) |
| feature builder | `_build_feature_row` | `_build_multi_feature_row(arima_results=...)` |
| ARIMA 사용 방식 | 미도입 | **격하 (IF 피처로 흡수)** — pkl은 가스명별 분리, MLModel row 없음 |
| 현재 학습된 sensor | `device_1, ch1, watt` 만 | CO+H2S+CO2 조합 (sensor_id 1개) |

**가스가 multivariate 인 이유**: 단일 가스만 보면 환기·날씨 같은 자연 변동을 이상으로 오인. CO+H2S+CO2 의 동시 패턴이 진짜 이상 신호 — 산업 안전 도메인 표준.

**가스가 ARIMA를 격하 채택한 이유**: 도메인 특성상 사고가 분 단위 즉시 위험(누출·화재). forecast·신뢰구간·seasonal 같은 ARIMA의 본질 가치 대부분이 가스에선 도메인 필요 없음. 잔차 크기 정보만 IF에 보태도 충분 → 격하가 정당. **단 이 선택은 가스 도메인 특성에 맞춤이지 일반 해결책 아님** (§2.4 참조).

### 1.3 4 features 의 의미

```python
arr = window_values[-30:]  # window=30 틱
value     = arr[-1]                    # 현재값
roll_mean = arr.mean()                 # 이동 평균
roll_std  = arr.std(ddof=0)            # 이동 표준편차
diff      = arr[-1] - arr[-2]          # 1차 차분
```

| feature | 잡는 이상 패턴 |
|---|---|
| `value` | 학습 분포 밖의 큰 값 — 갑자기 매우 높은 전력 |
| `roll_mean` | 점진적 상승·하강 trend — 모터 부하 증가 |
| `roll_std` | 변동 폭 증가 — 베어링 마모, 회로 노이즈 |
| `diff` | 단발 spike — 전력 surge |

**위치 + 분산 + 변화율** — 시간 시계열 기본 통계 요약.

### 1.4 `_INFERENCE_ENABLED_CHANNELS = {(1, "watt")}` 의 정체

- 본 sprint 학습 모델이 (device_1, ch1, watt) 분포만 학습됨
- 다른 채널 (ch2~16, current, voltage) 에 같은 모델 쓰면 분포 불일치 → false positive 폭주
- **코드 단 안전장치 (의도된 제약, 문제 X)**

## 2. 전력 AI 모델 도입 시 — 무엇이 변하는가

### 2.1 multivariate 확장 (전력 16 채널)

**현재**: ch1·watt 만 학습 (univariate, 4 features)
**확장**: 1 device 의 16 채널 동시 multivariate
- input: 16 channel × 4 = **64 features**
- 모델 input shape: (1, 64)
- 가스의 `_build_multi_feature_row` 패턴을 전력으로 확장

### 2.2 모델 수 — 매칭 단위에 따라 다름

| 접근 | 1 device 의 모델 수 |
|---|---|
| univariate (현재) | 채널 수만큼 = 16개 |
| **multivariate per device** | **1개** (전력 16 채널 흡수) |
| sensor_identifier 단위 (ARIMA 강제) | sensor 수만큼 = 16~ |

**multivariate 가 채널 차원은 흡수**. 모델 수 증가의 진짜 원인은 **device 수**:
- 10 device → 모델 10개
- 100 device → 모델 100개

### 2.3 ARIMA 도입 시 코드 변경 영역 — un-downgrade 경로 (정식 algorithm 분리)

ARIMA = 시계열 예측 모델. 다음 측정값의 예측값 + 신뢰구간 생성. 실측값이 신뢰구간 밖이면 알람 = **동적 임계치 위반** = 룰 알람의 변종.

framing: **정적 룰 / 동적 룰 (ARIMA) / IF (패턴)** 의 삼분 구조.

> **두 경로 존재**: 본 §2.3 은 ARIMA를 IF와 동급 algorithm으로 분리하는 **un-downgrade 경로**. §2.3-A 는 ARIMA를 IF 피처로 흡수하는 **격하 경로** (가스 채택안). 도메인에 따라 선택 — §2.4 의사결정 참조.

| 변경 | 위치 |
|---|---|
| `MLModel.algorithm` 필드 추가 (`isolation_forest` / `arima`) | apps/ml/models/ml_model.py |
| `_get_or_load` 분기 — algorithm 따라 sklearn vs statsmodels | fastapi-server/ai/router.py |
| inference 함수 분기 — IF score_samples vs ARIMA forecast+신뢰구간 | ai/router.py |
| AlarmType 신규 — `power_arima` (또는 룰 sub-type) | apps/core/constants.py |
| `train_arima_model` 신규 command | apps/ml/management/commands/ |
| 재학습 스케줄 (drift 큰 모델이라 정기) | config/celery beat |
| **Dedup/mute 정책 재설계** — IF·ARIMA 동시 발화 정책 | apps/alerts/services/alarm_dedupe.py + AI_TO_RULE_LEVEL |

**ARIMA 의 매칭 단위 강제**: ARIMA 는 단일 시계열 specific. "power" sensor_type 한 모델로 16 채널 처리 불가 → **반드시 sensor_identifier 단위 매칭** (`MLModel.sensor_identifier` 컬럼 추가 + `_get_or_load(sensor_identifier)` 전환).

### 2.3-A ARIMA 격하 경로 (가스 채택안)

ARIMA를 독립 algorithm이 아니라 **IF의 추가 피처 공급원**으로 사용. forecast/신뢰구간/AlarmType/dedup 모두 손대지 않고 잔차 1개만 IF 입력에 추가.

| 변경 | 위치 |
|---|---|
| `compute_arima_residuals()` + `build_multi_features(arima_results=...)` | apps/ml/services/feature_service.py |
| `train_arima_model` 신규 command (pkl 저장만, MLModel row 없음) | apps/ml/management/commands/ |
| 추론 측 모듈 import 시 ARIMA pkl 직접 로드 (router 우회) | fastapi-server/gas/services/gas_service.py |
| `_build_multi_feature_row(arima_results=...)` 매개변수 추가 | fastapi-server/ai/router.py |

**격하 경로의 trade-off**

| 보존되는 가치 | 손실되는 가치 |
|---|---|
| ✅ 잔차 크기 (스파이크형 이상 감지) | ❌ 신뢰구간 위반 판정 (정상 범위 내 미세 일탈) |
| ✅ IF 기존 파이프라인 재사용 | ❌ 점진적 trend break (누적 신호) |
| ✅ MLModel/router/AlarmType 무변경 | ❌ multi-step forecast (선행 경보) |
| ✅ Dedup 단순 (1 알람) | ❌ seasonal time-of-day 적응 |
| ✅ 시연 일정 영향 0 | ❌ 운영자 해석력 ("AI score -0.23" only) |

→ 격하 적합성은 **도메인이 위 손실 4가지를 필요로 하지 않을 때만** 정당화. §2.4 매트릭스 참조.

### 2.4 격하 vs un-downgrade — 도메인 의존 결정

**가스 도메인 — 격하 적합**

| 가스 이상 패턴 | 빈도 | ARIMA 특수 가치 필요? |
|---|---|---|
| 갑작스러운 CO 농도 ↑ (누출) | 다수 | ❌ 잔차 크면 IF가 즉시 잡음 |
| 점진적 O2 감소 (밀폐 환경) | 일부 | ⚠️ ramp_up dummy 시나리오로 학습 가능 |
| 야간/주간 베이스라인 변화 | 거의 없음 | ❌ 가스 농도는 시간 의존성 약함 |
| 1시간 앞 forecast 일탈 | 거의 없음 | ❌ 가스 사고는 분 단위 사건 |

→ 가스는 "즉시 위험 / 즉시 알람" 본질. ARIMA forecast·신뢰구간·seasonal 가치 대부분 도메인에서 불필요 → 격하 정당.

**전력 도메인 — un-downgrade 필수**

| 전력 이상 패턴 | 빈도 | ARIMA 특수 가치 필요? |
|---|---|---|
| 갑작스러운 부하 ↑ (단락) | 일부 | ❌ 잔차로 잡힘 |
| 모터 부하 점진 증가 (베어링 마모 초기) | 다수 | ✅ **누적 trend break — ARIMA 강점** |
| 시간대별 베이스라인 (주간 1500W vs 야간 200W) | 항상 | ✅ **seasonal — ARIMA 강점** |
| 1시간 앞 forecast 일탈 (선행 경보) | 다수 | ✅ **multi-step forecast — ARIMA 강점** |
| "곧 임계치 도달" 조기 경보 | **핵심 가치 제안** | ✅ **신뢰구간 위반 — ARIMA 본질** |

→ 전력은 **격하 시 사라지는 4가지 가치가 모두 비즈니스 핵심**. 특히 "예측 정비(Predictive Maintenance)" 비즈니스 framing 자체가 ARIMA forecast + 신뢰구간 위에 서있음 → 격하 시 무력화.

**결론 매트릭스**

| 도메인 | 사고 시간 척도 | ARIMA 도메인 필요성 | 격하 적합성 |
|---|---|---|---|
| 가스 | 분 단위 즉시 위험 | 낮음 | ✅ 적합 |
| 전력 | 시간~일 단위 점진 변화 | 높음 | ❌ 부적합 — **un-downgrade (§2.3) 채택** |

가스 격하는 도메인에 맞춘 최적화이지 일반 해결책이 아님. 전력에 그대로 가져오면 ARIMA가 잡으려던 가치 대부분을 잃음.

## 3. 도입 전 사전 확인 — 단점·리스크

### 3.1 핵심 리스크 3개

#### (1) DB 폭주

| 단계 | sensor 수 | 일 row | 1년 row |
|---|---|---|---|
| 현재 (ch1 watt only) | 1 | 2,880 | ~1M |
| 1 device, 16 채널 × 3 type | 48 | 138K | ~50M |
| 10 device, 480 sensor | 480 | 1.4M | ~500M |

**SQLite 로 불가** (이미 12GB 폭증 사례 있음). **PG 전환 + MLAnomalyResult TTL 7~30일** 필수.

#### (2) 모델 관리 운영 부담
- 모델 파일 1~2개 → 수십~수백개
- 학습·재학습 batch 시간 30초 → 수시간 (전체)
- 활성/비활성 관리, rollback 명령, staleness 모니터링 필요

#### (3) 추론 latency
- 현재 inline 추론 (Celery beat X, 데이터 도착 시점)
- asyncio 가 sklearn predict (CPU bound) 를 serial 처리
- 다수 sensor 동시 데이터 송출 → fastapi 응답 지연 → backpressure
- **완화**: inline → Celery background task

### 3.2 추론 시간 3초 목표 — 모델 수와 무관

```
fastapi 수신·validation       ~ 1 ms
DRF 데이터 저장 forward       ~ 50~200 ms (httpx async)
window 누적                   ~ 0.1 ms
모델 캐시 조회                ~ 0.5 ms (hit) / ~200 ms (miss)
IF.predict                    ~ 0.5~2 ms ← 모델 수와 무관
결과 forward                  ~ 50~200 ms
broadcast                     ~ 50~500 ms
                              ─────
                              ~ 200ms~1.3s 정상
```

3초 목표 달성 가능 — 모델 수 100개여도. 단 캐시 정책 + 네트워크 hop 최소화 + inline → background 화.

### 3.4 전력 단계별 모델 수 스케일 — un-downgrade 기준

전력 진행 단계별 모델 수가 어떻게 증가하는지 (un-downgrade 경로):

| 단계 | 범위 | IF | ARIMA | 합계 |
|---|---|---|---|---|
| 1 | 1채널 PoC (현재) | 1 | 0 | 1 |
| 2 | 1 device, 16 채널 | 1 (panel-multivariate) | 16 (channel별) | 17 |
| 3 | N devices (5~) | N | 16N | **17N** |

**ARIMA가 매칭 단위를 강제하는 이유**: ARIMA는 단일 시계열 specific. IF는 panel-multivariate 64-feature로 16채널을 한 모델에 흡수 가능하지만, ARIMA는 채널별로 분리 학습 강제. 따라서 device 수에 channel × 1(ARIMA) 곱이 그대로 모델 수.

**필수 인프라 (2단계 진입 전)**:
- `MLModel.sensor_identifier` 컬럼 + `_get_or_load(sensor_identifier)` 분기 — sensor_id 1개 가정 회피
- `MLModel.algorithm` 필드 — IF/ARIMA row 분리
- `MLAnomalyResult` TTL (DataRetentionPolicy 등록) — 17N 모델 × 추론 결과 누적 폭주 방지

**격하 경로 선택 시 모델 수**: N (IF only) + 16N pkl (운영자가 추적 안 하는 부속) = 운영 단위 N. 가스 패턴 그대로. 단 전력에선 §2.4 결론대로 격하 부적합.

## 4. 현재 코드의 quality_flag 처리 — 도입 시 확인용

### 4.1 통신 불능 (-1, null) — ✅ 전력 처리됨 / ⚠️ 가스 회귀

```python
# power_service.py
if value is None:                                          # 추론 skip
    continue
sensor_status = "comm_failure" if is_comm else "active"    # 분류
```

PowerData 모델:
- `value` nullable
- `sensor_status` ("active" / "comm_failure")
- `risk_level`, `is_anomaly`, `anomaly_type` 분리 필드

→ 전력 통신 불량 시나리오 추가 작업 없이 동작.

**⚠️ 가스 회귀**: `gas_service.process_gas_data()`는 `value is None` 가드가 없음. 통신 불능 시 None이 deque에 들어가 sklearn 입력 에러 가능. 전력 패턴(`if value is None: continue`)을 가스에도 백포팅 필요.

### 4.2 미처리 영역

| 항목 | 현재 코드 | 필요 작업 |

| 비정상 과대값 (99999) | ❌ | upper bound 검증 |
| 고정값 30분 지속 | ❌ | stuck detection (window std=0) |
| 시간 누락 (gap) | ❌ | measured_at gap detection |
| 중복 measured_at | 🔄 | UniqueConstraint 확인 필요 |

### 4.3 시나리오 라벨 필드 부재

## 5. 핵심 의사결정 포인트 — 도입 전 결정 필요

개발자 단독 판단 X. 비즈니스 요구 명확화 필요:

1. **ARIMA 격하 vs un-downgrade** ← **도메인 자동 답 있음** — 전력 = un-downgrade (§2.4). 가스 = 격하 유지. 도메인 특성이 결정 변수, 협의 불필요.
2. **device 수의 미래** — 5? 100? 1000? → 모델 수·저장 비용 결정 (전력 17N 스케일링)
3. **device 종류 다양성** — 같은 종류 묶어서 학습 가능?
4. **알람 누락 vs 오탐 — 어느 게 더 비싼가** → 모델 confidence 임계치 결정
5. **운영자 인력** — staleness 모니터링·rollback·재학습 누가?
6. **시뮬레이터 vs 실 IoT** — 운영 데이터 수집 후 일괄 재학습 시점

비즈니스 요구가 명확해지면 개발 결정도 명확해짐.

## 7. 전력 진행 단계 정리 — un-downgrade 채택 (도메인 결정 §2.4)

### 7.1 단계별 범위·모델 수

| 단계 | 시점 | 범위 | 모델 수 | 핵심 작업 |
|---|---|---|---|---|
| 1 | 현재 ~ D-27 (시연) | ch1·watt PoC | 1 (IF univariate) | dummy v3화·시연 시나리오·PoC 졸업 입증 |
| 2 | 시연 후 ~ D+30 | 1 device 16채널 | 17 (1 panel-multivariate IF + 16 channel ARIMA) | un-downgrade 정식 적용 (§2.3) |
| 3 | D+30 ~ | N devices | 17N | sensor_identifier 매칭 + 운영 도구 |

### 7.2 시연 (1단계, D-27까지)

**유지**: 현재 IF 1채널 PoC. ARIMA 신규 도입 X — 일정 리스크 회피.

**필수 작업**:
- `power_dummy` v3화 (가스 `_state_machine.py` 재사용) — motor_overload / voltage_sag 시나리오 정의 + `anomaly_type` 페이로드 동봉
- 시연 시나리오 1분 30초 리허설 — "정상 → 점진 상승 → IF 발화 → 룰 격상"
- 핸드오버 문서 — "1채널 PoC + 16채널 un-downgrade 확장 설계 검증 완료"

**시연 메시지 (정직 버전)**:
> "전력 1채널 IF PoC 결과 — 동작 검증 + 16채널 un-downgrade 확장 경로 설계 완료. 실데이터 누적 후 ARIMA 도입 + N device 스케일 확장 예정."

### 7.3 un-downgrade 전환 (2단계, 시연 후 ~ D+30)

§2.3 7가지 변경 영역 전체 적용:

1. `MLModel.algorithm` 필드 추가 + 마이그레이션
2. `MLModel.sensor_identifier` 컬럼 추가
3. `_get_or_load(sensor_type, algorithm, sensor_identifier)` 분기
4. 추론 함수 분기 — IF predict / ARIMA forecast+신뢰구간
5. `power_arima_forecast` AlarmType 신규 (또는 `power_anomaly_ai`에 `algorithm_source` 필드)
6. Dedup/mute 정책 재설계 — IF·ARIMA 동시 발화 처리
7. `MLAnomalyResult` TTL 정책 (DataRetentionPolicy)

**가스 동반 작업**: 가스도 동일 시점에 un-downgrade로 통일 — sensor_id 1개 가정 회피 + algorithm 분리 인프라 공유. 현재 격하 구현은 가스 사고 패턴엔 충분하지만 인프라 일관성 측면에서 정리 필요.

### 7.4 N device 확장 (3단계, D+90 ~)

- 17N 모델 운영 — `MLModel.sensor_identifier` 매칭으로 분포 분리
- 학습 batch 시간 ↑ → celery beat 정기 재학습 + 병렬 학습 지원
- 운영 도구: rollback 명령, staleness 모니터링, 모델 버전 추적 대시보드
- 인프라: ML 모델 저장소를 docker named volume → 영구 저장소 분리 검토

### 7.5 측정·검증 시점

- **PoC 졸업 측정** (D+0 ~ D-20): dummy 환경 FP rate, 룰 누락 감지율
- **실데이터 1차 측정** (D+30 ~ D+60): 클라이언트 환경 ACF/PACF, ARIMA 적합성 판정
- **운영 가시성 측정** (D+90 ~): IF·ARIMA 알람 비중·중복도·운영자 응답률
