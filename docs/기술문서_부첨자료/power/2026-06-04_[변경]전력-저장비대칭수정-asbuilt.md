# 전력 저장 비대칭 수정 — as-built (2026-06-04)

전력 데이터 저장이 가스와 비대칭이라는 점검에서 출발해, 두 건을 수정했다.
plan: [../plan/power-risk-writeback-fix.md](../plan/power-risk-writeback-fix.md)

## 배경 — 저장 레이어가 두 개

| 레이어 | 가스 | 전력 |
|---|---|---|
| A. 원시 측정 | `GasData` (wide, save()에서 위험도 재계산) | `PowerData` (long, **위험도 write-back 없음**) |
| B. ML 추론결과 | `MLAnomalyResult` (이상만 적재) | `MLAnomalyResult` (**정상까지 전부 적재**) |

점검에서 드러난 실측:
- `PowerData.risk_level` 387만 행 전부 `normal` → **① 대상**
- `MLAnomalyResult`(power) 정상행 333,682건이 아무 소비처 없이 적체 → **② 대상**

---

## ① PowerData.risk_level write-back

### 왜
`PowerData.risk_level`이 행 생성 후 한 번도 안 채워져 387만 행이 전부 `normal`.
위험도는 `trigger_power_alarms`에서 이미 계산되지만(`this_risk = eval_fn(...)`)
알람 발화에만 쓰고 행에는 반영하지 않았다. 가스는 `GasData.save()`에서 재계산해
채우지만, 전력은 `bulk_create`로 저장해 모델 `save()`/default를 우회하므로
가스식 `save()` override를 복사해도 작동하지 않는다.

### Before / After
- **Before:** FastAPI가 `risk_level="normal"` 하드코딩 송신 → `bulk_create`가 그대로
  저장 → 영구 normal. (위험도 계산값 `this_risk`는 알람 분기 후 버려짐)
- **After:** `trigger_power_alarms`가 위험행의 `risk_level`을 obj에 in-place 기록 →
  serializer가 위험행만 골라 `bulk_update`로 persist.

### 실제 코드
**`apps/monitoring/services/power_alarm.py` · `trigger_power_alarms`** (eval 직후)
```python
this_risk = eval_fn(value, channel=channel, device_id=device_id)
# 위험행만 obj 에 in-place 기록 — 호출자(serializer)가 persist.
if this_risk != RiskLevel.NORMAL:
    obj.risk_level = this_risk
aggregate = _aggregate_risk(device_id, channel, axis_name, this_risk)
```
**`apps/monitoring/serializers/power_data.py` · `create()`** (trigger 호출 직후)
```python
trigger_power_alarms(saved, device, ingress_ts=ingress_ts)
risk_rows = [o for o in saved if o.risk_level != RiskLevel.NORMAL]
if risk_rows:
    PowerData.objects.bulk_update(risk_rows, ["risk_level"])
return saved
```
- 기록값 = `this_risk`(행의 축 위험도), `aggregate`(3축 max) 아님 — PowerData 1행
  = (채널, 축) 단위라 watt 행엔 watt 축 위험도가 맞다.
- 위험일 때만 기록 → 정상 구간엔 `risk_rows` 빈 리스트 → **bulk_update 쿼리 0회**.

### 트러블슈팅 — 설계 1→2 피벗 (테스트 때문)
최초안은 `trigger_power_alarms` 안에서 `obj.risk_level`을 **읽고**(`!=` 가드)
직접 `bulk_update`까지 하려 했다. 그러나 `test_danger_confirm.py`의 dedup
단위테스트가 `SimpleNamespace(channel=1, value=4000.0, data_type="watt")`를
주입한다 — `risk_level` 속성이 없고 실제 `PowerData`도 아니다. 트리거 안에서
읽기/`bulk_update`를 하면:
- `obj.risk_level` 읽기 → `AttributeError` (SimpleNamespace에 속성 없음)
- `bulk_update([SimpleNamespace])` → 모델 인스턴스 아님 → 실패

→ **책임 분리**로 회피: 트리거는 `obj.risk_level = this_risk` **set만**(읽기 없음
→ SimpleNamespace에 새 속성 set은 허용됨), 영속화는 실제 `PowerData`(PK 보유)를
가진 serializer가 담당. 결과적으로 dedup 테스트는 **무수정 통과**.

### ⚠️ 활성화되는 부작용 — 작업자 지오펜스 전력 위험 알람 (의도된 잠재 수정)
`positioning/services/position_service.py:_get_dangerous_sensors_in_geofence`는
지오펜스 안 PowerDevice의 **최신 `PowerData.risk_level`**을 읽어 위험을 판정한다
(line 65-71). 그동안 risk_level이 항상 normal이라 전력 디바이스는 항상 `continue`로
빠져 **전력發 지오펜스 알람이 통째로 죽어 있었다**(가스만 실효).

①으로 risk_level이 채워지면 이 경로가 **되살아난다**:
```
작업자 위치 수신 → 지오펜스 30px 근접 → _get_dangerous_sensors_in_geofence
  → 전력 디바이스 최신행 risk_level=warning/danger 감지
  → fire_geofence_alarm_task.delay(...)  # 작업자 근접 위험 알람
```
- **flood 위험 없음:** 작업자가 해당 지오펜스 안/근접 + 디바이스 최신행이 위험일
  때만 발화 (작업자 존재로 자연 게이팅, 전력 알람 폭주와 다름).
- **자동 해소:** 위험 해제 시 새 행은 default normal → 최신행 normal → 판정 해제.
- 가스 지오펜스 경로(`max_risk_level`)와 이제 대칭.

### 영향·범위
- **forward-only.** 기존 387만 행은 그대로 normal. 신규 수신행부터 실제 위험도.
  과거 행 보정 필요 시 별도 backfill(범위 밖). `backfill_power_data.py`도 미경유.
- 성능: 위험행만 1 bulk_update. 정상 구간 0쿼리.
- 부수효과(positive): 전력 admin `risk_level` 필터가 유의미해짐.

## ② MLAnomalyResult 정상행 적재 중단 (잔해정리)

### 왜
`MLAnomalyResult`(power)에 정상 추론행(333,682건)이 쌓이는데, READ 경로가
보관기간 삭제(`data_retention_task`) 외엔 전무 — 라벨링 UI·재학습 입력·대시보드
어디서도 안 읽는 좀비 데이터였다(원래 "추론 실패→라벨링" 설계의 멈춘 잔해).
가스는 이상(`pred==-1`)만 적재해 대칭도 깨져 있었다.

### Before / After
- **Before:** `decide_alarm` 결과가 None이면(= AI normal + 정적 normal) ml_payload를
  `forward_inference_e2e`로 적재 → 정상행이 MLAnomalyResult에 쌓임.
- **After:** decision=None 분기에서 적재 생략(`continue`만). 이상행(caution↑)은
  그대로 적재.

### 실제 코드
**`fastapi-server/power/services/anomaly_inference.py`** (decide_alarm 직후)
```python
decision = decide_alarm(ai_state, ai_combined, static_risk)
if decision is None:
    # AI normal + 정적 normal = 순수 정상행 → 적재 생략 (소비처 부재 잔해).
    continue
```

### 안전성 근거 — 왜 정상행만 떨어지는가
`_FIRE_LEVELS = {caution, predict_warn, warning, danger}` (normal만 비-fire).
- combined가 _FIRE_LEVELS면 → `ai_state=FIRED` → `decide_alarm`이 "ai" 결정 반환
  (None 아님) → 적재 경로(rate-limit 459 / race 557)로 **그대로 적재**.
- 따라서 `decision is None` 분기엔 **순수 normal만 도달**. caution/predict_warn/
  warning/danger 행은 영향 없음. (분포 검증: normal 333,682만 감소, 나머지 유지)
- 정상 포함 전체 분포는 `POWER_AI_COMBINED_TOTAL` 메트릭(매 추론 카운트)에 이미
  집계 → "운영 추적" 목적도 행 적재 없이 충족. line 419 주석도 이에 맞게 정정.

### 영향·범위
- 신규 정상 추론부터 적재 중단. 기존 정상행은 보관정책 만료로 자연 정리.
- 가스(이상만 적재)와 대칭 회복.
- ml_model FK가 None인 건 별개 사안(전 행 공통) — 이번 범위 밖.
