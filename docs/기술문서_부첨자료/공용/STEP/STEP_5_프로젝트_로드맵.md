모델 개발 순서 가이드라인

```
질문할 때는 전체 AI 관제 시스템 만들어줘라고 묻지 말고,
현재 STEP 번호, 사용 중인 모듈 또는 모델명, 입력 데이터 구조, 원하는 결과를 함께 적어 질문하세요.

예를 들어 Threshold, Z-score, Sliding Window, Change Point는 대부분 규칙/통계/처리 구조에 가깝고, Isolation Forest와 ARIMA는 실제 모델에 가깝습니다.
```

질문예시:
```
현재 STEP D를 구현 중입니다.
Django 모델 GasReading에 device_id, measured_at, co, h2s, o2, voc 컬럼이 있습니다.
device_id별 최근 30개 데이터를 기준으로 z_score를 계산하고,
ANOMALY_WARNING 여부를 반환하는 service 함수를 만들어주세요.
```

STEP 1~4는
- 통계 기반 이상탐지
- 실시간 처리 구조
- 머신러닝 기반 이상탐지
- 상태 변화 탐지

같은 개별 기법과 처리 구조를 학습한 단계입니다.

STEP 5의 목적: 현재 구현 완료 상태

현재 프로젝트는 이미 다음 기능이 구현된 상태를 전제로 합니다.
- 관리자 페이지 구성
- 실시간 모니터링 대시보드 구성
- WebSocket 기반 테스트용 난수 데이터 스트리밍
- 9개 유해가스 데이터 화면 표시
- CO, H2S, CO2, O2, NO2, SO2, O3, NH3, VOC
- 스마트 파워 시스템의 전류, 전압, 전력 데이터 화면 표시
- 임계치 기반 위험 상태 표시 또는 위험 영역 UI 구성

따라서 STEP5는 대시보드 자체를 새로 만드는 문서가 아니라,
이미 구현된 실시간 관제 화면과 데이터 흐름 위에
통계/AI 기반 이상탐지와 미래 위험 예측 기능을 붙이기 위한 개발 가이드입니다.

즉, STEP5의 핵심 목표는 다음과 같습니다.

기존 실시간 데이터 수신/표시 구조
→ 원시 데이터 저장 및 시계열 정리
→ 임계치 판단 검증
→ Z-score / Change Point / Isolation Forest 기반 이상탐지
→ ARIMA 기반 단기 미래 예측
→ 예측 결과를 대시보드와 알람에 연결

---
이 과제에서 위험 판단 엔진은 보통 두 가지 역할을 수행해야 합니다.

첫째, 지금 위험한가?
- 현재 들어온 값이 위험 기준을 넘었는지
- 최근 정상 패턴에서 벗어났는지
- 급변이 시작되었는지

둘째, 곧 위험해질 가능성이 있는가?
- 다음 1~3 스텝 후 값이 위험 구간으로 갈 가능성
- 추세가 악화되는지
- 조기경보가 필요한지

이 기준으로 보면:
- 우리가 배웠던 STEP 1, 2, 3, 4는 주로 현재/최근 상태 판단
- ARIMA는 주로 가까운 미래 예측 입니다.

즉, 통계 기반 이상탐지와 ARIMA의 차이를 이상 여부 판단 vs 미래 예측으로 구분합니다.

#### 개발 핵심 구조

본 프로젝트의 목표는 실시간으로 들어오는 센서 데이터를 기준으로
현재 위험 여부를 판단하고, 이상 패턴을 탐지하며, 가까운 미래의 위험 가능성까지 예측하는
통계/AI 기반 위험 판단 엔진을 만드는 것입니다.

즉, 아래 기능들을 각각의 모듈 또는 모델로 나누어 개발한 뒤,  최종적으로 하나의 알람 판단 흐름으로 통합하는 구조입니다.

| 구분 | 개발 형태 | 역할 | 결과 |
|---|---|---|---|
| Threshold | 규칙 기반 판단 모듈 | 센서값이 미리 정한 위험 기준을 넘었는지 즉시 판단 | normal / warning / danger |
| Z-score | 통계 기반 이상탐지 모듈 | 최근 평균과 표준편차를 기준으로 평소보다 갑자기 튄 값을 탐지 | 정상 / 이상 |
| Sliding Window | 실시간 처리 구조 | 장비별·센서별 최근 N개 데이터를 계속 유지하면서 실시간 계산 가능하게 함 | 최근 데이터 묶음 |
| Change Point | 상태 변화 탐지 모듈 | 데이터 흐름이 예전과 달라지기 시작한 시점을 탐지 | 변화 감지 |
| Isolation Forest | 머신러닝 기반 이상탐지 모델 | 여러 센서값과 파생변수를 함께 보고 정상 패턴에서 벗어난 데이터를 탐지 | 정상 / 이상 |
| ARIMA | 시계열 예측 모델 | 다음 1~3분 뒤 센서값을 예측하여 미래 위험 가능성을 확인 | 예측값 |

정리하면 다음과 같습니다.

- Threshold는 가장 기본적인 안전장치입니다.
- Z-score는 통계적으로 튄 값을 찾는 모듈입니다.
- Sliding Window는 실시간 처리를 위한 데이터 유지 구조입니다.
- Change Point는 상태가 바뀌기 시작한 시점을 찾는 모듈입니다.
- Isolation Forest는 여러 센서 조합을 보고 이상 패턴을 찾는 머신러닝 모델입니다.
- ARIMA는 현재 이상을 찾는 모델이 아니라, 가까운 미래값을 예측하는 시계열 예측 모델입니다.

따라서 이 프로젝트는
Isolation Forest 하나로 이상탐지를 끝내는 구조가 아니라,
규칙 기반 판단, 통계 기반 판단, 머신러닝 기반 탐지, 시계열 예측을 단계적으로 조합하는 구조입니다.

개발 흐름은 아래와 같이 이해하면 됩니다.
```
센서 데이터 수신
→ DB 저장
→ Threshold 즉시 판단
→ Sliding Window로 최근 데이터 유지
→ Z-score 통계 이상탐지
→ Change Point 상태 변화 탐지
→ Isolation Forest 패턴 이상탐지
→ ARIMA 미래값 예측
→ 최종 알람 / 이벤트 / 대시보드 표시
```

### STEP A. 데이터 계약/저장 구조 확정

목적
기존 실시간 수집/대시보드 구조를 점검해서, 임계치 판단·통계 기반 이상탐지·AI 추론·예측 결과를 안정적으로 붙일 수 있는 입력 데이터 구조를 확정하는 단계

즉,
- 센서 데이터가 지금 어떻게 들어오고 있는지 확인하고
- 그 데이터가 임계치 판단, 통계 기반 탐지, AI/ML 모델 입력으로 바로 쓸 수 있는지 보고
- 부족한 필드와 저장 규칙만 보완하는 단계입니다

---
기존에 먼저 점검할 것
`1)` 현재 실시간으로 들어오는 데이터 형식 확인
`2)` 현재 대시보드의 “위험 예측” 영역이 무엇으로 계산되는지 확인
`3)` 원시 데이터가 저장되고 있는지 확인
`4)` 시간 기준이 정리되어 있는지 확인
`5)` 결측/에러값 규칙이 정리되어 있는지 확인

---
###### 1) 현재 실시간으로 들어오는 데이터 형식 확인

먼저 확인할 것은 현재 데이터 수신 방식입니다.
이때 테스트 환경과 실제 장비 연동 환경을 반드시 구분해서 확인합니다.

1. 테스트 환경 확인
	현재 개발 화면은 WebSocket 기반 테스트용 난수 데이터를 사용하고 있으므로 아래 내용을 확인합니다.

- 현재 대시보드가 WebSocket 기반 난수 스트리밍으로 동작하는가
- 난수 데이터의 필드명이 실제 센서 정의서의 필드명과 일치하는가
- 난수 데이터에 `device_id`가 포함되는가
- 난수 데이터에 측정 시각 또는 수신 시각이 포함되는가
- 난수 데이터가 DB에 저장되는가
- 아니면 화면 표시용으로만 흘러가는가
- 프론트 대시보드에서 지금 표시 중인 값의 원본이 무엇인가

2. 실제 장비 연동 환경 확인
	실제 장비 또는 게이트웨이와 연동할 때는 아래 내용을 확인합니다.

- 실제 센서 또는 게이트웨이 데이터가 HTTP API로 들어오는가
- 실제 프로토콜 문서 기준으로 `Content-Type: application/json` 구조인가
- 실제 payload에 어떤 필드가 포함되는가
- 실제 payload와 테스트용 WebSocket payload의 필드명이 일치하는가
- 실제 payload에 `device_id`가 항상 포함되는가
- 실제 payload에 측정 시각이 포함되는가
- 값이 비정상일 때 `null`, `-1`, 필드 누락 중 어떤 형태로 들어오는가

3. 정의서 기준 필드와 실제 payload 비교
	유해가스 센서는 정의서 기준으로 아래 9개 항목을 기본 필드로 봅니다.
```text
co, h2s, co2, o2, no2, so2, o3, nh3, voc
```
하지만 실제 payload에는 일부 항목만 들어올 수 있습니다.

예:
```
정의서 기준:
co, h2s, co2, o2, no2, so2, o3, nh3, voc

실제 수신 예시:
o2, co, co2, h2s, lel 등 일부 항목만 수신될 수 있음
```
따라서 STEP A에서는 아래 내용을 확정해야 합니다.

- 정의서 기준 필드와 실제 수신 필드가 일치하는가
- 실제 payload에 유해가스 9개 항목이 모두 들어오는가
- 일부 항목만 들어온다면 어떤 항목이 필수이고 어떤 항목이 선택인지 구분한다
- 테스트 WebSocket 데이터 구조와 실제 장비 연동 데이터 구조를 같은 schema로 맞춘다

즉, 현재 화면에 뿌리는 값의 원본이 무엇인지 확인하고,
테스트 데이터와 실제 장비 데이터가 AI 모델 입력으로 같은 구조를 갖도록 정리하는 단계입니다.

###### 2) 현재 대시보드의 위험 예측 영역이 무엇으로 계산되는지 확인

개발 구현 화면에
- 유해가스 위험 예측
- 스마트 전력 시스템 위험
- 예상 시간
- 예측 최대 농도/부하

같은 영역이 있으므로, 여기는 꼭 확인해야 합니다.

- 지금 이 값이 실제 AI 결과인지
- 아니면 임시 계산식인지
- 아니면 더미 데이터인지
- 프론트에서 계산하는지
- 백엔드에서 계산해서 보내는지

이걸 알아야 나중에
- 기존 값 교체
- 임계치 판단 결과, 통계 탐지 결과, AI/ML 결과, 예측 결과 연결
- 정책 엔진 연결을 어디서 해야 할지 결정할 수 있습니다.

###### 3) 원시 데이터가 저장되고 있는지 확인
AI를 붙이려면 실시간 표시만으로는 부족합니다.

확인할 것:
- 센서 수신 데이터가 DB에 저장되는가
- HTTP API, WebSocket, MQTT 등 어떤 방식으로 수신되든 최종적으로 DB에 저장되는가
- 저장된다면 어떤 테이블에 쌓이는가
- 최근 1분 단위 시계열이 계속 남는가
- 예측용 학습 데이터로 다시 꺼낼 수 있는가

이미 대시보드만 있고 저장이 약하면, AI는 못 붙입니다.
그래서 STEP A에서는 현재 들어오는 값을 학습/추론 가능한 형태로 저장하는 구조가 있는지를 점검해야 합니다

###### 4) 시간 기준이 정리되어 있는지 확인
AI에서 시간은 중요합니다.

확인할 것:
- 서버가 데이터를 받은 시각만 있는지
- 실제 센서가 측정한 시각도 있는지
- 1분 단위 전송 주기가 일정하게 유지되는지
- 누락/중복 데이터가 있는지
- 수신 시각과 측정 시각을 구분해서 저장할 수 있는지

ARIMA나 Change Point는 특히 시간축이 중요하므로, STEP A에서 이 부분을 반드시 확인해야 합니다.

###### 5) 결측/에러값 규칙이 정리되어 있는지 확인
기존 대시보드는 값만 보여주면 되지만, AI는 잘못된 값을 그대로 먹으면 안 됩니다.

확인할 것:
- 값이 없을 때 `null`인지
- `-1` 같은 특수값을 쓰는지
- 아예 필드가 빠지는지
- 통신 오류와 정상 0 값을 구분할 수 있는지

이건 나중에 `quality_flag` 같은 품질 상태 필드로 정리하면 됩니다.

###### 진행하는 실제 순서

1단계. 현재 센서 수신 방식과 payload 확인
- HTTP API, WebSocket, MQTT 등 실제 수신 방식을 확인한다.
- 실제 수신 payload를 캡처한다.
- `device_id`가 포함되는지 확인한다.
- 측정 시각 또는 수신 시각이 포함되는지 확인한다.
- 정의서 기준 필드와 실제 수신 필드가 일치하는지 확인한다.
- 값이 없거나 통신 불능일 때 어떤 값으로 들어오는지 확인한다.

2단계. 현재 대시보드 위험 영역의 계산 위치 확인
- 프론트에서 계산하는지 확인한다.
- 백엔드에서 계산해서 보내는지 확인한다.
- 더미 데이터인지 확인한다.
- 임시 계산식인지 확인한다.
- 이미 규칙 기반 판단이 들어가 있는지 확인한다.

3단계. 원시 데이터 저장 구조 확인
- 센서 수신 데이터가 DB에 저장되는지 확인한다.
- 저장된다면 어떤 테이블에 쌓이는지 확인한다.
- 1분 단위 시계열 데이터가 계속 누적되는지 확인한다.
- device_id별로 최근 데이터를 조회할 수 있는지 확인한다.
- AI/통계 탐지용 학습 데이터셋으로 다시 꺼낼 수 있는지 확인한다.

4단계. 유해가스 센서 입력 schema 확정

유해가스 센서용 기본 schema:
- `device_id`
- `measured_at`
- `received_at`
- `co`
- `h2s`
- `co2`
- `o2`
- `no2`
- `so2`
- `o3`
- `nh3`
- `voc`
- `quality_flag`
- `raw_payload`

주의:
실제 payload에 9개 항목이 모두 들어오지 않을 수 있으므로,
정의서 기준 필드와 실제 수신 필드를 비교하여 필수 필드와 선택 필드를 구분한다.

5단계. 스마트 파워 시스템 입력 schema 확정

스마트 파워 시스템용 기본 schema:
- `device_id`
- `measured_at`
- `received_at`
- `channel_no`
- `power_status`
- `current_a`
- `voltage_v`
- `power_w`
- `quality_flag`
- `raw_payload`

주의:
스마트 파워 시스템은 유해가스 센서와 데이터 구조가 다르므로,
유해가스 센서와 같은 테이블이나 같은 판단 로직으로 묶지 않는다.

6단계. 검증 규칙 확정
- 값이 없는 경우를 어떻게 처리할지 정한다.
- `-1` 같은 통신 불능 값을 어떻게 처리할지 정한다.
- 정상 0과 오류 0을 구분할 수 있는지 확인한다.
- 결측, 오류, 통신 불능은 `quality_flag`로 표시한다.

7단계. 기존 대시보드와 위험 판단 결과 연결 지점 정리
- 기존 위험 예측 카드에 어떤 API 결과를 붙일지 정한다.
- 임계치 판단 결과와 AI/통계 판단 결과를 어떻게 구분해서 보여줄지 정한다.
- normal / warning / danger / anomaly / predictive_warning 같은 상태값을 화면에서 어떻게 표현할지 정한다.

STEP A의 확인 순서는 아래와 같습니다.

센서 수신 방식 확인
→ 실제 payload 확인
→ 원시 데이터 저장 여부 확인
→ 유해가스 schema 확정
→ 스마트 파워 schema 확정
→ 결측/오류 검증 규칙 확정
→ 기존 대시보드 연결 흐름 확정

즉, STEP A는 모델을 만들기 전에
센서 데이터가 어떤 방식으로 들어오고, 어떤 필드로 저장되며,
결측/오류값을 어떻게 처리하고, 기존 대시보드와 어떻게 연결할지를 확정하는 단계입니다.

STEP A가 정리되어야 이후 STEP B~G의 임계치 판단, Z-score, Change Point, Isolation Forest, ARIMA를 안정적으로 붙일 수 있습니다.

---
### STEP B. 임계치 기반 즉시 알람

목적
센서에서 들어온 값을 기준으로, 미리 정해둔 임계치 규칙으로 위험 여부를 즉시 판단하고 알람을 발생시키는 것입니다.
이 부분은 3차 프로젝트에서 이미 일부 구현되어 있을 가능성이 있습니다.
따라서 STEP B에서는 새로 만드는 것뿐만 아니라, 기존 임계치 판정 로직이 제대로 동작하는지 점검하고 보완하는 것이 중요합니다.

즉, AI가 없어도
지금 이 값이 위험한가? 를 바로 판단할 수 있는 기본 안전장치를 만드는 단계입니다.

이 단계가 중요한 이유는 다음과 같습니다.
- 센서 데이터가 실제로 잘 들어오는지 확인할 수 있음
- 저장 후 판정까지 전체 흐름을 검증할 수 있음
- 위험 상황 발생 시 즉시 알람을 줄 수 있음
- 이후 AI 판단 결과와 비교할 기준이 생김
- AI가 없어도 최소한의 안전 기능이 동작함

이 단계는 예측을 하는 단계가 아니라,
현재 측정값을 기준과 비교해서 즉시 판정하는 단계입니다.

예를 들어:
- CO가 200 이상이면 위험
- H2S가 15 이상이면 위험
- O2가 16 미만이면 위험
- VOC가 1.0 이상이면 위험

이처럼 각 가스별 기준값을 정해두고,
센서값이 그 기준을 넘는지 확인해서
- 정상
- 주의
- 위험 중 어떤 상태인지 판단합니다.

센서값을 임계치와 비교해서 위험 여부를 규칙 기반으로 즉시 판정하는 단계 입니다.

---
실제 작업 순서 체크

`1)` 가스별 임계치 기준 정리
정의서 기준으로 각 가스의 정상 / 주의 / 위험 기준을 확정합니다.

먼저 정리해야 하는 것

| 가스  |         정상 기준 |             주의 기준 |    위험 기준 | 판정 방향  |
| --- | ------------: | ----------------: | -------: | ------ |
| CO  |         25 미만 |      25 이상 200 미만 |   200 이상 | 높으면 위험 |
| H2S |         10 미만 |       10 이상 15 미만 |    15 이상 | 높으면 위험 |
| CO2 |      1,000 미만 | 1,000 이상 5,000 미만 | 5,000 이상 | 높으면 위험 |
| O2  | 18 이상 23.5 이하 |       16 이상 18 미만 |    16 미만 | 낮으면 위험 |
| NO2 |          3 미만 |         3 이상 5 미만 |     5 이상 | 높으면 위험 |
| SO2 |          2 미만 |         2 이상 5 미만 |     5 이상 | 높으면 위험 |
| O3  |       0.06 미만 |   0.06 이상 0.12 미만 |  0.12 이상 | 높으면 위험 |
| NH3 |         25 미만 |       25 이상 35 미만 |    35 이상 | 높으면 위험 |
| VOC |        0.5 미만 |     0.5 이상 1.0 미만 |   1.0 이상 | 높으면 위험 |

이건 코딩 전에 기준을 확정하는 작업입니다.

`2)` 센서 데이터 저장 구조 준비
센서에서 받은 값을 DB에 저장할 수 있게 테이블을 준비합니다.

예:
- device_id
- measured_at
- received_at
- co
- h2s
- co2
- o2
- no2
- so2
- o3
- nh3
- voc
- quality_flag
- raw_payload

이 작업은 원본 측정값 저장 구조를 만드는 작업입니다.

`3)` 임계치 판정 로직 구현
저장된 센서값을 읽어서 임계치와 비교하는 로직을 만듭니다.

예시:
```python
def judge_gas_threshold(data):
    danger_gases = []
    warning_gases = []

    # CO
    if data.co >= 200:
        danger_gases.append("CO")
    elif data.co >= 25:
        warning_gases.append("CO")

    # H2S
    if data.h2s >= 15:
        danger_gases.append("H2S")
    elif data.h2s >= 10:
        warning_gases.append("H2S")

    # CO2
    if data.co2 >= 5000:
        danger_gases.append("CO2")
    elif data.co2 >= 1000:
        warning_gases.append("CO2")

    # O2 - 낮으면 위험
    if data.o2 < 16:
        danger_gases.append("O2")
    elif data.o2 < 18:
        warning_gases.append("O2")

    # NO2
    if data.no2 >= 5:
        danger_gases.append("NO2")
    elif data.no2 >= 3:
        warning_gases.append("NO2")

    # SO2
    if data.so2 >= 5:
        danger_gases.append("SO2")
    elif data.so2 >= 2:
        warning_gases.append("SO2")

    # O3
    if data.o3 >= 0.12:
        danger_gases.append("O3")
    elif data.o3 >= 0.06:
        warning_gases.append("O3")

    # NH3
    if data.nh3 >= 35:
        danger_gases.append("NH3")
    elif data.nh3 >= 25:
        warning_gases.append("NH3")

    # VOC
    if data.voc >= 1.0:
        danger_gases.append("VOC")
    elif data.voc >= 0.5:
        warning_gases.append("VOC")

    if danger_gases:
        status = "danger"
    elif warning_gases:
        status = "warning"
    else:
        status = "normal"

    return {
        "status": status,
        "danger_gases": danger_gases,
        "warning_gases": warning_gases,
    }
```

`4)` 판정 결과 기록
판정 결과를 남깁니다.

예:
- overall_status = danger
- danger_gases = `["co", "voc"]`
- message = "CO, VOC 위험 기준 초과"

이 작업은 판정 결과를 저장하거나 반환하는 구조를 만드는 작업입니다.

`5)` 알람 상태 생성 및 갱신
주의 또는 위험으로 판정되면 알람 상태를 생성하거나 갱신합니다.

예:
- 어떤 장비인지
- 어떤 가스 때문인지
- 현재값이 얼마인지
- 기준값이 얼마인지
- warning인지 danger인지
- 언제 처음 발생했는지
- 마지막으로 언제 감지되었는지
- 현재 상태가 open인지 closed인지

주의할 점은 센서 데이터가 1분마다 계속 들어오기 때문에,
위험 상태가 지속된다고 해서 매번 새 AlarmEvent를 만들면 안 됩니다.

따라서 다음 흐름으로 처리합니다.

- 같은 장비 + 같은 가스 + 같은 알람 유형의 열린 알람이 없으면 새 AlarmEvent 생성
- 이미 열린 알람이 있으면 last_seen_at, current_value만 갱신
- 값이 정상 범위로 돌아오면 기존 AlarmEvent를 closed 처리

`6)` 화면 반영
위험 알람이 발생하면 모니터링 화면에서 해당 위치를 표시합니다.

예:
- 위험 셀 빨간색 표시
- 센서 상태 아이콘 경고 표시

이 작업은 백엔드 판정 결과를 프론트 화면과 연결하는 작업입니다.

센서 데이터가 들어오면 이렇게 흘러가야 합니다.
```
센서 데이터 수신
↓
DB 저장
↓
임계치 판정 함수 실행
↓
normal / warning / danger 상태 결정
↓
warning 또는 danger이면 AlarmEvent 생성 또는 기존 open 알람 갱신
↓
정상 복귀 시 기존 AlarmEvent closed 처리
↓
대시보드에 현재 상태와 알람 이력 표시
```

| 체크 항목                                                         | 되어 있으면 |
| ------------------------------------------------------------- | ------ |
| 센서값이 DB에 저장됨                                                  | 통과     |
| CO, H2S, CO2, O2, NO2, SO2, O3, NH3, VOC 9개 항목의 임계치 판정 로직이 있음 | 통과     |
| normal / warning / danger 결과가 저장되거나 API로 반환됨                  | 통과     |
| warning 또는 danger 발생 시 AlarmEvent가 생성되거나 기존 open 알람이 갱신됨      | 통과     |
| 정상 복귀 시 AlarmEvent가 closed 처리됨                                | 통과     |
| 현재 알람 상태와 이력이 화면에 표시됨                                         | 통과     |

단, 스마트 파워 시스템은 유해가스 센서와 같은 방식으로 고정 임계치만 적용하지 않습니다.

유해가스 센서는 정의서에 정상 / 주의 / 위험 임계치가 명확히 있으므로 고정 임계치 기반으로 우선 판단합니다.

반면 스마트 파워 시스템은 유해가스처럼 정상/주의/위험 임계치가 정의된 구조가 아니므로,
전력 사용량 위험도는 시간대별 평균 사용량, 직전 대비 증가량, 비정상적인 지속 사용 여부를 기준으로 별도 판단합니다.

다만 장비 보호 관점에서는 카탈로그에 제시된 전압 범위, 최대 허용 전류, 최대 전력 범위도 함께 확인해야 합니다.
즉, 운영 위험도 판단은 패턴 기반으로 하되, 장비 한계값은 별도의 안전 제한값으로 관리합니다.

따라서 STEP B의 임계치 기반 즉시 알람은 우선 유해가스 센서에 적용하고,
스마트 파워 시스템은 별도의 전력 사용 패턴 기반 판단 로직으로 분리합니다.

---
### STEP C. 실시간 처리 뼈대(Sliding Window)

실제 적용 흐름은 `센서 수신 → Sliding Window → Z-score / Change Point / Isolation Forest → 알람 엔진` 순서입니다.

즉 STEP C는 직접 이상을 판단하는 모델이 아니라,
STEP D, STEP E, STEP F가 계산할 수 있도록 장비별·센서항목별 최근 N개 데이터를 유지하는 실시간 처리 구조입니다.

STEP C의 목적
센서 데이터가 들어올 때마다 전체 데이터를 다시 계산하지 않고,
장비별·센서항목별 최근 N개 데이터만 유지하면서
이상탐지 계산을 즉시 수행할 수 있는 실시간 처리 구조를 만드는 것

쉽게 말하면 대시보드는 지금 현재값을 보여주고 있습니다.
그런데 AI/통계 탐지는 현재값 하나만 보면 안 됩니다.
예를 들어 CO 값이 지금 `180`이라고 해도:
```
평소 CO가 10 근처였는데 180이면 위험
평소 CO가 170~180 근처에서 계속 움직였으면 다른 판단 필요
```
그래서 최근 데이터 묶음이 필요합니다.
이 최근 데이터 묶음이 바로 Sliding Window입니다.


STEP C에서 실제로 만들어야 하는 것

`1.` device_id별 버퍼
센서 장비가 여러 개라면 장비마다 따로 최근 데이터를 관리해야 합니다.

예:
```
gas_sensor_1의 CO 최근 30개
gas_sensor_1의 H2S 최근 30개
gas_sensor_2의 CO 최근 30개
gas_sensor_2의 H2S 최근 30개
```
즉, 전체 센서를 하나로 섞으면 안 됩니다.

구조는 이렇게 생각하면 됩니다.
```python
buffers = {
    "gas_sensor_1": {
        "co": [최근 30개],
        "h2s": [최근 30개],
        "co2": [최근 30개],
        "o2": [최근 30개],
        "no2": [최근 30개],
        "so2": [최근 30개],
        "o3": [최근 30개],
        "nh3": [최근 30개],
        "voc": [최근 30개],
    },
    "gas_sensor_2": {
        "co": [최근 30개],
        "h2s": [최근 30개],
        "co2": [최근 30개],
        "o2": [최근 30개],
        "no2": [최근 30개],
        "so2": [최근 30개],
        "o3": [최근 30개],
        "nh3": [최근 30개],
        "voc": [최근 30개],
    }
}
```
초기 구현에서는 Python 메모리의 `dict`나 `deque`로 버퍼를 만들 수 있습니다.

다만 운영 환경에서는 서버가 재시작되면 메모리 버퍼가 사라질 수 있습니다.
따라서 실제 서비스에서는 다음 중 하나로 보완하는 것이 좋습니다.
- DB에서 최근 N개를 다시 조회해서 계산
- Redis에 device_id + metric 기준으로 최근 N개 저장
- 메모리 버퍼를 사용하되, 재시작 시 DB에서 초기화

초기 MVP:
- Python 메모리 버퍼 사용 가능

운영 고도화:
- Redis 또는 DB 기반 최근 N개 조회 구조 권장


`2.` 센서 항목별 버퍼
가스 센서는 CO, H2S, CO2, O2, NO2, SO2, O3, NH3, VOC 총 9개 항목이 있습니다.

각 항목은 기준과 성격이 다릅니다.

예:
```
CO  → 높으면 위험
H2S → 높으면 위험
O2  → 낮으면 위험
VOC → 높으면 위험
```

그래서 `가스 센서 전체 최근 30개`가 아니라, 가스 항목별 최근 30개가 필요합니다.
```
CO 최근 30개
H2S 최근 30개
CO2 최근 30개
O2 최근 30개
NO2 최근 30개
SO2 최근 30개
O3 최근 30개
NH3 최근 30개
VOC 최근 30개
```
---
`3.` 버퍼가 찼을 때만 계산
최근 30개 기준으로 판단한다면, 처음 1~29개는 아직 계산하면 안 됩니다.

왜냐하면 평균/표준편차가 불안정하기 때문입니다.
```
1개 들어옴 → 아직 판단 보류
2개 들어옴 → 아직 판단 보류
...
30개 들어옴 → 이제 계산 가능
31개 들어옴 → 가장 오래된 것 제거 후 최근 30개로 계산
```
WINDOW개가 쌓이기 전에는 판단하지 않고, WINDOW개가 채워진 후 최근 N개로 통계를 계산하는 구조입니다.

실제 코드는 이런 흐름으로 갑니다.
```
1. 센서 데이터 1건 수신
2. DB에 원본 저장
3. device_id 확인
4. 각 센서 항목 값 추출
5. device_id + 센서항목 기준으로 Sliding Window 버퍼에 추가
6. 버퍼 크기가 N보다 크면 가장 오래된 값 제거
7. 버퍼가 N개 미만이면 통계/AI 판단 보류
8. 버퍼가 N개 이상이면 STEP D의 Z-score 계산 가능
9. 버퍼가 2W개 이상이면 STEP E의 Change Point 계산 가능
10. 필요한 feature가 준비되면 STEP F의 Isolation Forest 예측 가능
11. 각 결과를 알람 판단 엔진으로 전달
12. 상태가 경고/위험으로 바뀌면 ENTER_ALERT 기록
13. 정상 복귀하면 EXIT_ALERT 기록
```
즉, STEP C는 아래 구조입니다.
```
센서 수신
   ↓
원본 저장
   ↓
최근 N개 버퍼 갱신
   ↓
계산 가능 여부 확인
   ↓
STEP D: Z-score 계산
   ↓
STEP E: Change Point 계산
   ↓
STEP F: Isolation Forest 예측
   ↓
탐지 결과 통합
   ↓
알람 상태 변화 확인
   ↓
알람 이벤트 기록
```

###### STEP C 완료 기준
| 체크 항목    | 완료 기준                                             |
| -------- | ------------------------------------------------- |
| 장비별 버퍼   | device_id마다 따로 최근 N개 유지                           |
| 항목별 버퍼   | CO, H2S, CO2, O2, NO2, SO2, O3, NH3, VOC 각각 따로 유지 |
| 계산 시점    | N개가 찼을 때만 계산                                      |
| 통계 결과    | mean, std, z_score 계산 가능                          |
| 변화 탐지 준비 | 이전 구간과 최근 구간을 비교할 수 있도록 2W개 데이터 유지 가능             |
| ML 예측 준비 | Isolation Forest 입력 feature 생성 가능                 |
| 상태 관리    | NORMAL / ALERT 상태 유지                              |
| 이벤트 기록   | ENTER_ALERT / EXIT_ALERT 발생                       |
| API 연결   | 센서 수신 후 자동 실행                                     |

정리하면 STEP C는 센서 데이터가 들어올 때마다 장비별·항목별 최근 N개를 유지하고,
그 데이터를 이용해 STEP D, STEP E, STEP F의 계산과 알람 상태 전환을 가능하게 만드는 실시간 처리 구조입니다.

---
### STEP D. 통계 기반 조기 이상탐지(Z-score)

STEP D의 목적
가스 센서값이 법적/정의서상 위험 임계치를 넘지 않았더라도,
평소 자기 장비의 정상 패턴과 비교해서 비정상적으로 튄 값을 조기 감지하는 것
즉, STEP D는 상대적 이상 탐지입니다.

###### STEP B와 STEP D의 차이
|구분|STEP B 임계치 기반|STEP D 통계 기반 이상탐지|
|---|---|---|
|판단 기준|고정 기준값|최근 N개 데이터의 평균/표준편차|
|질문|위험 기준을 넘었나?|평소보다 많이 튀었나?|
|예시|CO 200 이상이면 위험|평소 CO가 20인데 갑자기 80이면 이상|
|성격|절대 위험 판단|상대적 이상 판단|
|장점|설명 쉽고 즉시 판단|조기 이상 징후 감지 가능|
왜 STEP D가 필요한가?
	예를 들어 CO2 위험 기준이 `5000ppm`이라고 가정해보겠습니다.
	현재 CO2가 `900ppm`이면 임계치 기준으로는 위험이 아닐 수 있습니다.
	하지만 평소 해당 장비의 CO2가 계속 `400~450ppm` 수준이었다면?
```
평소: 400 ~ 450
현재: 900
```
이건 아직 법적 위험은 아니어도, 평소 패턴에서 크게 벗어난 이상 신호입니다.
즉 STEP D는 이런 상황을 잡기 위한 단계입니다.
위험 기준 초과 전, 이상 징후를 먼저 발견하는 단계입니다.

---
실제로 해야 하는 작업

`1.` 적용할 가스 항목 정하기
	먼저 어떤 항목에 통계 기반 이상탐지를 적용할지 정합니다.
```
co
h2s
co2
o2
no2
so2
o3
nh3
voc
```
처음부터 9개 전체를 다 해도 되지만, 실습 단계에서는 먼저 3~4개만 해도 됩니다.

초기 추천:
```
co
h2s
co2
o2
voc
```

`2.` 최근 N개 데이터를 가져올 구조 확인
	STEP D는 단일 현재값만으로는 계산할 수 없습니다.
	반드시 최근 데이터 묶음이 필요합니다.
	그래서 STEP C에서 만든 Sliding Window가 필요합니다.
```
device_id별 최근 N개
+
가스 항목별 최근 N개
```

예:
```
gas_sensor_1의 co 최근 30개
gas_sensor_1의 h2s 최근 30개
gas_sensor_1의 co2 최근 30개
gas_sensor_1의 o2 최근 30개
```
이 최근 N개를 기준으로 평균과 표준편차를 계산합니다.

`3.` Rolling Mean 계산
	Rolling Mean은 최근 N개 평균입니다.

예를 들어 CO 최근 5개가 다음과 같다면:
```
[20, 22, 21, 23, 24]
```
평균은 약 `22`입니다.
이 값이 해당 장비의 최근 정상 기준선이 됩니다.
최근 평균 = 현재 장비의 최근 정상 수준

`4.` Rolling Std 계산
	Rolling Std는 최근 N개 값이 얼마나 흔들리는지입니다.

예:
```
[20, 21, 20, 22, 21]
```
이 값들은 거의 안 흔들립니다.

반대로:
```
[20, 40, 15, 50, 25]
```
이 값들은 많이 흔들립니다.
표준편차는 이 흔들림 정도를 숫자로 나타냅니다.
그래서 같은 증가라도 표준편차에 따라 판단이 달라집니다.

평소 거의 안 흔들리는 장비 → 작은 변화도 이상 가능
평소 많이 흔들리는 장비 → 어느 정도 변화는 정상 가능

`5.` Z-score 계산
	Z-score는 현재값이 최근 평균에서 얼마나 떨어져 있는지 계산하는 점수입니다.

개념은 이렇습니다.
```
z_score = (현재값 - 최근 평균) / 최근 표준편차
```

예:
```
최근 평균: 100
최근 표준편차: 5
현재값: 120
```

그러면:
```
z_score = (120 - 100) / 5 = 4
```
`z_score`가 4라는 뜻은, 현재값이 평소보다 꽤 많이 튀었다는 의미입니다.

---
STEP D의 판단 기준
여기서는 절대 위험과 상대적 이상을 분리해서 봐야 합니다.

1순위: 임계치 위험
	먼저 STEP B의 임계치 기준을 봅니다.

예:
```
CO >= 200 → CRITICAL
H2S >= 15 → CRITICAL
O2 < 16 → CRITICAL
VOC >= 1.0 → CRITICAL
```
이 경우는 Z-score 계산 결과와 상관없이 위험입니다.

임계치 위험 = 즉시 CRITICAL

---
2순위: Z-score 이상
임계치는 넘지 않았지만, 평소보다 많이 튄 경우입니다.

예:
```
CO 위험 기준: 200
현재 CO: 80
평소 CO: 20
z_score: 4.2
```
이 경우 CO 80은 임계치상 위험은 아니지만, 평소보다 너무 튄 값입니다.

따라서:
```
ANOMALY_WARNING
```
으로 표시할 수 있습니다.

---
3순위: 정상
임계치도 넘지 않고, Z-score도 크지 않다면 정상입니다.
```
NORMAL
```
---
최종 판단 구조

실제 판단 순서는 이렇게 가는 것이 좋습니다.
1. 센서 데이터 수신
2. 임계치 기준 먼저 확인
3. 최근 N개 데이터로 평균/표준편차 계산
4. 현재값의 z-score 계산
5. 최종 상태 결정

판단 로직은 이렇게 정리할 수 있습니다.
```
임계치는 아니지만 abs(z_score) >= 3이면
→ ANOMALY_WARNING

다만 O2는 낮아질수록 위험한 항목이므로,
z_score의 절댓값뿐 아니라 방향도 함께 기록합니다.

예:
- CO, H2S, CO2, VOC: 값이 높아지는 방향이 주로 위험
- O2: 값이 낮아지는 방향이 위험

따라서 결과 저장 시 `z_score`와 함께 `direction`을 기록하면 좋습니다.

예:
{
"metric": "o2",
"z_score": -3.8,
"direction": "decrease",
"stat_status": "ANOMALY_WARNING"
}
```
---
예시

예시 1. 임계치 위험
```
CO 현재값: 230
CO 위험 기준: 200
```

결과:
```
CRITICAL
```
이건 평소 패턴을 볼 필요 없이 즉시 위험입니다.

---
예시 2. 상대적 이상
```
CO 현재값: 80
CO 위험 기준: 200
최근 평균: 20
최근 표준편차: 10
z_score: 6
```

결과:
```
ANOMALY_WARNING
```
위험 기준은 넘지 않았지만, 평소보다 너무 튀었기 때문입니다.

---
예시 3. 정상
```
CO 현재값: 25
최근 평균: 22
최근 표준편차: 3
z_score: 1
```

결과:
```
NORMAL
```

STEP D의 실제 작업 순서

1단계. STEP B 임계치 판정 로직 재사용
새로 만들 필요는 없습니다.
이미 STEP B에서 만든 임계치 판단 함수를 사용합니다.

2단계. STEP C Sliding Window에서 최근 N개 가져오기
여기서 최근 N개가 충분히 쌓였는지 확인합니다.
```
N개 미만이면 통계 판단 보류
N개 이상이면 계산 시작
```

3단계. 각 가스별 평균/표준편차 계산
```python
mean = np.mean(values)
std = np.std(values)
```

이 계산을 가스 항목별로 반복합니다.
```
MVP에서는 먼저 아래 5개 항목부터 적용할 수 있습니다.
co_mean, co_std
h2s_mean, h2s_std
co2_mean, co2_std
o2_mean, o2_std
voc_mean, voc_std

전체 적용 시에는 정의서 기준 9개 항목으로 확장합니다.
co, h2s, co2, o2, no2, so2, o3, nh3, voc
```

4단계. 각 가스별 Z-score 계산
```python
z_score = (current_value - mean) / (std + 1e-9)
```
결과는 이렇게 저장할 수 있습니다.
```
co_zscore
h2s_zscore
co2_zscore
o2_zscore
voc_zscore
```

5단계. 최종 상태 결정
판단 순서는 반드시 이렇게 가는 것이 좋습니다.
```
1순위: 임계치 위험
2순위: Z-score 이상
3순위: 정상
```

6단계. 결과 저장 또는 반환
STEP D 결과는 저장하거나 API 응답에 포함해야 합니다.
```python
{
    "device_id": "gas_sensor_1",
    "metric": "co",
    "value": 80,
    "mean": 20,
    "std": 10,
    "z_score": 6.0,
    "threshold_status": "normal",
    "stat_status": "ANOMALY_WARNING",
    "final_status": "ANOMALY_WARNING",
    "measured_at": "2026-01-15 12:00:00"
}
```
이 결과는 나중에 대시보드, 알람, AI 비교, 로그 분석에 사용됩니다.

STEP D에서 만들어야 하는 결과물

|결과물|설명|
|---|---|
|가스 항목 목록|어떤 센서값을 검사할지|
|최근 N개 조회 구조|STEP C Sliding Window 재사용|
|평균 계산|항목별 최근 평균|
|표준편차 계산|항목별 최근 변동성|
|Z-score 계산|현재값의 상대적 이상 정도|
|상태 판정|NORMAL / ANOMALY_WARNING / CRITICAL|
|결과 저장/API 반환|화면과 알람 엔진에서 사용|

완료기준
```
센서값이 들어온다
↓
최근 N개 평균/표준편차를 계산한다
↓
현재값의 z-score를 계산한다
↓
임계치 위험과 z-score 이상을 분리해서 판단한다
↓
NORMAL / ANOMALY_WARNING / CRITICAL 결과가 나온다
↓
결과가 저장되거나 대시보드/API에 표시된다
```
STEP D는 기존 임계치 알람을 대체하는 단계가 아니라, 임계치에 걸리기 전 평소보다 이상하게 튄 값을 잡아내는 1차 통계 기반 조기경보 단계입니다.

---
### STEP E. 상태 변화 감지(Change Point)

STEP E의 목적
가스 농도가 임계치를 넘기 전이라도,
평균 수준이나 변동성이 예전과 달라지기 시작한 시점을 찾아
위험 전조를 조기에 감지하는 것으로 STEP E는 상태 변화 감지입니다.

주의할 점은 Change Point가 발생했다고 해서 반드시 즉시 위험이라는 뜻은 아닙니다.
Change Point는 “상태가 바뀌기 시작했다”는 신호입니다.

따라서 최종 알람에서는 다음과 같이 사용합니다.

- 임계치 초과 + Change Point 발생 → 위험 근거 강화
- Z-score 이상 + Change Point 발생 → 조기경보 근거 강화
- Change Point만 발생 → 관찰 또는 주의 이벤트로 기록

예를 들어 CO2가 아래처럼 움직인다고 보겠습니다.
```
기존: 400, 410, 420, 405, 415
최근: 650, 680, 700, 720, 740
```
아직 위험 임계치까지는 아니어도, 분명히 상태가 바뀌었습니다.

이때 STEP E는 이렇게 판단합니다.
```
이 장비/공간의 CO2 평균 수준이 이전보다 올라가기 시작했다.
```
이게 Change Point입니다.

왜 유해가스 관제에서 중요한가?
```
CO2가 서서히 상승한다
VOC가 계속 높은 평균으로 유지된다
O2가 조금씩 낮아진다
H2S의 변동성이 갑자기 커진다
```
이런 경우는 임계치를 바로 넘지 않아서 STEP B가 못 잡을 수 있고,
한 순간 크게 튄 값이 아니라서 STEP D의 Z-score도 약하게 나올 수 있습니다.

그래서 Change Point가 필요합니다.
STEP E는 위험이 터진 순간보다 위험해지는 흐름이 시작된 순간을 찾는 단계입니다.

---
STEP E에서 실제로 해야 하는 작업
STEP E는 STEP C의 Sliding Window 구조를 그대로 사용합니다.

다만 STEP D가 최근 N개 안에서 평균/표준편차를 계산했다면,
STEP E는 최근 데이터를 두 구간으로 나눠서 비교합니다.
```
이전 구간(prev)
최근 구간(curr)
```
예를 들어 W=30이면:
```
이전 30개 vs 최근 30개
```
즉 총 60개가 있어야 계산할 수 있습니다.

---
###### STEP E의 핵심 계산 방식

`1.` 이전 구간 평균 계산
```
prev_mean = 이전 30개의 평균
```

`2.` 최근 구간 평균 계산
```
curr_mean = 최근 30개의 평균
```

`3.` 평균 차이 계산
```
mean_shift_score = |curr_mean - prev_mean| / prev_std
```
이 값이 크면 평균 수준이 바뀐 것입니다.

예:
```
이전 평균: 400
최근 평균: 700
이전 표준편차: 50
mean_shift_score = 6
```
이건 상태가 크게 바뀐 것입니다.

`4.` 변동성 변화 계산
평균만 보는 게 아니라 흔들림도 봅니다.
```
std_ratio = curr_std / prev_std
```

예:
```
이전 표준편차: 5
최근 표준편차: 30
std_ratio = 6
```
이건 값의 흔들림이 갑자기 커진 것입니다.
즉, 상태가 불안정해졌다고 볼 수 있습니다.

---
###### STEP E의 판단 기준
초기 기준은 단순하게 가면 됩니다.
```
mean_shift_score >= 3.0 이면 변화 감지
또는
std_ratio >= 3.0 이면 변화 감지
또는
std_ratio <= 1/3.0 이면 변화 감지
```

쉽게 말하면:
```
평균이 크게 달라졌거나
변동성이 크게 커졌거나
변동성이 갑자기 작아졌다면
상태 변화로 본다
```

---
###### STEP E 작업 순서

1단계. Change Point 적용 항목 정하기
처음부터 모든 항목에 적용해도 되지만, 우선 중요한 항목부터 시작하는 게 좋습니다.

초기 추천:
```
co
co2
h2s
o2
voc
```
특히 Change Point는 이런 항목에 잘 맞습니다.

|항목|보는 이유|
|---|---|
|CO2|환기 문제, 체류 인원 증가 감지|
|VOC|유해 물질 누출/작업 환경 변화|
|O2|산소 농도 저하 흐름 감지|
|CO|연소/누출 관련 변화 감지|
|H2S|위험 가스 증가 흐름 감지|

2단계. 비교 구간 크기 정하기
Change Point는 두 구간 비교가 핵심입니다.

예:
```
W = 30
이전 30개 vs 최근 30개
```

1분 주기라면:
```
이전 30분 vs 최근 30분
```

초기 테스트는 작게 잡아도 됩니다.
```
테스트용: W = 10
운영 시작용: W = 30
안정화 후: W = 60
```

주의할 점은 W=30이면 최소 60개 데이터가 있어야 합니다.
```
STEP D는 30개 있으면 계산 가능
STEP E는 60개 있어야 계산 가능
```


3단계. device_id + metric별 최근 2W개 유지
STEP C에서는 최근 N개를 유지했습니다.
STEP E에서는 비교를 위해 최근 `2W개`가 필요합니다.

예:
```
deque(maxlen=2 * W)
```

구조는 이렇게 갑니다.
```
gas_sensor_1 / co  / 최근 60개
gas_sensor_1 / co2 / 최근 60개
gas_sensor_1 / o2  / 최근 60개
```


4단계. 이전 구간과 최근 구간 나누기
최근 60개가 쌓이면 이렇게 나눕니다.
```
prev = 앞쪽 30개
curr = 뒤쪽 30개
```

예:
```
values = list(buffer)

prev = values[:W]
curr = values[W:]
```


5단계. 평균과 표준편차 계산
각 구간별로 계산합니다.
```
prev_mean = np.mean(prev)
prev_std = np.std(prev)

curr_mean = np.mean(curr)
curr_std = np.std(curr)
```
이 값들이 Change Point 판단의 기본 재료입니다.


6단계. 변화 점수 계산
평균 변화:
```
mean_shift_score = abs(curr_mean - prev_mean) / (prev_std + 1e-9)
```

변동성 변화:
```
std_ratio = (curr_std + 1e-9) / (prev_std + 1e-9)
```


7단계. 상태 변화 여부 판단
```
is_change = (
    mean_shift_score >= 3.0
    or std_ratio >= 3.0
    or std_ratio <= 1 / 3.0
)
```
이 결과가 True면 상태가 바뀌었다고 봅니다.


8단계. 상태 머신으로 이벤트 기록
```
Change Point도 단순히 True/False만 찍으면 부족합니다.
```

중요한 건:
```
언제 CHANGE 상태로 들어갔는가?
언제 다시 STABLE로 돌아왔는가?
```

그래서 상태를 관리합니다.
```
STABLE → SHIFT
SHIFT → STABLE
```

이벤트는 이렇게 기록합니다.
```
CHANGE_POINT
BACK_TO_STABLE
```

예:
```
2026-01-15 12:30 / gas_sensor_1 / co2 / CHANGE_POINT
2026-01-15 13:10 / gas_sensor_1 / co2 / BACK_TO_STABLE
```

---
### STEP F. Isolation Forest

STEP F는 앞 단계들과 달리 실제 머신러닝 모델을 사용하는 단계입니다.

Isolation Forest는 실시간 데이터가 들어올 때마다 매번 새로 학습하는 구조가 아닙니다.

개발 구조는 두 단계로 나눕니다.

1. 학습 단계
- 과거 정상 또는 정상에 가까운 데이터를 모음
- value, rolling mean, rolling std, diff 같은 feature를 구성
- Isolation Forest 모델을 fit
- 학습된 모델을 저장

2. 실시간 예측 단계
- 새 센서 데이터 수신
- STEP C/D에서 계산한 feature 생성
- 저장된 Isolation Forest 모델로 predict 수행
- 결과가 -1이면 ML_ANOMALY로 기록

즉, 실시간에서는 fit이 아니라 predict를 수행합니다.

STEP B, D, E는 대부분 규칙/통계 기반입니다.
반면 STEP F의 Isolation Forest는 정상 데이터의 패턴을 모델이 학습한 뒤, 그 패턴에서 벗어난 데이터를 이상으로 판단하는 비지도 학습 기반 이상탐지입니다.

---
STEP F의 목적
단일 센서값 하나만 보는 것이 아니라,
여러 센서값과 파생 특징을 함께 보고
평소와 다른 “복합 패턴 이상”을 감지하는 것

즉, STEP F는 이런 질문을 합니다.
```
각 값은 정상 범위 안에 있는데,
전체 조합 패턴도 정상인가?
```

예를 들어보겠습니다.
```
CO = 정상
O2 = 정상
VOC = 정상
CO2 = 정상
```
각각 따로 보면 모두 정상일 수 있습니다.

그런데 평소에는 이런 조합이 없었다면?
```
CO는 조금 상승
O2는 조금 하락
VOC도 함께 상승
CO2도 지속 상승
```
개별 값은 임계치를 넘지 않았지만, 조합 자체는 이상한 상황일 수 있습니다.

이걸 잡기 위해 Isolation Forest를 붙입니다.

---
STEP F가 필요한 이유
	앞 단계들은 각각 한계가 있습니다.

|단계|잘 잡는 것|한계|
|---|---|---|
|STEP B Threshold|명확한 위험 기준 초과|기준 이하의 이상 패턴은 놓칠 수 있음|
|STEP D Z-score|순간적으로 튄 값|여러 센서 조합 이상은 약함|
|STEP E Change Point|평균/분산 변화|복잡한 다변량 패턴 판단은 약함|
|STEP F Isolation Forest|여러 값의 조합 이상|학습 데이터와 feature 설계 필요|

---
STEP F에서 잡고 싶은 상황
예시 1. 개별 값은 정상인데 조합이 이상한 경우
```
CO = 80 정상
O2 = 18.5 정상
VOC = 0.4 정상
CO2 = 900 정상 또는 주의 전 단계
```
각각만 보면 큰 문제는 아닙니다.

하지만 평소 해당 공간에서는 이런 조합이 거의 없었다면, Isolation Forest는 이 데이터를 이상 패턴으로 볼 수 있습니다.

예시 2. 센서들이 동시에 움직이는 경우
```
CO2가 상승한다
O2가 하락한다
VOC가 상승한다
```
각각은 임계치 미만이어도, 동시에 이런 방향으로 움직이면 작업 환경이 바뀌었거나 환기 문제가 생겼을 수 있습니다.


예시 3. 단순 Z-score로는 오탐이 많은 경우
특정 센서는 원래 흔들림이 큽니다.

이 경우 Z-score는 자주 경고를 낼 수 있습니다.

Isolation Forest는 여러 feature를 함께 보면서 정상적인 흔들림인지, 진짜 특이한 패턴인지를 더 정교하게 판단할 수 있습니다.

---
STEP F의 전체 작업 흐름
	Isolation Forest의 작업 흐름은 크게 이렇게 갑니다.
```
1. 학습 데이터 준비
2. feature 구성
3. 모델 학습 fit()
4. 실시간 데이터에 대해 predict()
5. 정상/이상 결과 저장
6. 알람 엔진과 대시보드에 연결
```
학습 자료에서도 Isolation Forest는 `fit → predict` 구조로 설명됩니다. `fit(X)`로 데이터 분포를 학습하고, `predict(X)` 결과가 `1`이면 정상, `-1`이면 이상으로 판단합니다.

---
STEP F 작업 순서

1단계. 적용 대상을 정합니다
	처음부터 모든 센서를 대상으로 하지 말고, 우선 유해가스 데이터부터 적용하는 것이 좋습니다.

초기 적용 항목 예:
```
co
h2s
co2
o2
voc
```

이후 확장:
```
co
h2s
co2
o2
no2
so2
o3
nh3
voc
```
처음에는 주요 위험도와 관련이 큰 항목만 사용하면 됩니다.

---
2단계. 학습 데이터를 준비합니다
	Isolation Forest는 “정상/이상 라벨”은 필요 없지만, 학습할 데이터 X는 필요합니다.

예를 들어 DB에서 최근 정상 구간 데이터를 가져옵니다.
```
최근 7일 데이터
또는
최근 30일 데이터
또는
운영자가 정상이라고 판단한 기간의 데이터
```
중요한 점은 학습 데이터에 이미 위험 데이터가 너무 많이 섞이면 모델이 위험 패턴까지 정상으로 배울 수 있다는 것입니다.

그래서 처음에는 이렇게 잡는 것이 좋습니다.
```
명확한 임계치 위험 데이터는 제외
센서 오류값 제외
결측값 제외
정상 운영 시간대 데이터 우선 사용
```

---
3단계. 원본 feature를 만듭니다
	가장 기본 feature는 가스 원본값입니다.

예:
```
co
h2s
co2
o2
voc
```

이걸 표로 만들면 이런 형태입니다.

|measured_at|co|h2s|co2|o2|voc|
|---|---|---|---|---|---|
|12:00|20|1|420|20.8|0.2|
|12:01|22|1|430|20.7|0.2|
|12:02|25|2|440|20.6|0.3|

Isolation Forest는 이런 수치형 테이블을 입력으로 받습니다.

---
4단계. 파생 feature를 추가합니다
	원본값만 넣어도 되지만, 실무에서는 파생값이 중요합니다.

추천 feature는 다음과 같습니다.

|feature|의미|
|---|---|
|`co`|현재 CO 값|
|`co_diff`|직전 값 대비 CO 변화량|
|`co_roll_mean`|최근 N개 CO 평균|
|`co_roll_std`|최근 N개 CO 변동성|
|`co2`|현재 CO2 값|
|`co2_diff`|CO2 변화량|
|`o2`|현재 O2 값|
|`o2_diff`|O2 변화량|
|`voc`|현재 VOC 값|
|`voc_diff`|VOC 변화량|

예를 들어 `diff`는 이런 역할입니다.
```
현재값 자체는 정상인데,
변화 속도가 빠른가?
```

`rolling_mean`은 이런 역할입니다.
```
최근 평균 수준이 높아졌는가?
```

`rolling_std`는 이런 역할입니다.
```
최근 값이 불안정하게 흔들리는가?
```
---
5단계. 관계 feature를 추가합니다
	여러 센서 조합 이상을 잡으려면 관계값도 유용합니다.

예:
```
co_o2_ratio = co / o2
co2_o2_ratio = co2 / o2
voc_diff = 현재 voc - 이전 voc
co2_trend = 최근 co2 증가 방향
```

이런 feature는 다음 상황을 잡는 데 도움 됩니다.
```
CO는 조금 올라가고
O2는 조금 내려가고
CO2는 같이 올라가는 조합
```
즉, 단일 값이 아니라 관계의 변화를 모델이 볼 수 있게 해줍니다.

---
feature 구성 예시
	최초 버전은 너무 복잡하게 가지 말고 이렇게 시작하면 됩니다.

```
원본값:
co, h2s, co2, o2, voc

변화량:
co_diff, h2s_diff, co2_diff, o2_diff, voc_diff

최근 평균:
co_roll_mean, h2s_roll_mean, co2_roll_mean, o2_roll_mean, voc_roll_mean

최근 표준편차:
co_roll_std, h2s_roll_std, co2_roll_std, o2_roll_std, voc_roll_std
```

즉, 총 feature는 이런 구조입니다.
```
현재값 + 변화량 + 최근 평균 + 최근 변동성
```
처음부터 너무 많은 feature를 넣기보다, 위 정도로 시작하는 것이 좋습니다.

---
6단계. 학습용 데이터 X를 만듭니다
	모델에 넣을 컬럼을 정합니다.

```python
FEATURES = [
    "co", "h2s", "co2", "o2", "voc",
    "co_diff", "h2s_diff", "co2_diff", "o2_diff", "voc_diff",
    "co_roll_mean", "h2s_roll_mean", "co2_roll_mean", "o2_roll_mean", "voc_roll_mean",
    "co_roll_std", "h2s_roll_std", "co2_roll_std", "o2_roll_std", "voc_roll_std",
]
```
그리고 결측값을 제거합니다.
```
df_model = df.dropna().copy()
X = df_model[FEATURES]
```

왜 `dropna()`가 필요하냐면, `diff()`나 `rolling()`을 만들면 초반 데이터에는 계산할 수 없는 값이 생기기 때문입니다.

예:
```
첫 번째 diff는 이전 값이 없으므로 NaN
rolling_mean은 N개가 쌓이기 전까지 NaN
```


7단계. Isolation Forest 모델을 학습합니다
	초기 모델은 이렇게 만듭니다.

```python
from sklearn.ensemble import IsolationForest

model = IsolationForest(
    contamination=0.01,
    random_state=42
)

model.fit(X)
```
여기서 `contamination=0.01`은 전체 데이터 중 이상치가 약 1% 정도 있을 것이라고 가정한다는 뜻입니다.

처음에는 보통 이렇게 시작합니다.
```
0.01 = 이상치 1% 가정
0.03 = 이상치 3% 가정
0.05 = 이상치 5% 가정
```
너무 높게 잡으면 이상치가 많이 잡히고, 너무 낮게 잡으면 거의 안 잡힐 수 있습니다.

초기 추천은 다음과 같습니다.
```
처음 테스트: 0.01
민감하게 보고 싶으면: 0.03
오탐이 많으면: 0.005 또는 0.01 유지
```

---
8단계. 예측합니다
	학습된 모델로 예측합니다.
```python
labels = model.predict(X)
```

결과는 이렇게 나옵니다.
```python
1  = 정상
-1 = 이상
```

예:
```python
df_model["iforest_label"] = model.predict(X)
df_model["iforest_is_anomaly"] = df_model["iforest_label"] == -1
```
---
9단계. 이상 결과를 저장합니다
	결과는 나중에 대시보드와 알람에서 사용해야 하므로 저장하는 것이 좋습니다.

저장 예:
```
device_id
measured_at
model_name
features
score
label
is_anomaly
reason
created_at
```

초기에는 최소한 이 정도만 저장해도 됩니다.
```
device_id
measured_at
iforest_label
iforest_is_anomaly
```

추가로 가능하면 anomaly score도 저장합니다.
```python
df_model["iforest_score"] = model.decision_function(X)
```
일반적으로 score가 낮을수록 더 이상에 가깝다고 해석할 수 있습니다.

---
10단계. 알람 엔진과 연결합니다
	Isolation Forest 결과는 최종 위험도 판단에 들어갑니다.

예:

|조건|최종 상태|
|---|---|
|임계치 위험|`CRITICAL`|
|Z-score 이상|`ANOMALY_WARNING`|
|Change Point 발생|`TREND_SHIFT`|
|Isolation Forest 이상|`ML_ANOMALY`|
|모두 아님|`NORMAL`|

우선순위는 이렇게 둘 수 있습니다.
```python
CRITICAL > ML_ANOMALY > ANOMALY_WARNING > TREND_SHIFT > NORMAL
```

또는 운영 초기에는 ML 결과가 오탐이 있을 수 있으므로 이렇게 낮춰도 됩니다.
```python
CRITICAL > ANOMALY_WARNING > TREND_SHIFT > ML_ANOMALY > NORMAL
```

처음에는 ML 결과를 바로 위험 알람으로 쓰기보다 **참고 경고**로 두는 것이 안전합니다.
```
Isolation Forest 이상 감지
→ AI 이상 패턴 감지로 표시
→ 운영자가 확인
→ 오탐 여부 검토
```
---
실시간 적용시 구조
	실시간에서는 매번 새로 학습하면 안 됩니다.

운영 흐름은 이렇게 가야 합니다.
```
과거 정상 데이터로 모델 학습
   ↓
학습된 모델 저장
   ↓
실시간 센서 데이터 수신
   ↓
최근 N개로 feature 생성
   ↓
model.predict()
   ↓
ML_ANOMALY 여부 판단
   ↓
알람/대시보드 반영
```
즉, 실시간 수신 때마다 `fit()`을 하는 것이 아니라, 보통은 미리 학습한 모델로 `predict()`만 합니다.

예를 들면:
```
fit(): 하루 1회 또는 수동 재학습
predict(): 센서 데이터가 들어올 때마다
```
---
STEP F 완료 기준
```
정상 구간 데이터가 준비된다
   ↓
원본값 + 파생값 feature가 만들어진다
   ↓
Isolation Forest 모델이 fit() 된다
   ↓
새 데이터에 대해 predict()가 된다
   ↓
1 / -1 결과가 나온다
   ↓
-1이면 ML_ANOMALY로 기록된다
   ↓
대시보드 또는 이벤트 현황에서 확인된다
```
아래가 되면 STEP F는 완료입니다.

---
주의할점
1. 처음부터 알람 기준으로 강하게 쓰지 마세요. 왜냐하면 Isolation Forest는 오탐이 있을 수 있습니다.
초기에는 이렇게 쓰는 것이 좋습니다.
```
AI 참고 경고
AI 이상 패턴 감지
운영자 확인 필요
```
바로 긴급 알람으로 쓰면 오탐 때문에 신뢰도가 떨어질 수 있습니다.

2. feature가 모델 성능을 좌우합니다
	Isolation Forest 자체보다 더 중요한 것은 feature입니다.
```
원본값만 넣으면 단순 이상 탐지
diff/rolling_mean/rolling_std를 넣으면 흐름 기반 탐지
비율/관계값을 넣으면 조합 이상 탐지
```
즉, STEP F의 핵심은 모델 코드보다 어떤 feature를 넣을 것인가입니다.


3. 모델은 주기적으로 재학습해야 합니다
	센서 환경은 계속 바뀔 수 있습니다.
```
계절 변화
작업 공정 변화
환기 환경 변화
센서 노후화
```
그래서 모델은 한 번 학습하고 끝내는 것이 아니라 주기적으로 다시 학습하는 구조가 필요합니다.

초기에는 이렇게 잡으면 됩니다.
```
수동 재학습 버튼
또는 하루 1회 재학습
또는 주 1회 재학습
```

STEP F는 임계치·Z-score·Change Point로 잡기 어려운 여러 센서 조합의 이상 패턴을 비지도 학습으로 탐지하는 단계입니다.

---
### STEP G. ARIMA 예측

ARIMA의 목적
지금 당장 위험하지는 않지만,
최근 흐름을 보면 몇 분 뒤 위험 기준에 도달할 가능성이 있는지 예측하는 것

주의:
ARIMA는 이상데이터를 직접 추출하는 모델이 아닙니다.
ARIMA는 다음 값을 예측하는 시계열 예측 모델입니다.

따라서 ARIMA의 결과는 반드시 다시 임계치 또는 정책 엔진에 넣어서 판단해야 합니다.

ARIMA 예측값
→ 주의/위험 임계치와 비교
→ PREDICTIVE_WARNING 또는 PREDICTIVE_ALERT 생성

예를 들어 현재 CO2가 위험 기준은 아니지만, 주의 기준에 가까워지고 있다고 가정합니다.
```
현재 CO2 = 850
CO2 주의 기준 = 1,000
CO2 위험 기준 = 5,000
```
현재는 위험이 아닙니다.

그런데 ARIMA가 이렇게 예측한다면?
```
1분 뒤 CO2 = 920
2분 뒤 CO2 = 980
3분 뒤 CO2 = 1040
```

그러면 지금 바로 CRITICAL은 아니지만, 이렇게 알람을 줄 수 있습니다.
```
PREDICTIVE_WARNING
3분 뒤 CO2 주의 기준 초과 예상
```
이게 ARIMA의 역할입니다.

---
그러나 ARIMA만으로 부족합니다.

`1.` ARIMA는 단일 시계열 예측에 강함
ARIMA는 보통 이런 질문에 적합합니다.
```
이 센서의 다음 값은 얼마일까?
```

예:
```
device_A의 CO2 다음 3분 예측
device_A의 VOC 다음 3분 예측
device_A의 O2 다음 3분 예측
```

하지만 이런 질문에는 약합니다.
```
CO, O2, VOC 조합이 평소와 다른가?
```
이건 Isolation Forest 같은 다변량 이상탐지가 더 적합합니다.

---
`2.` 센서마다 모델을 따로 둬야 함
ARIMA는 보통 센서별/가스별로 따로 관리해야 합니다.

예:
```
device_A_co2_arima
device_A_o2_arima
device_A_voc_arima
device_B_co2_arima
device_B_o2_arima
device_B_voc_arima
```
센서가 많아질수록 운영 부담이 커집니다.

---
`3.` 결측/노이즈에 민감함
ARIMA는 시간 간격이 일정해야 합니다.

그런데 실제 센서 데이터는 이런 문제가 생깁니다.
```
1분 데이터 누락
통신 지연
센서 재부팅
갑작스러운 오류값
측정값 튐
```
이걸 전처리하지 않으면 예측 품질이 떨어집니다.

---
`4.` 위험 판단 모델은 아님
ARIMA는 기본적으로 “다음 값”을 예측합니다.

즉, 직접적으로 이런 판단을 하는 모델은 아닙니다.
```
정상인가?
이상인가?
위험인가?
```
그래서 예측값을 다시 임계치나 정책 엔진에 넣어야 합니다.
```
ARIMA 예측값
↓
위험 임계치와 비교
↓
PREDICTIVE_ALERT 판단
```

---
그래서 AI 모델과 병행해야 합니다. 다만 순서가 중요합니다.
바로 ARIMA + Isolation Forest + 모든 모델을 한 번에 붙이면 복잡도가 너무 커집니다.

추천 구조는 아래입니다.
```
STEP B 임계치
   ↓
STEP C Sliding Window
   ↓
STEP D Z-score
   ↓
STEP E Change Point
   ↓
STEP F Isolation Forest
   ↓
STEP G ARIMA
```

이유는 간단합니다.

|구분|역할|
|---|---|
|Threshold|절대 위험 감지|
|Z-score|순간 튐 감지|
|Change Point|상태 변화 시작 감지|
|Isolation Forest|여러 센서 조합 이상 감지|
|ARIMA|가까운 미래 위험 예측|
즉, ARIMA는 미래 예측 담당이고, Isolation Forest는 복합 패턴 이상탐지 담당입니다.
둘은 경쟁 관계가 아니라 역할이 다릅니다.

---
전체 구조는 이렇게 보는 게 좋습니다.
```
센서 스트리밍 데이터 수신
   ↓
원시 데이터 저장
   ↓
임계치 위험 판단
   ↓
Sliding Window에 최근 N개 누적
   ↓
Z-score로 순간 이상 여부 판단
   ↓
Change Point로 상태 변화 여부 판단
   ↓
이미 학습된 Isolation Forest 모델로 복합 패턴 이상 여부 예측
   ↓
ARIMA로 가까운 미래값 예측
   ↓
정책 엔진에서 최종 상태 결정
   ↓
대시보드 / 문자 / 디스코드 / 경광등
```
여기서 ARIMA는 마지막 쪽에 붙습니다.

왜냐하면 ARIMA는 현재 위험을 판단하는 게 아니라, 앞으로 위험해질 가능성을 보는 단계이기 때문입니다.

```
Isolation Forest
= 과거 정상 데이터로 미리 학습
= 실시간 데이터가 들어오면 predict만 수행

ARIMA
= 과거 시계열로 미리 적합/학습
= 실시간에서는 다음 1~3분 예측만 수행
```

---
STEP G에서 실제로 해야 하는 작업

1단계. ARIMA 적용 대상을 줄입니다
처음부터 9개 가스를 전부 하지 마세요.

초기 추천은 3개 정도입니다.
```
CO2
O2
VOC
```
이유는 이 항목들이 추세 변화나 환경 변화와 연결해서 설명하기 좋기 때문입니다.

예:
```
CO2 상승 → 환기 문제 가능성
O2 하락 → 산소 부족 위험 가능성
VOC 상승 → 유해물질 증가 가능성
```

처음 MVP에서는 이렇게 해도 충분합니다.
```
device 1개 + CO2 1개만 ARIMA 적용
```

---
2단계. 학습용 데이터셋을 만듭니다
	ARIMA용 데이터는 단순해야 합니다.

필수 컬럼은 이 정도입니다.
```
device_id
gas_type
measured_at
value
```

###### 예:
| device_id | gas_type | measured_at | value |
| --------- | -------- | ----------- | ----- |
| device_A  | co2      | 10:00       | 420   |
| device_A  | co2      | 10:01       | 430   |
| device_A  | co2      | 10:02       | 440   |
또는 가스별 컬럼 구조라면 이렇게 꺼내서 사용합니다.
```
device_id
measured_at
co2
```

ARIMA는 보통 **device_id + gas_type 단위로 독립 모델**을 둡니다.
```
device_A + co2 모델
device_A + o2 모델
device_A + voc 모델
```

---
3단계. 전처리를 합니다
	ARIMA에서 전처리는 매우 중요합니다.

해야 할 작업은 다음입니다.
```
시간순 정렬
1분 단위로 resampling
누락된 시간 보정
이상한 오류값 제거
센서 재부팅 직후 값 제외
결측값 처리
훈련/검증 기간 분리
```

예를 들어 1분 단위 데이터라면 반드시 이런 식으로 정리되어야 합니다.
```
10:00
10:01
10:02
10:03
10:04
```
중간에 10:02가 빠져 있으면 ARIMA가 불안정해질 수 있습니다.

---
4단계. 정상성 확인 또는 차분 여부 결정
	ARIMA는 시계열이 안정적인지 보는 과정이 필요합니다.

쉽게 말하면 이런 질문입니다.
```
값이 계속 한 방향으로 증가하는가?
평균 수준이 계속 바뀌는가?
계절성이나 반복 패턴이 있는가?
차분이 필요한가?
```

초보자용으로는 너무 깊게 들어가기보다 이렇게 정리하면 됩니다.
```
값이 추세를 보이면 d=1부터 시도
값이 안정적이면 d=0도 가능
```
즉, 처음에는 복잡한 이론보다 `auto_arima`로 후보를 찾고, 이후 고정 파라미터로 운영하는 방식이 현실적입니다.

---
5단계. 모델 학습을 합니다
	ARIMA도 넓은 의미에서는 학습 또는 적합을 합니다.

다만 딥러닝처럼 대량의 라벨 데이터를 학습하는 구조가 아니라,
과거 시계열 패턴에 맞게 통계 모델을 적합하는 방식입니다.

개념은 이렇습니다.
```
과거 CO2 시계열
↓
ARIMA 모델 적합
↓
미래 1~3분 값 예측 가능
```

초기 전략은 이렇게 잡으면 됩니다.
```
개발/분석 단계: auto_arima로 p,d,q 후보 찾기
운영 단계: 선택된 p,d,q를 고정해서 사용
```
운영 중 매번 auto_arima를 돌리면 무겁습니다.

---
6단계. 예측합니다
실시간에서는 새 데이터가 들어올 때마다 다음 값을 예측합니다.
```
최근 N개 CO2 데이터 추출
↓
학습된 ARIMA 모델 로드
↓
1분 뒤 / 2분 뒤 / 3분 뒤 예측
↓
예측값과 임계치 비교
```

예:
```
현재 CO2 = 850
1분 뒤 예측 = 920
2분 뒤 예측 = 980
3분 뒤 예측 = 1040
```

주의 기준이 1000이라면:
```
3분 뒤 주의 기준 초과 예상
→ PREDICTIVE_ALERT
```

---
7단계. 예측 결과를 저장합니다
ARIMA 결과는 따로 저장하는 것이 좋습니다.

예: `arima_forecasts`

|컬럼|의미|
|---|---|
|`device_id`|장비|
|`gas_type`|가스 종류|
|`forecast_time`|예측을 수행한 시각|
|`target_time`|예측 대상 시각|
|`predicted_value`|예측값|
|`threshold_value`|기준값|
|`threshold_cross_expected`|위험 초과 예상 여부|
|`model_version`|모델 버전|

이렇게 저장해두면 나중에 실제값과 비교할 수 있습니다.
```
예측값 1040
실제값 1010
→ 예측이 어느 정도 맞았는지 평가 가능
```

---
8단계. 예측 알람을 생성합니다
ARIMA는 바로 위험 알람을 발생시키기보다 **예측 경고**로 분리하는 것이 좋습니다.

예:
```
PREDICTIVE_ALERT
3분 뒤 CO2 위험 기준 초과 예상
```

임계치 초과 알람과는 다르게 처리합니다.

|알람|의미|처리|
|---|---|---|
|`CRITICAL`|이미 위험|즉시 조치|
|`WARNING`|이상 징후|확인 필요|
|`ML_ANOMALY`|복합 패턴 이상|검토 필요|
|`PREDICTIVE_ALERT`|위험 예상|선제 점검|

개발 순서 추천

MVP 단계에서는 이렇게 하세요
```
1. CO2 하나만 선택
2. device 1개만 선택
3. 최근 1일~7일 데이터 준비
4. 1분 간격으로 정리
5. ARIMA 모델 1개 학습
6. 1~3분 뒤 예측
7. 예측값이 임계치 초과하면 PREDICTIVE_ALERT
8. 실제값과 비교해서 정확도 확인
```
이렇게 작게 성공시키는 게 가장 좋습니다.

---
전체 개발 순서

STEP G-1. 대상 선정
```
device_A의 CO2부터 시작
```

STEP G-2. 데이터 추출
```
gas_readings에서 device_A의 CO2 시계열 추출
```

STEP G-3. 전처리
```
시간 정렬
1분 간격 보정
결측 처리
오류값 제거
```

STEP G-4. 학습/검증 분리
```
앞 80% 학습
뒤 20% 검증
```

STEP G-5. 모델 선택
```
auto_arima 또는 수동 ARIMA(p,d,q)
```

STEP G-6. 예측 테스트
```
1 step ahead
2 step ahead
3 step ahead
```

STEP G-7. 임계치 비교
```
예측값 >= 주의 기준
→ PREDICTIVE_WARNING

예측값 >= 위험 기준
→ PREDICTIVE_ALERT
```

O2처럼 낮아지면 위험인 항목은 반대로 봅니다.
```
예측 O2 < 18
→ PREDICTIVE_WARNING

예측 O2 < 16
→ PREDICTIVE_ALERT
```

STEP G-8. 결과 저장

```

```

STEP G-10. 대시보드 표시
```
“3분 뒤 CO2 주의 기준 초과 예상”
“예측값: 1040ppm”
“현재값: 850ppm”
```

### 병행 AI 모델은 어떻게 가져가야 하나?

처음부터 모든 센서, 모든 모델, 모든 예측을 한 번에 구현하지 않습니다.

이 프로젝트의 위험 판단 구조는 하나의 AI 모델로 끝내는 방식이 아니라,
규칙 기반 판단, 통계 기반 이상탐지, 머신러닝 기반 이상탐지, 시계열 예측을 단계적으로 붙이는 방식입니다.

즉 개발 순서는 아래처럼 이해하면 됩니다.
```
현재 위험 판단
   ↓
이상 징후 감지
   ↓
복합 패턴 이상 감지
   ↓
미래 위험 예측
```

세부 개발 범위와 우선순위는 문서 마지막의 `최종 개발 우선순위`를 기준으로 진행합니다.

---

ARIMA와 Isolation Forest는 역할이 다릅니다 둘 중 하나를 선택하는 문제가 아닙니다.

| 구분 | 개발 형태 | 목적 | 출력 |
|---|---|---|---|
| Threshold | 규칙 기반 판단 모듈 | 위험 기준을 넘었는가? | normal / warning / danger |
| Z-score | 통계 기반 이상탐지 모듈 | 평소보다 튀었는가? | 정상 / 이상 |
| Change Point | 상태 변화 탐지 모듈 | 상태가 바뀌기 시작했는가? | 변화 감지 |
| Isolation Forest | 머신러닝 이상탐지 모델 | 여러 센서 조합이 이상한가? | 정상 / 이상 |
| ARIMA | 시계열 예측 모델 | 미래 값이 어떻게 될까? | 예측값 |

따라서 최종 정책 엔진에서는 이렇게 합칩니다.
```
임계치 위험
→ CRITICAL

Z-score 이상
→ ANOMALY_WARNING

Change Point 발생
→ TREND_SHIFT

Isolation Forest 이상
→ ML_ANOMALY

ARIMA 예측 위험
→ PREDICTIVE_ALERT
```

---
최종 정책 엔진 예시
	최종 판단은 모델 하나가 아니라 여러 판단 결과를 합쳐서 결정합니다.
	다만 현재 위험 상태와 미래 예측 상태는 구분해서 보는 것이 좋습니다.

###### 1) 현재 상태 판단
| 우선순위 | 상태              | 의미                              |
| ---- | --------------- | ------------------------------- |
| 1순위  | CRITICAL        | 현재 센서값이 위험 임계치를 초과한 상태          |
| 2순위  | ML_ANOMALY      | 여러 센서 조합이 평소와 다른 이상 패턴          |
| 3순위  | ANOMALY_WARNING | Z-score 기준으로 평소보다 크게 튄 상태       |
| 4순위  | TREND_SHIFT     | Change Point 기준으로 상태 변화가 시작된 상태 |
| 5순위  | NORMAL          | 특별한 이상 없음                       |

###### 2) 미래 예측 상태
| 상태                 | 의미                  |
| ------------------ | ------------------- |
| PREDICTIVE_WARNING | 가까운 미래에 주의 기준 초과 가능 |
| PREDICTIVE_ALERT   | 가까운 미래에 위험 기준 초과 가능 |
주의:
`PREDICTIVE_WARNING`과 `PREDICTIVE_ALERT`는 현재 위험 확정이 아니라 선제 경고입니다.
따라서 대시보드에서는 현재 상태와 예측 상태를 분리해서 표시하는 것이 좋습니다.

예:
- 현재 상태: NORMAL
- 예측 상태: PREDICTIVE_WARNING
- 메시지: "현재는 정상이나 3분 뒤 CO2 주의 기준 초과 가능성이 있습니다."

최종적으로는 아래처럼 합쳐서 표현할 수 있습니다.
```
Threshold 결과
→ 현재 위험 여부 판단

Z-score 결과
→ 순간 이상 여부 판단

Change Point 결과
→ 상태 변화 시작 여부 판단

Isolation Forest 결과
→ 복합 패턴 이상 여부 판단

ARIMA 결과
→ 미래 위험 가능성 판단
```
초기에는 이 순서도 괜찮습니다.

다만 실무에서는 ARIMA 예측이 틀릴 수 있으므로 `PREDICTIVE_ALERT`는 즉시 위험이 아니라 선제 경고로 분리해야 합니다.

핵심 결론

ARIMA는 실질적인 AI 학습 이후 이상 데이터를 보고 추론하는 모델이라기보다는,
과거 시계열을 기반으로 가까운 미래 값을 예측하고, 그 예측값이 위험 기준을 넘을지 판단하는 예측 엔진입니다.

따라서 ARIMA만으로 이상탐지를 완성하지 말고, 아래처럼 병행합니다.
```
Threshold
= 이미 위험한가?

Z-score
= 순간적으로 튀었는가?

Change Point
= 상태가 바뀌기 시작했는가?

Isolation Forest
= 여러 센서 조합이 이상한가?

ARIMA
= 곧 위험해질 가능성이 있는가?
```

#### 현재 구현 상태 이후의 AI 개발 우선순위

현재 프로젝트는 이미 대시보드, 관리자 페이지, WebSocket 기반 테스트 데이터 스트리밍,
유해가스 및 스마트 파워 데이터 화면 구성이 완료된 상태를 전제로 합니다.

따라서 이후 개발 우선순위는 아래와 같이 진행합니다.

1차 AI 연동 준비
- WebSocket 난수 데이터와 실제 센서 정의서 필드 일치 여부 확인
- 유해가스 9개 항목의 원시 데이터 DB 저장 확인
- 스마트 파워 전류/전압/전력 데이터 DB 저장 확인
- `measured_at`, `received_at`, `device_id`, `quality_flag` 정리
- 1분 단위 시계열 데이터가 누적되는지 확인
- 임계치 기반 normal / warning / danger 판단 로직 검증
- AlarmEvent 생성 / 갱신 / 종료 처리 검증

2차 통계 기반 조기 이상탐지
- Sliding Window 구조 추가
- device_id + 센서항목별 최근 N개 유지
- Z-score 기반 ANOMALY_WARNING 판단
- O2처럼 낮아질수록 위험한 항목의 방향성 처리
- 대시보드에 현재값, 평균, z_score, 이상 여부 표시

3차 상태 변화 감지
- Change Point 추가
- 이전 구간과 최근 구간 비교
- 상태 변화 시작 시점 기록
- TREND_SHIFT / BACK_TO_STABLE 이벤트 저장
- 임계치 초과 전 위험 전조 표시

4차 머신러닝 기반 복합 이상탐지
- Isolation Forest 추가
- 여러 센서 조합 이상 탐지
- 원본값, 변화량, rolling mean, rolling std feature 구성
- ML_ANOMALY 결과 저장
- 초기에는 긴급 알람이 아니라 참고 경고로 표시

5차 단기 미래 위험 예측
- ARIMA 추가
- CO2 1개 센서부터 시작
- 이후 O2, VOC로 확장
- 1분, 2분, 3분 뒤 예측값 생성
- 예측값을 임계치와 비교
- PREDICTIVE_WARNING / PREDICTIVE_ALERT 표시

6차 스마트 파워 패턴 예측
- 채널별 전류, 전압, 전력 시계열 구성
- 시간대별 평균 사용량 계산
- 평균 대비 과사용 여부 판단
- 직전 대비 급증 여부 판단
- 카탈로그 기준 전압 범위, 최대 허용 전류, 최대 전력 범위와 비교
- 과부하 위험 또는 비정상 사용 패턴을 대시보드에 표시

정리하면 개발 순서는 아래와 같습니다.
```
데이터 저장
→ 임계치 알람
→ Sliding Window
→ Z-score
→ Change Point
→ Isolation Forest
→ ARIMA
→ 최종 정책 엔진
→ 대시보드 표시
```
