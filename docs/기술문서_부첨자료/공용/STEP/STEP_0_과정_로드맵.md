
STEP 1~ 4까지의 목적
> 실제로 운영 가능한 센서 위험 신호 탐지 엔진을 단계별로 만드는 과정입니다.

센서 데이터의 특징은 보통 다음과 같습니다.
- 데이터가 계속 들어옵니다(스트리밍).
- 정답 라벨이 없는 경우가 많습니다.
- 즉시 판단이 필요합니다.
- 계산이 너무 무거우면 운영이 어렵습니다.
- 왜 위험인지 설명 가능해야 합니다.

그래서 이 로드맵은 “실시간으로 들어오는 값이 정상인지 이상인지”를 자동으로 판단하고, 필요하면 이상이 언제부터 시작됐는지까지 추적할 수 있게 설계되어 있습니다.

### 전체 학습 순서
| 순서     | 구분           | 사고 방식              | 탐지 방식                              |
| ------ | ------------ | ------------------ | ---------------------------------- |
| STEP 1 | 통계 기반        | 수학적 규칙 / 분포 가정     | Rolling Mean/Std, <br>Z-score, 임계치 |
| STEP 2 | 실시간 처리 구조    | 데이터를 최근 구간으로 유지/갱신 | Sliding Window                     |
| STEP 3 | ML 기반 이상 탐지  | 머신러닝 모델로 패턴 학습     | Isolation Forest                   |
| STEP 4 | 상태 전이(급변) 탐지 | 시계열 구조 변화 추적       | Change Point                       |

여기서 중요한 포인트는, STEP 2는 모델이 아니라 실시간으로 계속 돌아가게 만드는 처리 구조라는 점입니다.

그리고 STEP 1과 STEP 3은 지금 이 값이 이상인가?를 판단하는 성격이 강하고, STEP 4는 언제부터 상태가 바뀌기 시작했는가?를 찾는 성격이 강합니다.


관제 플랫폼 설계 시 실제 적용 흐름
```
센서 데이터 수신
↓
STEP 2: Sliding Window로 최근 N개 데이터 유지/갱신 (실시간 처리 구조)
↓
STEP 1: 통계 기반 1차 알람 (Rolling Mean/Std → Z-score → 임계치)
↓
STEP 3: ML 기반 2차 정밀 탐지 (Isolation Forest로 패턴 이상 판단)
↓
STEP 4: Change Point로 상태 변화 시작 시점 추적 (급변/전이 탐지)
↓
알람 / 이벤트 / 로그 / 대시보드 표시
```

```
센서 데이터 수집
   ↓
STEP 2 (실시간 처리 구조)
   ↓
 ├─ STEP 1 (Z-score 탐지)
 ├─ STEP 3 (ML 기반 탐지)
 └─ STEP 4 (Change Point 감지)
   ↓
탐지 결과 통합
   ↓
알람 판단 엔진
```

이 흐름은 하나의 모델 안에 로직을 넣는다”라기보다, 각 단계가 역할을 나눠서 순차적으로(또는 병렬로) 평가한 뒤 결과를 통합하는 형태라고 이해하시면 됩니다.

STEP별로 무엇을 만들고, 왜 필요한지 이해하기

---
✅ STEP 2 — 실시간 처리 구조 만들기

센서 데이터는 계속 들어오기 때문에, 과거 전체를 매번 계산하는 방식은 운영에 부담이 됩니다.
그래서 최근 N개만 유지하면서 새 데이터가 오면 윈도우를 갱신하고, 그 상태에서 바로 계산할 수 있게 만드는 구조가 필요합니다.

이 단계는 관제 시스템의 뼈대입니다.
즉, 탐지 모델 이전에 실시간으로 돌아가는 처리 엔진을 먼저 마련하는 단계입니다.

---
✅ STEP 1 — 통계 기반 1차 이상 탐지 (빠르고 가벼운 알람)

STEP 1은 학습이 아니라 공식 기반 즉시 계산입니다.

최근 데이터 구간을 기준으로 평균과 흔들림(표준편차)을 구하고, 현재 값이 얼마나 많이 튀었는지(Z-score)로 수치화한 뒤, 임계치를 넘으면 위험으로 판단합니다.

- 빠르고, 실시간 경보에 적합합니다.
- 설명이 쉬워서 운영자가 납득하기 좋습니다.
- 다만 복잡한 패턴(서서히 망가지는 형태, 다변량 상호작용 등)은 한계가 있을 수 있습니다.

---
✅ STEP 3 — ML 기반 2차 정밀 탐지 (Isolation Forest)

STEP 3은 여기서부터 실제로 모델을 학습(fit) 합니다.

단순히 한 번 튄 값”이 아니라, 정상 구간에서 반복되던 패턴과 비교해서 패턴 자체가 이상한지를 잡아내기 위해 ML을 사용합니다.

Isolation Forest는 라벨이 없어도(정답이 없어도) 정상 패턴을 학습하고, 그 패턴에서 멀어진 데이터를 이상으로 판단합니다.
그래서 STEP 1의 단점을 보완해 정교한 이상 탐지(오탐 감소)에 도움을 줍니다.

---
✅ STEP 4 — 상태 변화(급변/전이) 탐지 (Change Point)
STEP 4는 이 값이 이상인가?를 넘어서, 상태가 언제부터 달라졌는지를 찾는 단계입니다.

예를 들어
- 평균이 갑자기 다른 수준으로 바뀌기 시작하거나
- 변동성이 커지기 시작하거나
- 패턴이 다른 국면으로 전환되는 경우

이때 Change Point는 고장(이상) 시작 시점을 추적하는 데 유용합니다.
즉, 관제에서 원인 분석이나 유지보수 판단 근거로 연결되기 좋은 단계입니다.

---
레이어 구조는 무엇을 의미하나요?

관제 시스템에서 위험 신호는 한 가지 모델로만 판단하지 않는다는 말은, 보통 서로 다른 목적의 판단을 겹쳐서 운영한다는 뜻입니다.

- STEP 1은 빠른 1차 경보를 담당하고,
- STEP 3은 정밀한 패턴 기반 탐지를 담당하고,
- STEP 4는 상태 변화 시점 추적을 담당합니다.

이 셋은 A면 A모델, B면 B모델처럼 하나만 선택하는 구조라기보다는, 같은 데이터를 서로 다른 관점으로 평가하고 결과를 합쳐서 최종 위험도를 만드는 구조로 이해하시면 됩니다.

---
마지막 정리: 각 단계가 맡는 역할
초보자 관점에서 이 단계가 뭘 해주는지만 남기면 아래처럼 정리됩니다.

|단계|실제로 얻는 것|한 줄 역할|
|---|---|---|
|STEP 2|실시간 처리 구조|최근 N개를 유지하며 계속 계산 가능하게 만드는 토대|
|STEP 1|경량 알람 로직|지금 값이 튀었는지 빠르게 판단|
|STEP 3|ML 이상 탐지 모델|복잡한 패턴 이상을 정밀하게 탐지|
|STEP 4|변화 시점 탐지|언제부터 상태가 바뀌기 시작했는지 추적|

이 로드맵의 최종 목적은, 센서 데이터가 들어왔을 때 자동으로 위험을 판단하고, 필요하면 이상이 시작된 시점을 추적할 수 있는 운영 가능한 관제 엔진을 만드는 것입니다.
