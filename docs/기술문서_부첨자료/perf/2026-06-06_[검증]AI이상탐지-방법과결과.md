# AI 이상탐지 검증 — 방법 · 결과 · 발전방향 (2026-06-06)

> **이 문서의 목적**: 가스/전력 AI 이상탐지의 "정확도가 실제로 얼마인가"를 처음으로 측정한
> 기록. **검증 방법을 직접 해보지 않은 사람도 이해할 수 있도록** 방법 설명을 충실히 담았다.
> 코드를 열어보지 않아도 "어떻게 채점했고, 결과가 뭐고, 다음에 뭘 해야 하는지"를 파악할 수 있다.

---

## 0. 30초 요약 (TL;DR)

- **검증 방법**: 시뮬레이터가 찍어둔 **정답 라벨**과 **AI의 판단**을 행마다 비교(= 시험 채점).
- **가스 AI**: 큰 사고(화재·유출·산소결핍)는 **99%** 탐지. 약점은 **co_leak 59%**, 오탐 18.8%.
- **전력 AI(5축)**: 탐지율 **87%**로 우수하나 헛알람(오탐률) **34%**로 심각.
- **핵심 발견**: 헛알람의 91%가 **"야간 격상" 규칙 하나**에서 나옴. 이 규칙은 탐지에 기여 0.
  → 끄면 **오탐률 34% → 3% (10배 감소), 탐지율은 87% → 85%로 유지**.
- **가장 큰 성과**: 이제 **측정할 수 있다**. 앞으로 튜닝은 추측이 아니라 숫자로 한다.

## 1. 배경 — 왜 검증이 필요했나

AI 추론 로그에 `score`, `prediction` 같은 값이 찍히는데, **"이 AI가 실제로 잘 맞히는지"를
한 번도 측정한 적이 없었다.** 정확도/precision/recall을 계산하는 코드도 시스템 어디에도 없었다.
"정확도가 얼마냐"는 질문에 답할 근거 자체가 없는 상태였다.

다행히 **검증에 필요한 정답지는 이미 존재**했다 — 더미 시뮬레이터가 데이터를 만들 때 각 행에
"이게 정상이냐/이상이냐"를 함께 기록해 두었기 때문이다. 이 정답지를 활용해 처음으로 채점을 했다.

## 2. 검증 방법 (★ 이 장이 핵심)

### 2.1 한 줄 원리 — "시험 채점"

> **정답지(라벨)** 와 **AI의 답안(추론 결과)** 을 행마다 비교해서 몇 개 맞췄는지 센다.

라이브 서버를 켜서 "알람이 울리나" 지켜보는 방식이 **아니다.** 과거에 쌓인 데이터를 가지고
오프라인에서 채점한다.

### 2.2 정답지 — 라벨이 어디 있나

더미 시뮬레이터가 데이터 생성 시 각 행에 두 컬럼을 찍어둔다:

| 컬럼 | 의미 | 위치 |
|---|---|---|
| `is_anomaly` | True=이상 / False=정상 | `PowerData` · `GasData` 모델 |
| `anomaly_type` | overload, co_leak 등 시나리오 이름 | 동상 |

→ **`is_anomaly`가 채점 기준(정답)이고, `anomaly_type`은 "어떤 종류의 이상을 잘/못 잡나"를
시나리오별로 쪼개 보기 위한 꼬리표**다.

### 2.3 AI 답안 — "오프라인 재현(replay)"이란

라이브 fastapi를 켜지 않고도 AI의 판단을 다시 얻을 수 있다. 비결은 **production이 쓰는 것과
똑같은 재료를 그대로 쓰는 것**:

1. **같은 모델 파일**을 로드한다 (`MLModel.file_path` → `.pkl`. production이 로드하는 바로 그 파일)
2. **같은 피처 계산 코드**로 입력을 만든다 (`apps/ml/services/feature_service.py` — 학습·추론이
   쓰는 그 함수: 최근 30틱의 value·이동평균·이동표준편차·차분)
3. 모델에 넣어 **이상/정상 판정**을 얻는다

> **왜 이게 라이브와 같은가?** 모델과 피처 수식은 **결정론적**이다. 같은 30틱 윈도우를 넣으면
> 항상 같은 결과가 나온다. 그래서 과거 데이터를 같은 모델에 다시 통과시키면, production이 그때
> 내렸을 판단과 동일하다. → 서버를 켤 필요가 없다.

### 2.4 채점 — 혼동행렬과 3개 지표

행마다 (AI 답안) × (정답)을 네 칸으로 분류한다:

```
                  정답: 이상        정답: 정상
  AI: 이상 발화     TP (맞게 잡음)    FP (헛알람) ← 오탐
  AI: 정상         FN (놓침) ← 미탐   TN (맞게 통과)
```

숫자를 세면 지표가 나온다:

| 지표 | 계산 | 뜻 | 왜 중요 |
|---|---|---|---|
| **재현율 (recall)** | TP ÷ (TP+FN) | 이상을 **놓치지 않은** 비율 | 안전: 사고를 놓치면 치명적 |
| **정밀도 (precision)** | TP ÷ (TP+FP) | 발화 중 **진짜** 비율 | 운영: 헛알람 피로도 |
| **오탐률 (FP율)** | FP ÷ (FP+TN) | 정상을 **헛알람**한 비율 | 운영: 알람 신뢰도 |

### 2.5 왜 "정확도(accuracy)"는 안 믿나

전력 데이터는 **98%가 정상**이다. "전부 정상"이라고만 찍어도 정확도가 98% 나온다 → **정확도는
함정**이다. 그래서 불균형 데이터에 맞는 recall·precision·FP율을 본다.

### 2.6 단계적 검증 설계 (이게 검증의 핵심 아이디어)

한 번에 "정확도 X%"를 내지 않고, **층을 한 겹씩 벗겨가며** 성능이 어디서 나오는지 분리했다:

1. **IF 단독** — 순수 AI 모델 한 개만의 실력
2. **5축 전체** — threshold + IF + ARIMA + Z-score + Change Point + 야간격상 = 실제 운영 시스템
3. **축 하나씩 제거(ablation)** — 각 축의 순기여를 측정 (← "야간 격상"의 정체를 밝힌 결정적 단계)

전력은 모델 하나가 아니라 여러 규칙의 합이므로, 5축 재현 시 각 축의 발화 조건을 **실제 코드에서
그대로 베껴** 구현했다:

| 축 | 출처 파일 | 재현한 판정 |
|---|---|---|
| threshold(정적임계) | `threshold_eval.py` | watt ÷ 정격 ≥ 80%(주의)/100%(위험) |
| IF | `ai/router.py` + 모델 .pkl | 모델 predict = 이상/정상 |
| Z-score | `zscore_anomaly.py` | \|값−평균\|/표준편차 ≥ 3 |
| Change Point | `change_point_service.py` | 앞30틱 vs 뒤30틱 평균·분산 비교 |
| 야간 격상 | `night_escalation.py` | KST 22~05시 + 정격 30% 초과 |
| 품질 스킵 | `quality_guard.py` | stuck(값 고정)/overflow면 발화 안 함 |
| 최종 발화 | `decide_alarm.py` | 위를 합쳐 "발화/무발화" 결정 |

### 2.7 왜 저장된 추론결과(MLAnomalyResult)를 그냥 안 썼나

이미 추론 결과가 DB에 쌓이는데도 다시 측정한 이유:
1. 현재 **이상 행만 저장**(정상은 스킵)해서 → 혼동행렬의 TN/FP를 만들 수 없음
2. 가스는 change-point 게이트를 통과한 행만 저장 → 전체 비교 불가
3. 축별 기여(어느 축이 잡았나)가 저장 안 됨

→ 그래서 "전체 라벨 데이터에 동일 모델로 다시 돌리는" 재현이 필요했다.

### 3.4 검증으로 드러난 핵심 발견 3가지

1. **야간 격상은 "탐지 기여 0, 오탐 31%"** — 축별 기여 측정에서 night는 *이상 0건 / 정상

## 4. 향후 발전 방향 (우선순위)

검증으로 "어디가 문제인지"가 숫자로 나왔으니, 추측이 아니라 **근거로** 개선한다.

### 🥇 1. 야간 격상 제거 또는 재설계 (효과 이미 검증됨)
- **근거**: 오탐률 34%→3% (10배↓), 탐지율 87%→85% (유지)
- **무엇을**: 지금의 `정격 30% 초과` 절대 규칙은 24시간 가동 설비(메인전력반·공조·조명)에
  원천적으로 안 맞음 → **끄거나**, "채널별 야간 정상 baseline 대비 편차" 방식으로 재설계
- **기대**: 운영자 헛알람 피로도 즉시 10배 감소

### 🥈 2. 검증 체계 자체를 자산화 (다른 모든 개선의 전제)
- **근거**: 지금은 측정을 매번 손으로 재현 → 튜닝 결과를 비교 불가
- **무엇을**: `manage.py eval_anomaly_accuracy` 커맨드로 정식화 (야간 on/off·축별 ablation·
  시나리오별 리포트 옵션) + **학습/평가 기간 분리(held-out)** 로 in-sample 낙관 편향 제거
- **기대**: "노브 바꿈 → 한 줄로 재채점" 루프 확보 = 진짜 튜닝 가능

### 🥉 3. 가스 co_leak 약점 + 오탐 개선
- **근거**: co_leak 0.59, 정밀도 0.39
- **무엇을**: contamination·CP penalty를 바꿔 실험 후 2번 커맨드로 재측정. 느린 누출용 피처
  (기울기·누적합) 보강 검토
- **기대**: 약점 시나리오 재현율↑, 오탐↓

### 4. 전력 사각지대 메우기
- **근거**: motor_stuck 0.06~0.19, phase_loss 0.24~0.26
- **무엇을**: current/voltage 채널에도 추론 확대(전압 이상 가시화), motor_stuck은 "탐지"가
  아니라 **"센서 고장 알람"으로 별도 처리**(이미 스킵 중이니 알람만 연결)

### 5. ARIMA 가치 재판정
- **근거**: 5축에서 기여 거의 0
- **무엇을**: 차수(p,d,q) 재학습으로 살릴지, 비용 대비 걷어낼지 2번 체계로 A/B

### 발전의 진짜 의미
> 이번 검증의 가장 큰 성과는 특정 숫자가 아니라 **"이제 측정할 수 있다"** 는 점이다. 앞으로
> 모든 개선은 **측정 → 발견 → 수정 → 재측정** 루프로 돌릴 수 있다. 2번(검증 자산화)이 1순위와
> 묶여야 하는 이유다.

## 5. 한계와 주의 (숫자 신뢰도)

- ✅ **믿을 수 있는 부분**: 모델·피처는 production과 동일 파일/코드라 재현 결과가 동일.
- ⚠️ **재현의 틈**: 5축 로직은 코드를 보고 *재현*한 것이라 production과 미묘하게 다를 수 있음
  (예: 상태머신 초기화 타이밍). 각 수식은 실제 코드에서 가져옴.
- ⚠️ **데이터 스냅샷 차이**: 측정 사이 라이브 데이터가 늘어 행 수가 달라짐(86k→106k). 단,
  지표가 모두 비율이라 비교는 유효.
- ⚠️ **in-sample 낙관**: IF 학습기간과 평가 데이터가 겹쳐 오탐이 다소 낙관적 → held-out 필요(4장 2번).
- ⚠️ **FP율 과장 가능성**: 합성 데이터가 야간 부하를 안 낮춰 night FP가 부풀려졌을 수 있음.
  실제 현장 수치는 운영 패턴에 따라 달라짐.

## 부록 A. 재현 방법 (다시 측정하려면)

라이브 서버 없이, drf 컨테이너 안에서 다음 재료로 배치 추론 → 채점:

| 단계 | 사용한 것 |
|---|---|
| 정답지 | `PowerData`/`GasData`의 `is_anomaly`, `anomaly_type` |
| 모델 | `MLModel`(is_active=True)의 `file_path` → `joblib.load` |
| 피처 | `apps.ml.services.feature_service.build_features` / `build_multi_features` |
| 5축 재현 | `power/services/`의 threshold_eval·zscore_anomaly·change_point_service·night_escalation·quality_guard·decide_alarm 로직 복제 |
| 채점 | 행별 (fire) vs (is_anomaly) → TP/FP/FN/TN → recall·precision·FP율 |

> 이 절차를 `manage.py eval_anomaly_accuracy` 커맨드로 박제하는 것이 4장 2번 과제다.
> 그 전까지는 본 문서의 절차를 따라 일회성 스크립트로 재현한다.

## 부록 B. 활성 모델 (측정 시점 2026-06-06)

- 가스 IF v5: `gas_if_v5.pkl` (9종, 45피처 = 9가스 × {value·이동평균·이동표준편차·차분·ARIMA잔차})
