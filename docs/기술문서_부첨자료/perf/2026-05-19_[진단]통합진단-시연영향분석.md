## §2. 항목별 상세 (통합본 ①~⑲ 순서)

### ① 가스 데이터 들어올 때마다 에러 발생  〔📝 가스 인계〕

- **무엇이 문제**: 가스 AI 모델이 미등록 상태일 때 코드가 존재하지 않는 함수 `_save_gas_data`를 호출 → `NameError`. 외부 `except`가 잡아서 DRF 저장 흐름은 살아남지만 **매 가스 데이터마다 NameError 로그**가 찍힘.
- **처리**: 가스 영역이라 다른 팀원에게 인계 ([handoff 문서](../../../fastapi-server/docs/handoff/gas-p0-fixes-2026-05-19.md)).
- **시연 중 영향**: 가스 AI 모델 등록 전까지 로그 노이즈만 발생. **DRF 저장과 화면은 정상**. 가스 모델 등록 시 자동 해소.

---

### ② 매 초 랜덤 "위험/정상" 전송  〔✅ 적용〕

- **무엇이 문제**: 실제 센서 상태와 무관하게 1초마다 10% 확률로 "위험" 신호가 페이로드에 실려 나갔어요. 개발 초기 더미 코드 잔재.
- **수정**: 로직과 `level` 키 자체를 제거. (프론트는 실제로 이 값을 사용하지 않고 있었음 확인.)
- **시연 효과**: 화면 변화 없음. WS 트래픽이 의미 없는 신호로부터 깨끗해짐.

---

### ③ 가스 데이터 stale 시 페이로드에서 필드 자체가 사라짐  〔✅ 적용〕

- **무엇이 문제**: 가스가 5초 이상 안 오면 페이로드에서 `co`/`h2s`/`co2` 같은 가스 키가 **아예 빠져나감** → 프론트가 그 값을 꺼내려다 에러.
- **수정**: 가스가 끊겨도 키는 항상 유지하고 값만 `null`로 송신. 전력의 `equipment: []` 정책과 통일.
- **시연 효과**: 시연 중 가스 시뮬레이터 일시 정지/재시작 시나리오에서 **대시보드가 깨지지 않음**.

---

### ④ 전력 데이터가 있는데도 "로딩 중"으로 표시됨  〔✅ 적용〕

- **무엇이 문제**: `power_loading` 조건이 "장비 개수가 0이면 로딩 중"이었음. "전원 꺼짐"과 "데이터 미수신"을 구분 못 함 (의미 모호).
- **수정**: `power_loading`을 **데이터 도착 여부(stale)** 직접 반영으로 단순화.
- **시연 효과**: 실제 동작 결과는 동일하지만 의미가 명확. 시뮬레이터 정지 시 정확히 "로딩 중" 표시.

---

### ⑤ WARNING 알람이 3초 만에 뜨도록 설정됨  〔⏭️ 시연 후 / A. 의도 유지〕

- **무엇이 문제**: WARNING 알람이 "위험 수치 X초 지속 시" 발화하는데 현재 X=3초. 산업 기준은 30초~1분.
- **왜 시연 후**: **시연용 더미 시나리오 때문에 의도적으로 3초**. 더미가 위험 수치를 빠르게 DANGER까지 올려서 30초로 두면 WARNING 단계가 거의 발화 안 됨 → 시연에서 WARNING 시연 불가. 코드 주석에 "Phase 2 P2 운영 피드백"으로 명시.
- **후속 처리**: 시연 직후. 환경변수화(운영 30초 / 개발 3초 분리).

---

### ⑥ E2E 레이턴시 측정 시작 시각이 잘못됨  〔✅ 전력 적용 / 📝 가스 인계〕

- **무엇이 문제**: "IoT 도착 → 브라우저까지의 시간" 측정 시작점이 핸들러 첫 줄이 아니라 백그라운드 작업 등록 직전에 잡혀 있었음. **Grafana E2E 수치가 실제보다 짧게** 표시.
- **수정 (전력)**: `recv_current` / `recv_voltage` / `recv_watt` 3개 엔드포인트 모두 핸들러 첫 줄에서 `ingress_ts` 캡처.
- **인계 (가스)**: `process_gas_data`의 동일 패턴 적용 — 가스 작업자 인계.
- **시연 효과**: Grafana E2E P95가 **AI 추론 시간(30~80ms)만큼 증가**. 수치 악화가 아니라 **정확해진 것**. 250~300ms대로 올라와도 정상.

---

### ⑦ AI 가스 알람에 ingress_ts가 빠짐  〔✅ 전력 적용 / 📝 가스 인계〕

- **무엇이 문제**: 룰 기반 알람은 E2E를 측정하는데 **AI 알람은 측정 정보 자체가 빠져** Grafana에서 AI 알람 속도 파악 불가.
- **수정 (전력)**: `process_anomaly_inference`에 `ingress_ts` 옵션 인자 추가, `push_payload`까지 전달. (진단 통합본에 명시 안 된 전력 버전을 추가 발견하여 같이 처리.)
- **인계 (가스)**: 가스 AI 알람의 `push_payload`에 동일 키 추가 — 가스 작업자 인계.
- **시연 효과**: Grafana 알람 E2E 패널에 **`alarm_type="power_anomaly_ai"` 시리즈 새로 등장**. 처음 보이는 시리즈이므로 어떤 값이 나와도 정상.

---

### ⑧ 브로드캐스트 timestamp에 timezone 정보 없음  〔✅ 적용〕

- **무엇이 문제**: 매 초 보내는 `timestamp`가 서버 로컬 시간(시간대 정보 없음). 다른 환경에 배포 시 9시간 어긋날 가능성.
- **수정**: 다른 timestamp 필드와 동일하게 UTC로 통일.
- **시연 효과**: 변화 없음(시연은 한국 서버). 배포 환경 안전성 확보.

---

### ⑨ 브로드캐스트의 device_id가 "sensor-01"로 하드코딩  〔✅ 적용〕

- **무엇이 문제**: 통합 페이로드(가스+전력+위치 합본)에 `device_id: "sensor-01"`이 하드코딩. 단일 통합 페이로드에서 "어떤 장비"인지 의미 모호하고 프론트도 미사용.
- **수정**: 키 자체 제거. 가스/전력은 각자 도메인 페이로드에 별도 device_id 보유 → 정보 손실 없음.
- **시연 효과**: 변화 없음.

---

### ⑩ AI 예측 값이 매 초 랜덤으로 바뀜  〔✅ 적용〕

- **무엇이 문제**: 대시보드 AI 예측 카드(`예상 도달 시간`, `최대 부하`)가 매 초 임의 숫자로 흔들림. 더미 코드.
- **수정**: 세 수치 필드(`ai_eta_min`, `ai_max_load_kw`, `ai_max_load_pct`)를 `null` 송신으로 변경. 프론트는 `null` → "-" 표시.
- **시연 효과**: **시연 화면의 AI 예측 카드 수치 영역이 "-"로 표시**. 의도된 동작 — 전력 AI Phase 2 (un-downgrade) 완성 시 실제 예측값으로 교체.
- **모니터링 포인트**: 시연 중 "AI 카드가 비어있다"는 질문이 들어올 수 있으니 사전 안내 필요.

---

### ⑪ AI 추론 중에 모든 센서 처리가 멈춤 (이벤트 루프 블로킹)  〔⏭️ 시연 후 / B. 발생조건 미충족〕

- **무엇이 문제**: AI 모델 추론이 동기 함수로 호출돼 그동안 다른 센서 요청 처리가 잠시 멈춤.
- **왜 시연 후**: 현재 IF 모델 추론이 매우 빨라(수 ms) 사실상 체감 안 됨. 데이터량/모델 크기 증가 시에만 드러남.
- **시연 중 관찰**: **`process_request_duration_seconds` P95 > 100ms이면 보고** — 블로킹이 표면화됐다는 신호.
- **후속 처리**: 시연 후. `run_in_executor`로 비동기 분리.

---

### ⑫ WARNING 취소 키가 "_pending_"으로 잔류  〔⏭️ 시연 후 / B〕

- **무엇이 문제**: WARNING 타이머 취소 열쇠(task ID)를 저장하기 전 예외가 나면 `"_pending_"` 문자열만 캐시에 남음 → 이후 취소 시도가 무력화 → 중복 알람.
- **왜 시연 후**: Celery `apply_async` 자체가 실패해야 발생하는 예외 경로. 시연 중 Redis/Celery 안정 → 발생 확률 매우 낮음.
- **시연 중 관찰**: **같은 센서/가스에 대해 WARNING이 2회 연속 발화하면 보고** (정상은 1회).
- **후속 처리**: ⑭와 함께 시연 후. 예외 핸들러에서 `"_pending_"` 키 명시적 정리.

---

### ⑬ _TASK_KEY_TTL이 retry 누적시간보다 짧음  〔⏭️ 시연 후 / A. ⑤ 묶음〕

- **무엇이 문제**: 캐시 키 TTL = WARNING(3s) + 5s = 8초. 그런데 retry 최대 누적은 15초. 8초 뒤 키가 사라지면 새 WARNING이 또 시작되어 중복 발화 가능.
- **왜 시연 후**: ⑤를 30초로 올리면 TTL=35초가 되어 자동 해결. 단독으로 처리하기보다 ⑤와 묶음.
- **후속 처리**: ⑤와 동시에 시연 후.

---

### ⑭ DANGER가 와도 실행 중인 WARNING은 revoke 불가  〔⏭️ 시연 후 / B〕

- **무엇이 문제**: WARNING 타이머 카운트다운 끝나 실행 시작된 후 DANGER가 오면, WARNING 취소 시도 실패 → WARNING + DANGER 동시 발화.
- **왜 시연 후**: ⑤가 3초로 짧아 발생 빈도 매우 낮음. ⑤를 30초로 올리는 시점에 빈도 증가 → 그때 가드 추가가 적절.
- **시연 중 관찰**: **WARNING 토스트 직후(1~2초 내) 같은 센서/채널의 DANGER 토스트 연속 발화 시 보고**.
- **후속 처리**: ⑤+⑬과 함께 시연 후. WARNING 태스크 진입 시 state 확인 후 abort 가드.

---

### ⑮ Celery worker concurrency = 2 → 알람 폭주 시 큐 적체  〔⏭️ 시연 후 / B〕

- **무엇이 문제**: 알람 워커 2개가 각자 FastAPI 응답을 최대 10초까지 대기. 가스 9종 + 전력 16채널 동시 폭주 시 25개 줄 서기 → 마지막 알람 지연.
- **왜 시연 후**: **시연 시나리오는 가스 1~2종 / 전력 1~2채널 발화** → 동시 25발화 상황 미발생.
- **시연 중 관찰**: **`e2e_latency_seconds` P95가 5초 이상으로 튀면 보고** (정상은 1~2초). 큐 적체의 신호.
- **후속 처리**: 시연 후. 단기 `--concurrency=4` 상향, 중기 가스/전력 큐 분리.

---

### ⑯ LPUSH → LTRIM 비원자적  〔⏭️ 시연 후 / B〕

- **무엇이 문제**: 알람 큐에 넣은 직후 크기 자르는 두 동작 사이에 다른 알람이 끼어들 수 있어 큐 제한(10,000개)을 살짝 초과 가능.
- **왜 시연 후**: 평소 큐 크기 < 10. 1만 개 근접 = 폭주 상태인데 시연 시나리오는 그 수준이 아님.
- **시연 중 관찰**: **Redis `alarm_queue` LIST 길이 > 100 시 보고** — 폭주 상황 진입.
- **후속 처리**: 시연 후. Lua 스크립트로 LPUSH+LTRIM 원자화.

---

### ⑰ sensor_clients가 list라서 연결 해제가 O(n)  〔⏭️ 시연 후 / B〕

- **무엇이 문제**: 브라우저 연결 해제 시 목록에서 찾는 동작이 O(n). 100명 넘으면 체감.
- **왜 시연 후**: 시연 동시 접속자 ≪ 100명.
- **시연 중 관찰**: 없음 (자명).
- **후속 처리**: 시연 후. `list` → `set` 단순 교체.

---

### ⑱ SQLite의 동시성 한계 → PostgreSQL 전환 필요  〔⏭️ 시연 후 / C. 인프라 작업〕

- **무엇이 문제**: SQLite는 동시 저장이 한 번에 하나씩만 처리. 최대 6개 프로세스가 동시 저장 시도 → 알람 폭주 시 적체. 전에 파일 크기 12GB까지 폭증한 적도 있음.
- **현재 완충 장치**: WAL + busy_timeout으로 동시성 한계 완화 중.
- **왜 시연 후**: 인프라 전체 영향이 큰 작업(연결 설정, 마이그레이션, 백업 정책 전부 변경). 시연 D-26 안에 안전 진행 불가.
- **시연 중 관찰**: **`sqlite_db_size_bytes` 5GB 근접 시 즉시 보고** — 12GB 폭증 사례 재현 우려. WAL 체크포인트 동작도 확인.
- **후속 처리**: 시연 후 별도 plan. 5GB 트리거 도달 시 즉시 착수.

---

### ⑲ 가스 슬라이딩 윈도우가 인메모리 전역 변수  〔⏭️ 시연 후 / C〕

- **무엇이 문제**: 가스 AI 추론용 최근 30개 데이터가 서버 메모리에만 있음 → FastAPI 재시작 시 처음부터 다시 모음 (30초 공백). multi-worker 전환 시 워커별 윈도우 분리 → 추론 결과가 워커마다 달라짐.
- **왜 시연 후**: 시연 중 FastAPI 재시작 가능성 낮고, 현재 단일 워커라 분리 문제도 미발생. ⑱(PG 전환) + multi-worker 전환과 한 번에 진행하는 게 효율적.
- **시연 중 관찰**: **FastAPI를 시연 중 재시작하지 말 것**. 부득이 재시작했다면 그 후 30초간 가스 AI 알람 미발화는 **정상 동작**.
- **후속 처리**: ⑱과 함께. Redis LIST로 윈도우 이전.
