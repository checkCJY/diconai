# FastAPI Multi-Replica(수평 확장) 검토 — 팀 공유용

> **작성일:** 2026-06-01
> **목적:** 4차 프로젝트 문서의 ③Redis(Pub/Sub) + ⑦Kubernetes(자동확장) 항목을 실제로 구현하려면
> 무엇을 바꿔야 하는지, 왜 단순하지 않은지를 **개념부터** 정리한 팀 공유 문서.
> **전제:** 팀 전원 이해도 낮음 → 용어·배경부터 설명. 끝에 용어집 있음.
> **중요:** 여기 나오는 작업은 **6/14 시연 동작에 영향이 0**입니다. "확장 가능한 시스템"을 보여주기
> 위한 아키텍처 작업이지, 시연 버그 수정이 아닙니다. 그래서 "정공법이냐"보다 **"어디까지가
> 시연/기술문서에 값을 만드느냐"가 판단 기준**입니다.

## 0. 한 줄 요약

> 지금 FastAPI는 **단일 프로세스라서** 잘 돌아간다. 프로세스를 여러 개로 늘리는(=multi-replica)
> 순간, **프로세스 메모리에 들고 있던 상태들**이 전부 깨진다. 그 상태를 성격별로 3계층으로 나눠
> 각각 다른 방법으로 풀어야 하며, 그중 AI 부분(계층 2)이 가장 어렵다.

## 1. 배경 개념 — "왜 지금은 되고, 늘리면 깨지나"

### 1-1. Replica(레플리카)가 뭔가
**똑같은 FastAPI 프로세스를 여러 개 복제해서 동시에 띄우는 것.**

```
        [로드밸런서 / k8s Service]
        ↙        ↓        ↘
   FastAPI-A  FastAPI-B  FastAPI-C   ← 같은 코드, 다른 프로세스 (각자 독립 메모리)
```

트래픽이 커지면 프로세스를 늘려 부하를 나눠 받습니다. k8s의 **HPA**(자동 확장)가 "CPU 높으면
2개→4개로 자동 증식"을 해주는 거고, 4차 문서 ⑦의 "자동 확장"이 이겁니다.

**가장 중요한 함정:** replica는 각자 **독립된 메모리**를 가집니다. A 프로세스의 변수와 B 프로세스의
변수는 완전히 별개예요. 같은 코드라도 메모리는 공유되지 않습니다.

### 1-2. 지금(단일 프로세스)은 왜 되나
현재 FastAPI는 딱 1개 프로세스(`workers=1`, `replicas=1`)라 모든 게 한 메모리 안에 있습니다.

```
IoT 가스 POST ──→ [단일 프로세스]
                   ├ latest_gas_snapshot = {...}   ← 여기 저장(write)
                   └ broadcast 루프가 같은 변수 읽음(read) ──→ 브라우저
```
저장하는 곳과 읽는 곳이 **같은 메모리**라 자연히 일치합니다. 그래서 문제가 없습니다.

### 1-3. Multi-replica가 되면 왜 깨지나
프로세스가 3개가 되면 메모리가 3개로 쪼개집니다.

```
가스 POST ─→ [A] latest_gas_snapshot = {co:50}   (A만 앎)
                  [B] {}  (비어있음)
                  [C] {}  (비어있음)

브라우저 ──연결──→ [B] → B는 자기 빈 메모리를 읽음 → 화면에 가스 안 뜸 ❌
```

- 센서 POST는 로드밸런서가 **아무 replica에나** 던짐 → 한 replica 메모리만 갱신됨
- 브라우저 WebSocket도 **아무 replica에나** 붙음 → 다른 replica에 붙으면 데이터 못 봄

**해법의 원리는 하나:** 프로세스마다 따로 들고 있는 상태를, **모든 프로세스가 공유하는 한 곳
(= Redis)** 으로 옮긴다. Redis는 프로세스 밖 별도 서버라 A·B·C가 같은 값을 봅니다.

## 2. 핵심 발견 — 상태는 5개가 아니라 "성격이 다른 3계층"

코드를 정독해 보니, 프로세스 메모리에 든 상태가 단순히 5개가 아니라 **성격이 전혀 다른 3계층**으로
나뉩니다. 계층마다 해법과 난이도가 다릅니다.

### 계층 1 — 브로드캐스트 스냅샷 (쉬움) 🟢
화면에 그릴 **"현재 값"**. "마지막 값이 이김(last-value)" 성격.

| 상태 | 무엇 | 위치 |
|---|---|---|
| `latest_gas_snapshot` | 가스 9종 현재값+위험도 | `gas/services/gas_service.py` |
| `power_latest` | 전력 채널별 onoff/current/voltage/watt | `power/services/power_service.py` |
| `worker_positions` | 작업자 좌표 | `positioning/routers/position_router.py` |
| `scenario_mode` | 시연 모드값 | `internal/routers/scenario_router.py` |

**해법: Redis 키(SET/HSET/GET)로 이관.** 센서가 `SET`으로 쓰고, 각 replica의 broadcast 루프가
틱마다 `GET`으로 읽음. 잃는 정보 없음(아래 2-1 참고). 난이도 낮음.

### 계층 2 — AI 시계열 윈도우 (어려움) 🔴 ★이번 검토의 핵심
"마지막 값"이 아니라 **최근 N개 샘플을 순서대로 누적**해 이상탐지를 돌리는 버퍼.

| 윈도우 | 용도 | 위치 |
|---|---|---|
| `_co_window`/`_h2s_window`/`_co2_window` | 가스 변화점(CP) 탐지 | `gas/services/gas_service.py` |
| `_cp_windows` (+`_cp_states`) | 전력 변화점(CP) | `power/services/change_point_service.py` |
| `_power_windows` (Z-score) | 전력 Z-score | `power/services/zscore_anomaly.py` |
| `_power_windows` (IF) | 전력 Isolation Forest | `power/services/anomaly_inference.py` |

**왜 어려운가 → 3장에서 따로 자세히 설명.**

### 계층 3 — Read-through 캐시 (무변경) ⚪
| 캐시 | 위치 |
|---|---|
| `_channel_meta_by_device` | `power/services/channel_meta_cache.py` |
| `_threshold_meta` | `power/services/threshold_sync.py` |

DRF에서 5분마다 polling해 갱신 → 모든 replica가 독립적으로 같은 값을 가져옴.
**multi-replica여도 그대로 안전.** 작업 불필요(견적에서 제외).

### 2-1. "계층 1 덮어쓰기는 임시방편 아닌가?" — 아닙니다
판별 기준은 **"브로드캐스트 틱 사이의 값을 누가 필요로 하는가?"**
- 화면 표시용 → **아니오.** 대시보드는 "지금 값"만 그림. 중간값 놓쳐도 다음 틱에 덮이면 그만.
- AI 탐지용 → 예. 하지만 그건 **계층 2의 몫**(별도 시계열).
- 감사·이력용 → 예. 하지만 그건 **DRF/Postgres 영속화**가 이미 함.

즉 브로드캐스트 스냅샷은 데이터가 본질적으로 last-value라, 시퀀스는 딴 데 살아있으니
덮어쓰기로 **잃는 정보가 없습니다.** → 그 데이터에 맞는 **올바른 패턴**(임시방편 아님).
(단 `power_latest`만 필드별 부분 갱신이라 `HSET`으로 처리 — 이것도 정석.)

## 3. 계층 2(AI)가 왜 어려운가 — 자세히

### 3-1. "윈도우"가 뭔가
이상탐지(CP·IF·Z-score)는 "지금 값 하나"로 판단하지 않고, **최근 N개 샘플을 순서대로 쌓아둔
버퍼**로 판단합니다. 그 버퍼가 윈도우.

```python
_co_window = deque(maxlen=30)   # 최근 30개 CO값만 유지 (오래된 건 자동 탈락)
_co_window.append(payload.co)   # 샘플 올 때마다 누적
if len(_co_window) >= 30:
    _detect_change_point(list(_co_window))   # 윈도우 전체로 변화점 탐지
```

탐지기가 보는 것:
- **Change Point(변화점):** "패턴이 *바뀌었나*" — CO가 5ppm이다 15ppm으로 점프 = 30~60개 연속값
  추세를 봐야 앎.
- **Isolation Forest:** 윈도우로 특징(평균·분산·기울기)을 만들어 "정상에서 벗어났나" 점수화.
- **Z-score:** 윈도우의 평균·표준편차 대비 현재값이 몇 σ 벗어났나.

→ 셋 다 **"연속된 시계열"** 이 전제.

### 3-2. 왜 multi-replica가 이걸 깨나 (숫자 예시)
센서가 1초마다 샘플을 보내고 replica가 2개면, 로드밸런서가 번갈아 던집니다:

```
샘플:  t1=5  t2=6  t3=5  t4=7  t5=14  t6=15  t7=16   ← t5부터 누출 시작
        ▼     ▼     ▼     ▼     ▼      ▼      ▼
      [A]   [B]   [A]   [B]   [A]    [B]    [A]
A 윈도우:  5, 5, 14, 16 …   (6,7,15 없음 = 구멍)
B 윈도우:  6, 7, 15 …       (5,5,14,16 없음 = 구멍)
```

- **어느 replica도 완전한 시계열을 못 가짐.** 둘 다 절반씩 구멍.
- CP는 "5→14 점프"를 봐야 하는데 조각나서 **탐지 실패 또는 오탐**.
- 게다가 `_cp_states`(상태머신)·Z-score 통계처럼 **샘플 사이에 정보를 들고 다니는 내부 상태**까지
  있어서, 윈도우를 Redis로 옮겨도 동시 접근 락이 필요 → 더 복잡.

### 3-3. 그래서 계층 2는 "그냥 Redis로 못 옮긴다" — 선택지 3개

| 방안 | 내용 | 평가 |
|---|---|---|
| **(a) Sticky 라우팅** | "한 센서 = 항상 같은 pod"로 고정. 윈도우는 메모리 유지 | 가장 쌈. 단 아래 ⚠ 한계 |
| **(b) AI를 단일 컨슈머로 분리** | 수신은 N개, **AI 추론은 별도 단일 프로세스**가 스트림을 순서대로 소비 | **정공법.** 시계열 무결성 보장 |
| (c) 윈도우 Redis 이관 + 채널 락 | deque를 Redis로, 호출을 직렬화 | 복잡·지연 큼. 비권장 |

#### ⚠ (a) Sticky 라우팅의 함정 — "이거 하나로 끝"이 아님
Sticky는 **계층 2(AI 윈도우)만 부분 해결**하고, **계층 1(브로드캐스트)·알람은 전혀 못 풉니다.**
이유: **센서와 브라우저는 다른 클라이언트**라서.

```
가스 센서  ──sticky──→ [pod A]  (윈도우는 A에 온전히 쌓임 ✅ 계층 2 해결)
브라우저   ──별도 연결─→ [pod B]  (B엔 가스 데이터 없음 ❌ 계층 1 여전히 깨짐)
```
→ sticky를 깔아도 **계층 1(Redis 키)·알람(Streams)은 그대로 해야 함.**
sticky 하나로 multi-replica가 굴러가지 않습니다.

추가로, k8s에서 실제 적용 시:
- `sessionAffinity: ClientIP` 는 **소스 IP** 기준 해시 → 더미가 한 pod에서 돌면 모든 센서가 한 pod로
  몰려 분산 0.
- nginx `upstream-hash-by: $arg_sensor_id` 는 (1) ingress 컨트롤러 미배포 상태라 먼저 깔아야 하고,
  (2) 현재 sensor_id가 body에 있어 쿼리로 못 보냄 → **클라이언트 코드 수정 필요**("코드 안 댐"과 모순).

## 4. 알람은 어떻게 되나 — 별도 트랙

### 4-1. 현재 구조
알람은 이미 Redis **LIST + BRPOP**을 씁니다(`websocket/services/alarm_queue.py`). 재시작 시 휘발
방지 + race 해소를 위해 일부러 메모리→LIST로 옮긴 설계.

### 4-2. Multi-replica에서 깨지는 이유
BRPOP은 **"경쟁 소비자"** — 알람 1건을 **한 replica만** 꺼내갑니다. replica가 2개 이상이면, 알람을
꺼낸 한 pod에 붙은 브라우저만 받고 **나머지 pod의 브라우저는 누락**.

### 4-3. 해법과 함정
- raw **pub/sub**(발행/구독)은 **영속성이 없음** → 구독 중인 replica가 0이면 알람 영구 소실.
  현재 LIST가 일부러 확보한 durability가 깨짐. **그래서 부적합.**
- 정답은 **Redis Streams + consumer group** (fan-out + durability 둘 다) 또는 replica별 전용 LIST.

### 4-4. 중요한 순서 규칙
브로드캐스트(계층 1)와 알람은 **코드상 독립**이라 따로 작업 가능. 단:
> **둘 다 끝나야 비로소 replicas≥2를 켤 수 있다.**
> 브로드캐스트만 끝난 상태에서는 **replicas=1을 유지**해야 한다. multi-replica의 "이득"은 알람까지
> 끝나야 실현된다.

그래서 **브로드캐스트 먼저 → 알람 Streams 나중**은 유효한 점진 경로. 단 그동안 replicas=1 유지.

## 5. ⚠ 용어 교정 — "pub/sub" vs "Redis 키"

흔히 헷갈리는 부분. **다른 도구입니다.**

| | pub/sub (발행/구독) | Redis 키 (SET/HSET/GET) |
|---|---|---|
| 성격 | **이벤트** — "방금 이런 일이 생겼다" | **상태** — "지금 값은 이거다" |
| 영속성 | 없음 (놓치면 끝) | 있음 (언제 읽어도 최신값) |
| 맞는 대상 | **알람**(1회성 사건) | **가스/전력/위치 스냅샷**(마지막 값) |

→ **"브로드캐스트를 pub/sub로"가 아니라 "브로드캐스트 상태를 Redis 키로 이관"이 정확한 표현.**
pub/sub(나중엔 Streams)은 **알람 전용**.

## 6. 통합 목표 아키텍처 (계층 2를 b로 갈 경우)

b(AI 분리)를 고르면 AI가 순서대로 읽을 **센서 스트림(Redis Stream)** 이 생기고, 그러면 계층 1의
latest 키는 "스트림 위의 파생 뷰"가 되어 전체가 **하나의 그림**으로 합쳐집니다.

```
                ┌──────── Redis Stream (센서 backbone, 순서·영속) ────────┐
센서 POST → 수신(N replica) ─XADD─→                                        │
                                  ├─→ [AI 컨슈머 1개]  순서대로 CP/IF/Z-score (계층 2 = b)
                                  └─→ latest 키 → [WS 게이트웨이 N] GET → broadcast (계층 1)
알람 → 별도 Stream + consumer group (나중에)                  (계층 3 캐시는 그대로)
```

b 안에서도 강약 노브:
- **b-mini:** 수신이 둘 다 함 — `XADD`(AI용) + `SET latest`(브로드캐스트용). 가볍다. **권장 시작점.**
- **b-pure:** 수신은 `XADD`만, 컨슈머가 latest 키를 만들고 게이트웨이가 읽음. CQRS 정석이지만
  부품 증가. 추가 가치가 *코드 동작*이 아니라 *서사*뿐 → 시연 후로 미뤄도 됨.

## 8. 현재 k8s 실태 (참고)

`k8s/fastapi.yaml` 기준:
- `replicas: 1` (주석에 "복수 replica 불가" 명시)
- `sessionAffinity` **없음** (기본 라운드로빈)
- Ingress **없음** (`type: NodePort`), nginx ingress 컨트롤러 미배포
- HPA **없음**

→ 즉 sticky·자동확장 관련 설정은 **아무것도 없는 상태.** 무엇을 하든 새로 작성해야 함.

   핵심이지, 시연 버그 수정이 아님.
2. **계층 1(브로드캐스트)** 은 Redis 키로 깔끔히 가능 — 정석, 난이도 낮음.
3. **계층 2(AI)** 가 진짜 걸림돌. 깨끗이 가려면 **b(분리)** 가 정공법. sticky(a)는 더 싸지만 AI 조각만
   부분 해결 + 자동확장 사실상 포기.
4. **알람**은 별도 트랙(Streams), 그리고 **replicas≥2는 브로드캐스트+알람 둘 다 끝난 뒤에만** 켠다.
5. 따라서 가장 작고 안전한 첫 스텝은 **계층 1만 Redis 키로 이관(replicas=1 유지)**. 그다음 알람
   Streams → AI 분리 → 최종 replicas≥2 순서.

## 부록. 용어집 (이해도 낮은 팀원용)

| 용어 | 쉬운 설명 |
|---|---|
| **Replica / 레플리카** | 같은 서버 프로그램을 여러 개 복제해 동시에 띄운 것. 각자 메모리 따로. |
| **Multi-replica / 수평 확장** | replica를 2개 이상으로 늘려 부하를 나눠 받는 것. |
| **프로세스 메모리** | 한 프로그램이 자기 안에 들고 있는 변수들. 다른 프로세스와 공유 안 됨. |
| **로드밸런서** | 들어오는 요청을 여러 replica에 나눠 보내는 분배기. |
| **Sticky 라우팅 / sessionAffinity** | "같은 출처는 항상 같은 replica로" 고정 보내는 설정. |
| **last-value (마지막 값이 이김)** | 같은 항목을 여러 번 쓰면 최신 값만 남는 것. 화면 표시용 데이터에 적합. |
| **Redis 키 (SET/GET/HSET)** | Redis에 "이름=값"으로 저장/조회. **상태**(현재 값) 저장에 씀. |
| **pub/sub (발행/구독)** | "방송"처럼 발행하면 구독자 전원이 받음. **이벤트** 전달용. 영속성 없음. |
| **Redis Stream** | 순서가 보장되는 append-only 로그(이벤트 기록장). 영속+순서+여러 소비자 가능. |
| **consumer group** | 한 Stream을 여러 소비자가 나눠/중복 없이 읽게 해주는 Redis 기능. |
| **BRPOP** | Redis LIST에서 "한 명만 꺼내가는" 블로킹 꺼내기. = 경쟁 소비자. |
| **경쟁 소비자** | 메시지 1건을 여러 소비자 중 **딱 한 명만** 가져가는 모델. |
| **윈도우(시계열)** | 최근 N개 샘플을 순서대로 모아둔 버퍼. 이상탐지에 필요. |
| **CP (Change Point)** | "데이터 패턴이 바뀐 시점"을 찾는 이상탐지. |
| **IF (Isolation Forest)** | 정상 군집에서 떨어진 이상치를 점수화하는 ML 모델. |
| **Z-score** | 평균에서 표준편차의 몇 배 벗어났는지로 이상 판단. |
| **HPA** | k8s 자동 확장기. 부하 높으면 replica를 자동으로 늘림. |
| **Ingress** | k8s에서 외부 트래픽을 클러스터 안으로 들이는 입구(+라우팅 규칙). |
| **CQRS / 파생 뷰** | "쓰기 모델"과 "읽기 모델"을 분리. latest 키를 스트림에서 만들어내는 식. |
| **durability(영속성)** | 재시작·장애에도 데이터가 살아남는 성질. |
