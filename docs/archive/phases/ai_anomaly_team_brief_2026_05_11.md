# AI 이상탐지 도입 검토 — 팀 공유용 문서

> 목적: 가스 센서 알람 시스템에 AI 이상탐지를 추가하는 방안을 팀이 같이 검토하기 위한 자료. **실행 지시서가 아니라, 같은 그림을 보기 위한 자료**입니다.

---

## 1. 한 장 요약

- **지금**: 가스 농도가 미리 정해둔 임계치(예: CO 30ppm)를 넘으면 알람을 보냄
- **문제**: "임계치 안 넘었지만 평소와 다른" 사전 신호를 못 잡음
- **제안**: AI 이상탐지를 **임계치 알람 위에 추가**해서 사전 신호도 잡자
- **범위**: 가스 도메인만, skill 문서 STEP 1~4 + multivariate IF 한 가지
- **빼는 것**: ARIMA(STEP 5), 전력 도메인, Docker 도입은 다음 스프린트
- **기간**: 영업일 13~20일 (1달 안에 버퍼 포함)
- **원칙**: 기존 임계치 알람은 한 줄도 안 건드림. AI가 망가져도 기존 시스템은 그대로

---

## 2. 지금 시스템이 어떻게 움직이고 있나

```
[센서] → [fastapi: 임계치 검사] → 위험이면 [Celery: 알람 task] → [브라우저 실시간 표시]
                              → 안전이면 그냥 DB 저장
```

알람 판정 기준은 **임계치 하나뿐**입니다.

> 비유: **체온계가 38도 넘으면 알림이 온다.** 명확하지만, 37.5도가 며칠째 이어지는 건 못 잡습니다.

이 "사전 신호"를 잡는 게 AI 이상탐지의 목적입니다.

---

## 3. AI 이상탐지의 핵심 발상

> **"평소가 어떤지를 시스템이 학습하고, 평소와 다르면 알린다"**

| 비교 | 임계치 알람 | AI 이상탐지 |
|---|---|---|
| 기준 | 절대값 (CO 30ppm 넘었나?) | 상대값 (평소보다 이상한가?) |
| 강점 | 명확, 규제·책임 명료 | 사전 신호, 미세 변화 감지 |
| 약점 | 사전 신호 못 잡음 | 정답 라벨이 없어 평가 어려움 |

**둘은 대체 관계가 아니라 보완 관계**입니다. AI는 임계치 알람을 절대 대체하지 않고, 그 위에 **얇은 감지 레이어 하나**를 더 얹는 것입니다.

---

## 4. skill 문서의 5단계 — 일상 비유로 이해하기

| 단계 | 이름 | 비유 | 잡는 것 |
|---|---|---|---|
| 1 | Z-score | "방금 값이 평소 30초보다 훨씬 다르다" | 갑자기 튀는 값 |
| 2 | Sliding Window 상태머신 | "잠깐 튀는 건 무시, 3초 이상 계속되면 알람" | 노이즈 vs 진짜 이상 구분 |
| 3 | Isolation Forest (가스 1종마다) | "CO 한 가지가 평소 학습된 패턴과 어긋난다" | 한 가스의 복합 이상 |
| 3+ | **Multivariate IF (추가 제안)** | "**9가지 가스가 함께** 이상하게 움직인다" | **가스 간 동시 이상 (화재 같은 패턴)** |
| 4 | Change Point | "분위기 자체가 바뀐 순간" | baseline 이동 (환기 멈춤, 누출 시작) |
| 5 | ARIMA (이번엔 빼는 것) | "지금 추세 보면 3분 뒤 위험할 듯" | 미래 예측 |

**중요한 점**: 단계마다 잡는 anomaly가 다릅니다. 한 단계가 다른 단계를 대체하지 않고 **각자 다른 종류**를 잡습니다.

---

## 5. 각 단계가 어떻게 동작하는지 (쉬운 버전)

### 5-1. STEP 1 — Z-score

평균에서 얼마나 멀리 떨어졌는지를 숫자로 잰 값입니다.

| 측정 | 값 |
|---|---|
| 최근 30초 평균 CO | 2.0 ppm |
| 최근 30초 표준편차 | 0.3 ppm |
| 방금 측정값 | 5.0 ppm |
| Z-score | (5.0 − 2.0) / 0.3 = **10** |
| 판정 | 정상 범위(±3) 초과 → 이상 |

규칙: 절대값 3.0을 넘으면 이상.

### 5-2. STEP 2 — Sliding Window 상태머신

STEP 1만 쓰면 잠깐 튀는 노이즈도 다 알람이 됩니다. 이걸 거르려고 **상태**를 둡니다.

```
NORMAL (평소) ──[3초 연속 이상]──▶ ALERT (이상 진행 중)
   ▲                                    │
   └─────────── [정상 복귀] ─────────────┘
```

상태가 바뀌는 순간만 알람으로 처리하므로 **알람 폭주가 막힙니다**.

### 5-3. STEP 3 — Isolation Forest (가스 1종마다)

기계학습 모델입니다. **"평소 패턴"을 학습**시킵니다.

- 학습 입력: 지난 14일간 정상으로 분류된 데이터
- 모델이 학습하는 것: 4가지 특징의 조합 — `[현재값, 30초 평균, 30초 표준편차, 직전과의 차이]`
- 추론: 새 데이터가 학습된 정상 영역과 얼마나 다른지 점수로 출력

Z-score보다 정교한 이유: Z-score는 평균과의 거리만 보지만, IF는 **값 + 변동성 + 변화율을 종합한 패턴**을 봅니다.

### 5-4. STEP 3+ — Multivariate IF (추가 제안)

skill 원안의 IF는 가스 1종마다 별도 모델입니다 (CO 모델, CO₂ 모델, …).

**문제**: 화재가 나면 CO + CO₂ + VOC가 **동시에** 오릅니다. 각각은 임계치 이내일 수 있어도, **함께 오르는 조합 자체가 위험 신호**인 경우가 있습니다. 단일 가스 모델은 이걸 못 잡습니다.

**해결**: 9가지 가스를 한꺼번에 입력으로 받는 모델을 **하나 더** 추가합니다.

- 입력: 9가지 가스 농도를 한 벡터로 `[co, h2s, co2, o2, no2, so2, o3, nh3, voc]`
- 학습: 평소 9가스가 어떤 조합으로 움직이는지 학습
- 추론: 새 조합이 학습 영역과 다르면 anomaly

**작업량**: 단일 IF 인프라가 만들어지면, 그 위에 1~2일이면 추가됩니다.

### 5-5. STEP 4 — Change Point

"한 점이 튀는" 게 아니라 **전체 분위기가 바뀐 시점**을 잡습니다.

| 시점 | 평균 |
|---|---|
| 직전 30초 | 2.0 ppm |
| 직후 30초 | 2.8 ppm |

Z-score는 못 잡습니다 (한 점씩 보면 다 정상 범위). Change Point는 **평균 자체가 이동**했음을 감지합니다.

용도: 환기 작동/정지, 가스 누출 시작/종료, 장비 ON/OFF 같은 **상태 전이** 시점.

### 5-6. STEP 5 — ARIMA (이번에는 뺍니다)

지금 추세를 보고 미래 값을 예측합니다 ("3분 뒤 50ppm 도달").

**왜 빼는가**:
- 가스 데이터는 평소 거의 안정적 → ARIMA 학습이 불안정해질 수 있음
- 다운샘플링·정상성 검증·예측 결과 활용까지 작업이 큼 (5~7일)
- 1달 안에 효과 검증까지 끝낼 시간이 부족
- → **다음 스프린트로 분리**가 안전

## 6. 우리가 선택한 범위와 이유

### 선택: skill 원안 STEP 1~4 + Multivariate IF (가스만)

| 단계 | 작업량 | 잡는 것 |
|---|---|---|
| 1. Z-score | 3~5일 | 갑자기 튀는 값 |
| 2. Sliding Window | 2~3일 | 노이즈 거름 + 지속성 판정 |
| 3. 단일 metric IF | 5~7일 | 한 가스의 복합 이상 |
| 3+. **Multivariate IF** | **1~2일** | **가스 간 동시 이상** |
| 4. Change Point | 2~3일 | 분위기 변화 |
| **합계** | **13~20일** | (1달 안에 버퍼 포함) |

### 제외한 것

- **ARIMA** — 효과 검증 시간 부족, 다음 스프린트
- **전력 도메인** — 가스 패턴을 그대로 복제하는 다음 스프린트
- **Docker 도입** — AI 가치 입증 후 별도 스프린트

### 왜 가스부터인가

1. **인명 직결** — 우선순위 자명
2. **데이터 양식이 이미 적합** — 가스 한 행에 9 metric이 같이 저장됨 → multivariate IF 입력 즉시 가능
3. **풀 스택 4층 방어를 한 도메인으로 완성**하면 다음 스프린트가 "전력에 복제"라 명확하고 빠름

## 8. 알람이 사용자에게 어떻게 보이는가

| 알람 종류 | 색상 (예) | 트리거 |
|---|---|---|
| DANGER | 빨강 | 임계치 위험 도달 |
| WARNING | 노랑 | 임계치 주의 도달 |
| **ANOMALY (신규)** | 보라 | AI가 잡은 이상 |

**원칙**: 임계치 알람과 절대 충돌하지 않습니다. 임계치 도달 시 AI 추론은 건너뜁니다 (이중 알람 방지, CPU 절약).

## 9. 리스크와 대응

| 리스크 | 대응 |
|---|---|
| AI 알람 false positive 폭주 | dwell time(3초 연속), feature flag로 즉시 OFF 가능 |
| AI 평가가 어렵다 (정답 라벨 없음) | "임계치 알람과 같이 잡은 비율", "도메인 검토자가 유효 판정한 비율"로 정성 평가 |
| 기존 임계치 알람 회귀 | 임계치 코드는 한 줄도 안 건드림. AI는 NORMAL 분기에서만 동작 |
| 학습 데이터 품질 | risk_level='normal' 행만 학습. 부족하면 학습 스킵 |
| Multivariate IF 해석성 | 단일 IF가 동시에 잡은 metric을 함께 표시 → 어느 가스 조합이 문제인지 추적 |

## 11. 부록 — 자주 나올 질문

**Q. AI가 잘못 잡으면 어떻게 되나요?**
A. 기존 임계치 알람은 그대로 작동합니다. AI 알람은 "추가 정보"이고, feature flag로 언제든 끌 수 있습니다.

**Q. 학습 데이터는 어디서 가져오나요?**
A. 운영 중 쌓인 GasData에서 `risk_level='normal'`인 직전 14일치를 사용합니다. 외부 데이터 필요 없습니다.

**Q. 다른 가스 종류가 추가되면 모델을 다시 학습시켜야 하나요?**
A. 단일 metric IF는 종류만큼 모델이 늘어납니다. multivariate IF는 입력 차원이 바뀌므로 재학습이 필요합니다. **재학습은 야간 배치로 자동화**하므로 운영자가 직접 할 일은 없습니다.

**Q. 1달 끝나면 무엇을 보여줄 수 있나요?**
A. 가스 도메인에서 임계치 + Z-score + 상태머신 + 단일/멀티 IF + Change Point가 동시에 동작하는 시스템. 데모 시나리오 1~2개로 "임계치만 있을 때는 못 잡았던 이상을 AI가 잡았다"를 시연합니다.

**Q. 이 작업이 끝난 후 다음 스프린트는?**
A. 가스에서 만든 패턴을 **전력 도메인에 복제**합니다 (1~2주). Docker, ARIMA는 그 후 단계적으로.
