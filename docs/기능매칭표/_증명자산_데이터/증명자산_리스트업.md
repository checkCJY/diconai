## 4. JSON 채울 대상 (정밀) — ✅ **완료 (2026-06-05)**

사용자 노출 REST 중 `@extend_schema` 0 이던 곳은 둘뿐이었고, serializer 는 이미 존재 → `@extend_schema(request=, responses=)` 부착으로 완료.

| 기능ID | 뷰 (근거) | 부착 결과 | 검증 |
|---|---|---|---|
| CMM-01, CMM-02 | [reference/views/code_admin.py](../../../drf-server/apps/reference/views/code_admin.py) `APIView` ×5 (9메서드) | `@extend_schema` 9건 (risk_standard 패턴) | ✅ |
| THR-01, THR-02 | [facilities/views/threshold_admin.py](../../../drf-server/apps/facilities/views/threshold_admin.py) `APIView` ×5 (9메서드) | `@extend_schema` 9건 | ✅ |

**검증 (step 2 — `manage.py spectacular`)**: 두 모듈 18개 오퍼레이션 전부 `requestBody`(쓰기)·`responses`·`tag(Admin — CommonCode/Threshold)` 채워짐 확인. 두 파일발 schema 경고 0 (이전 "unable to guess serializer" 해소). bulk 엔드포인트는 `inline_serializer({ids:[int]} → {updated:int})` 로 표현.

> 참고: `monitoring/views/admin_views.py`(@extend_schema 0)는 **라우팅 미연결 + serializer 참조 0 = 레거시 추정** → 대상 아님(확인 필요).

**WS(WebSocket) 기능 증명 (2026-06-05) — 서버 푸시 로그로 대체**

WS(`/ws/...`)는 curl 불가 → 서버측 알람 푸시 로그로 증명. `_push_to_ws()`([alerts/tasks.py:122](../../../drf-server/apps/alerts/tasks.py#L122))가 알람 발생 시 `IntegrationLog(integration_type=transmit, target_system=DRF→FastAPI, description=alarm_type=...)` 기록.

- **증명 curl**: `GET /api/admin/integration-logs/?integration_type=transmit&keyword=alarm_type` (keyword=alarm_type 이 WS 푸시행만 분리 — FastAPI→DRF 수집행 제외)
- **실적**: DRF→FastAPI 푸시 **43,261건 (success 41,450 / failure 1,811 ≈ 95.8%)**. 증빙: [증빙/ws-push-evidence-2026-06-05.json](../증빙/ws-push-evidence-2026-06-05.json)
- **연결 진단**: 설정 `FASTAPI_INTERNAL_URL=http://fastapi:8001` 정상 / 라이브 연결 OK(404·405=앱 응답, 네트워크 정상) / `celery-worker-alarm` healthy → **연결 문제 아님**. transmit 이 상시 없는 건 "알람 발생 시에만" 기록되기 때문(failure 1,811 = FastAPI 일시 재기동 시 retry 흡수 설계).
- 대상 WS 기능: DSH-14/15 · MGS-01 · MPW-01 · WKR-02 · MEV-11 · RTM-14
