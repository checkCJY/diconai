## 1. 왜 PG 로 전환했나

### SQLite 한계 (전환 동기)

- **5.3GB 누적** (power_data 9.2M 행, 9일치) — 시연일까지 약 30M 도달 예상. SQLite 단일 파일 락 한계.
- **다중 writer 충돌** — Celery worker 2 + DRF 4 thread + 더미 송출이 동시 INSERT 시 `SQLITE_BUSY_SNAPSHOT` 폭주 사례 (2026-05-14).
- **시계열 인덱싱 부족** — ARIMA 학습 쿼리가 GB 단위 데이터에서 분 단위로 느려짐.
- **FK 강제 부재** — 깨진 FK 가 계속 쌓임 (실제 PG 전환 시 발견됨, §5).

### PG 가 해결하는 것

- **multi-writer OK** — Celery + DRF + 더미 동시 INSERT 정상 처리.
- **FK 강제** — 부정합 데이터 사전 차단.
- **jsonb 타입** — `AlertPolicy.target_user_types` 등 JSON 컬럼이 indexable (GIN).
- **시계열 인덱싱** — BRIN (저비용 시계열) + TimescaleDB hypertable (시연 후 도입 후보).
- **statement_timeout** — 장시간 쿼리 자동 abort, 시스템 안정성.

### 옵션 (a) vs (b) 중 (b) 선택한 이유

원래 (a) "retention 1회 실행으로 데이터 축소 후 이관" 계획이었으나, retention dry_run 측정 결과 **삭제 예정 0건** (현재 데이터 분포 모두 30일 보존 윈도우 안). → (b) "SQLite 백업 후 Raw 이관 제외, PG 에 새로 쌓기" 로 확정.

자세한 의사결정 근거: [docs/plan/postgresql-polished-wren.md](../plan/postgresql-polished-wren.md).

## 2. PostgreSQL 기초 (PG 처음인 팀원용)

이 섹션은 PG 처음 만나는 펌웨어/디자인 멤버를 위한 입문. 백엔드는 §3 부터 봐도 OK.

### 2.1 SQLite vs PostgreSQL 핵심 차이 5가지

| 항목 | SQLite | PostgreSQL |
|---|---|---|
| **저장 형식** | 단일 파일 `db.sqlite3` | 서버 프로세스 + 데이터 디렉토리 |
| **FK 강제** | off (default) → 깨진 FK 통과 | always on → 부정합 즉시 ERROR |
| **PK auto** | `AUTOINCREMENT` (rowid 기반, 자동 advance) | `SEQUENCE` (별도 객체, 명시 INSERT 후 수동 reset 필요) |
| **JSON 타입** | TEXT + JSON_VALID 함수 (인덱스 X) | `jsonb` 네이티브 (GIN 인덱스 가능) |
| **NaN/Inf** | 허용 (FLOAT 그대로 저장) | `double precision` 은 NaN OK 하나 INSERT 시 일부 어댑터가 거부. PG INSERT 실패 가능 |

### 2.2 우리가 쓰는 PG 구성

```yaml
# docker-compose.yml (요약)
postgres:
  image: postgres:16-alpine
  env_file: ./.env.docker  # POSTGRES_DB/USER/PASSWORD 주입
  command:
    - postgres
    - -c shared_buffers=256MB
    - -c statement_timeout=60s
    - -c max_connections=50
  volumes:
    - postgres_data:/var/lib/postgresql/data  # docker named volume
```

- **이미지:** 16-alpine (가벼움, 시연 환경 적합)
- **DB/User:** `diconai` / `diconai`
- **비밀번호:** `secrets.token_urlsafe(24)` 평문 (gitignored `.env.docker`)
- **포트:** host 노출 안 함 (컨테이너 간 통신만)

### 2.3 자주 쓰는 psql 명령 6개

```bash
# 컨테이너 안 psql 진입
docker compose exec postgres psql -U diconai -d diconai

# 자주 쓰는 명령 (psql 안에서)
\dt                              # 테이블 목록
\d alarm_record                  # 테이블 컬럼·인덱스·FK 상세
\l                               # 데이터베이스 목록
SELECT pg_size_pretty(pg_database_size('diconai'));   # DB 사이즈
\du                              # 사용자 목록
\q                               # 종료
```

한 줄로 쿼리:
```bash
docker compose exec postgres psql -U diconai -d diconai -c "SELECT COUNT(*) FROM alarm_record;"
```

### 2.4 "SQLite 파일 cat 대신 어떻게 보나?"

```bash
# 이전 (SQLite)
sqlite3 drf-server/db.sqlite3 "SELECT * FROM custom_user;"

# 이후 (PG)
docker compose exec postgres psql -U diconai -d diconai -c "SELECT * FROM custom_user;"

# 또는 컨테이너 안 진입 후
docker compose exec postgres psql -U diconai -d diconai
diconai=# SELECT * FROM custom_user;
diconai=# \q
```

## 3. Before / After 변경 사항

### 3.1 `docker-compose.yml`

**추가된 것:**
- `postgres` 서비스 (16-alpine + shm_size 256mb + command 7개 -c 설정 + healthcheck `pg_isready`)
- `postgres_data` named volume (data 영속화)
- `depends_on` 에 postgres healthy 조건 — drf / celery-worker / celery-beat 3곳

**제거된 것:**
- 각 서비스의 `DATABASE_URL: sqlite:////app/db.sqlite3` 환경변수 (env_file 의 PG 변수로 통일)

**잔존 (롤백 안전망):**
- `./drf-server/db.sqlite3:/app/db.sqlite3` 볼륨 마운트 2곳 — PG 사용 시 무해, 비상 시 SQLite 복귀 위한 경로 유지

### 3.2 `.env.docker`

**추가:**
```bash
# ── PostgreSQL (2026-05-22 W2 D-Day 전환) ────────────────────
POSTGRES_DB=diconai
POSTGRES_USER=diconai
POSTGRES_PASSWORD=<secrets.token_urlsafe(24) 생성값>
POSTGRES_HOST=postgres
POSTGRES_PORT=5432
```

**제거:** `DATABASE_URL` (직접 두지 않음 — settings.py 가 분리 변수로 합성)

### 3.3 `drf-server/config/settings.py`

**Before:**
```python
DATABASES = {
    "default": env.db(
        "DATABASE_URL",
        default=f"sqlite:///{BASE_DIR / 'db.sqlite3'}",
    )
}
```

**After:**
```python
if env("POSTGRES_HOST", default=""):
    DATABASES = {
        "default": {
            "ENGINE": "django.db.backends.postgresql",
            "NAME": env("POSTGRES_DB"),
            "USER": env("POSTGRES_USER"),
            "PASSWORD": env("POSTGRES_PASSWORD"),
            "HOST": env("POSTGRES_HOST"),
            "PORT": env("POSTGRES_PORT", default="5432"),
        }
    }
else:
    DATABASES = {
        "default": {
            "ENGINE": "django.db.backends.sqlite3",
            "NAME": str(BASE_DIR / "db.sqlite3"),
        }
    }
```

**키 포인트:** `POSTGRES_HOST` 가 시그널. 비우면 자동 SQLite 폴백 (비상시 또는 신규 환경).

### 3.4 `.gitignore`

```diff

+ # DB backups & dumps (PG 전환 + 일상 백업)
```

### 3.5 데이터 위치

| 시점 | 위치 |
|---|---|
| Before | `./drf-server/db.sqlite3` (host 파일, 5.3GB) |

| 3 | **sequence reset SQL 의 string PK 테이블 ERROR** — `django_session` 처럼 PK 가 id 아닌 테이블에서 `pg_get_serial_sequence(..., 'id')` 호출 자체가 ERROR | 다른 sequence reset 은 정상, 미세한 경고만 | inner BEGIN/EXCEPTION 블록 |

### 6.2 자주 발생할 트러블 (시연 중 만나면 대응)

#### A. `IntegrityError: duplicate key value violates unique constraint "..._pkey"`

**의미:** PG sequence 가 데이터의 max(id) 보다 작은 값 반환 → INSERT 충돌. 보통 seed migration 또는 loaddata 직후.

**조치:**
```sql
-- 모든 application 테이블 sequence 를 max(id)+1 로 reset
DO $$
DECLARE r RECORD;
DECLARE seq TEXT;
BEGIN
  FOR r IN SELECT tablename FROM pg_tables WHERE schemaname='public' LOOP
    BEGIN
      seq := pg_get_serial_sequence(format('public.%I', r.tablename), 'id');
      IF seq IS NOT NULL THEN
        EXECUTE format(
          'SELECT setval(%L, COALESCE((SELECT MAX(id) FROM %I), 1), (SELECT MAX(id) IS NOT NULL FROM %I))',
          seq, r.tablename, r.tablename
        );
      END IF;
    EXCEPTION WHEN OTHERS THEN NULL;
    END;
  END LOOP;
END $$;
```

#### B. `ForeignKeyViolation: insert or update on table "..." violates foreign key constraint`

**의미:** SQLite 시절 부정합 데이터가 PG FK 강제로 노출. 예: 더미가 사용하는 worker_id 가 custom_user 에 없음.

**조치:**
1. 어느 FK 가 깨졌는지 에러 메시지 확인 (`alarm_record_worker_id_fk_...`)
2. 부모 테이블에 해당 id 가 있는지 확인 (`SELECT id FROM custom_user WHERE id = <값>`)
3. 없으면 → 더미·코드 수정 (hardcoded id 대신 자연키 lookup) 또는 부모 row 추가
4. 자주 발생 패턴: [산출물 §1.9](../plan/postgres-migration-code-review.md)

#### C. `DeserializationError: Problem installing fixture` (loaddata 실패)

**의미:** JSON 파일 자체가 invalid (예: entrypoint 로그가 stdout 으로 섞임).

**조치:**
```bash
# JSON 첫 줄이 정확히 "[" 인지 확인
head -1 backup/<file>.json

# 헤더가 오염되어 있으면 첫 "[" 부터 추출
awk '/^\[$/{found=1} found' backup/<file>.json > backup/<file>_clean.json

# python 으로 검증
python3 -c "import json; data=json.load(open('backup/<file>_clean.json')); print(f'OK {len(data)} obj')"
```

#### D. "내 컨테이너에 PG 안 떴다 / 옛 데이터로 작동한다"

**진단 1단계:** ENGINE 확인
```bash
docker compose exec drf python -c "from django.db import connection; print(connection.settings_dict)"
# ENGINE=postgresql, HOST=postgres 인지 확인
```

**`sqlite3` 라고 나오면:**
- `.env.docker` 의 `POSTGRES_HOST=postgres` 확인 (값 비어있으면 SQLite 폴백)
- `docker compose down && docker compose up -d` (환경변수는 컨테이너 시작 시점에만 읽힘)

| **BRIN 인덱스** | Raw 시계열 (`measured_at` 기준) 저비용 인덱스 | TimescaleDB 와 함께 |
| **pg_stat_statements + postgres_exporter** | 쿼리별 latency Prometheus 메트릭, slow query 추적 | 시연 후 W+2 |
| **pg_dump cron / WAL 아카이빙** | 운영 진입 시 필수 | 운영 진입 전제 |
| **Read replica** | 분석·리포트 트래픽 분리 | 사용자 증가 시 |
