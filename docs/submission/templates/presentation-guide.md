### 발표 가이드
```
※ 이번 발표의 가장 중요한 평가 기준은 기능 개수가 아니라
데이터 흐름이 끊기지 않고 이어지는지입니다.

센서 데이터 → 저장 → AI 분석 → 결과 저장 → 알람 → 모니터링
이 흐름이 한 번이라도 완전히 동작해야 합니다.
```

발표의 목적
```
첫째, 3차에서 만든 관제 화면과 데이터 흐름을 실제 운영 구조로 확장했는가
둘째, AI 기반 이상 탐지와 모니터링으로 위험을 사전에 감지할 수 있는가
```

이 프로젝트는 산업 현장에서 발생할 수 있는 유해가스, 전력 이상, 작업자 위치 위험 등을 통합하여 운영자가 한 화면에서 파악하고 대응할 수 있는 관제 플랫폼입니다.

4차 발표의 핵심은 단순히 화면이 보이는 수준을 넘어서, 센서 데이터가 수집되고, Redis를 통해 안정적으로 전달되며, Django 관제 서버에서 저장/판단되고, Celery를 통해 AI 분석이 비동기로 실행되며, Prometheus와 Grafana로 시스템 상태를 확인할 수 있는 **운영 가능한 구조**를 보여주는 것입니다.

전체 전략은 다음처럼 설명할 수 있습니다.

```
1단계 = 보여주는 것 (UI + 데이터 흐름)
2단계 = 살아있는 시스템 (AI + 모니터링 + 운영)
```

따라서 4차 발표의 핵심은 아래 문장으로 정리할 수 있습니다.

```
3차 프로젝트: 보이게 만들어라
4차 프로젝트: 제대로 돌아가게 만들어라
```

## 발표에서 가장 중요하게 봐야 하는 기준

이번 발표에서는 아래 5가지를 중심으로 설명하는 것이 좋습니다.

### (1) 이 관제 플랫폼이 무엇을 해결하는지

- 산업 현장에서 어떤 위험을 감지하려는지
- 유해가스, 전력 이상, 작업자 위치 위험이 왜 통합 관리되어야 하는지
- 단순 모니터링이 아니라 사전 감지/예측이 왜 중요한지
- 운영자가 어떤 상황에서 이 시스템을 사용하게 되는지

---

### (2) 3차에서 4차로 무엇이 확장되었는지

3차는 UI와 실시간처럼 보이는 데이터 흐름, 지오펜스 판단, 알람 표시가 핵심이었습니다.

4차는 여기에 다음 구조를 추가합니다.

- 실제 운영 구조
- AI 이상 탐지
- Redis 기반 실시간 처리 안정화
- Celery 비동기 처리
- 알림 고도화
- Docker 기반 서비스 분리
- Prometheus / Grafana 모니터링
- Kubernetes 확장 구조 (선택 또는 설계 수준 설명 가능)

즉, 4차 발표에서는 아래 흐름을 명확히 보여줘야 합니다.

```
센서 데이터 → Redis → Django 저장/판단 → Celery 비동기 작업 → FastAPI AI 분석 → 결과 저장 → 알람/대시보드 출력 → Prometheus/Grafana 모니터링
```

※ Kubernetes는 일부 적용만으로도 충분합니다.

---

### (3) AI 이상 탐지가 실제 서비스 흐름에 연결되는지

AI 기능은 단순히 모델을 실행하는 것이 아니라 관제 서비스 안에서 위험 판단을 보조하는 구조로 동작해야 합니다.

반드시 설명할 내용:

- 어떤 데이터를 AI 분석 대상으로 삼았는지
- ARIMA 또는 이상 탐지 모델이 어떤 역할을 하는지
- 정상 / 이상 판단 결과가 어떻게 저장되는지
- AI 결과가 대시보드 또는 알람 흐름에 어떻게 반영되는지
- AI 결과가 단순 출력이 아니라 DB에 기록되는지

핵심 문장:

```
AI 기능 = 모델 실행이 아니라 위험 판단 흐름에 연결된 분석 결과
```

---

### (4) 비동기 처리와 운영 구조를 이해하고 있는지

AI 분석, 대량 센서 처리, 알림 처리는 요청 즉시 처리하면 서버가 느려질 수 있습니다. 따라서 4차에서는 Celery + Redis 기반 비동기 처리를 보여줘야 합니다.

반드시 설명할 내용:

- Redis의 역할
- Celery의 역할
- Django와 Celery가 어떻게 연결되는지
- 작업 상태가 어떻게 관리되는지
- 실패 시 어떤 상태로 저장되는지

기본 상태값 예시:

```
PENDING     → 요청됨
PROCESSING  → 처리중
SUCCESS     → 완료
FAIL        → 실패
```

핵심 문장:

```
비동기 처리 = 사용자 응답과 무거운 처리를 분리하는 구조
```

---

### (5) 어디가 느린지 / 어디가 문제인지 볼 수 있는지

4차의 중요한 차이는 운영 가능성입니다.

따라서 Prometheus와 Grafana를 통해 아래 지표를 확인할 수 있어야 합니다.

- 요청 수
- 응답 시간
- 에러율
- WebSocket 연결 상태 또는 데이터 처리량
- AI 처리 시간
- Celery task 처리 시간
- 알람 발생 수
- DB 저장 시간 또는 지오펜스 판단 시간

핵심 문장:

```
모니터링의 목적은 단순 그래프가 아니라 병목 지점 확인입니다.
```

## 시스템 구조 설명

4차 프로젝트는 다음 구조를 기준으로 설명합니다.

```
FastAPI (센서)
   ↓
Redis (스트림 / PubSub / 브로커)
   ↓
Django + DRF (관제 서버)
   ↓
Celery (비동기 처리)
   ↓
FastAPI (AI 분석)
   ↓
DB 저장 / 알람 처리
   ↓
Prometheus → Grafana
   ↓
Docker Compose / Kubernetes
```

또는 발표에서는 아래처럼 단순화해서 설명해도 됩니다.

```
[센서 데이터 수집] → [실시간 처리 안정화] → [관제 서버 저장/판단]
→ [AI 비동기 분석] → [알람/대시보드 반영] → [운영 모니터링]
```

## C. 발표 감점 기준

| 감점 항목                                         |       감점 |

| 역할 설명 없이 기술 이름만 나열                            |       -3 |
| 발표 시간 초과 또는 발표 흐름 붕괴                          |  -2 ~ -5 |
| Celery worker 또는 Redis 연결 문제로 비동기 흐름이 동작하지 않음 |       -5 |

## D. 가산점 항목 (최대 +20점)

| 선택 항목 | 가산점 |
|---|--:|
| ARIMA 외 Isolation Forest, Z-score, Change Point 등 이상 탐지 조합 구현 | +3 |
| AI 결과 기반 위험 점수 또는 예측 위험도 표시 | +3 |
| WebSocket 실시간 알림 안정화 또는 재연결 처리 | +2 |
| Discord / Slack Webhook 외부 알림 구현 | +2 |
| 커스텀 Prometheus 메트릭 추가 | +3 |
| Grafana 대시보드에 AI 처리 시간 / 알람 수 / task 처리 시간 포함 | +3 |
| k6 등 부하 테스트 결과와 병목 분석 포함 | +3 |
| GitHub Actions 기반 CI 또는 CI/CD 구성 | +3 |
| Kubernetes Deployment / Service / Ingress 적용 | +4 |
| ConfigMap / Secret 기반 환경 설정 분리 | +2 |
| HPA 자동 확장 또는 장애 복구 시연 | +3 |
| 트러블슈팅 사례를 로그 + 메트릭 기반으로 정리 | +2 |

가산점 적용 규칙:

- 가산점은 구현 + 시연 + 설명이 모두 확인된 경우만 인정합니다.
- 코드만 존재하거나 발표에서 보여주지 못하면 인정하지 않습니다.
- 가산점 총합은 최대 +20점까지만 반영합니다.
- 필수 구조가 부족한 경우 가산점이 기본점수를 대체하지 않습니다.

## E. 최종 점수 계산 방식

```
기본점수: 100점 만점
가산점: 최대 +20점 별도 표기
최종 표기: 기본점수 / 100, 가산점 / +20
```

예시:

```
기본점수 86 / 100
가산점 +12
총평: AI + 비동기 + 모니터링 구조가 안정적으로 연결됨
```

## F. 채점 총평 예시 기준

| 구간 | 평가 |
|---|---|
| 90점 이상 | AI, 비동기, 운영 모니터링 구조가 매우 안정적으로 구현됨 |
| 80~89점 | 핵심 구조는 완성되었고 일부 운영/설명 보완 필요 |
| 70~79점 | 기본 구현은 되었으나 AI/비동기/모니터링 중 일부 연결 보완 필요 |
| 60~69점 | 일부 핵심 기능은 있으나 운영 가능한 구조로 보기에는 부족함 |
| 59점 이하 | 4차 필수 흐름인 AI + 비동기 + 운영 구조 확인이 어려움 |

## G. 한 줄 채점 기준 정리

```
4차 발표는 화면이 보이는가가 아니라,
AI 분석 + 비동기 처리 + 모니터링 + 배포 구조가 실제 운영 흐름으로 연결되었는가를 평가합니다.
```


※ 평가 시 가장 우선적으로 확인하는 것은 “데이터 흐름 시연”입니다.
각 기능이 개별적으로 존재하더라도, 하나의 흐름으로 연결되지 않으면 높은 점수를 받을 수 없습니다.
