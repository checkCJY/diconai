## ① `entry is None` 낙하 버그 — `_save_gas_data` 호출 제거 + 추론 블록 `else:` 묶음

### 현재 상태 (코드 안 보고 이해)

가스 AI 모델이 미등록 상태일 때 코드가 "모델 없음을 메트릭에 기록하고 멈춰야" 하는데,
**존재하지 않는 함수 `_save_gas_data`를 호출**하고 있어요.
이 함수는 코드베이스 어디에도 정의되어 있지 않아서 `NameError`가 발생하지만,
바깥 `except Exception` 블록이 잡아서 DRF 저장 흐름 자체는 살아남는 상태.
→ **결과적으로 가스 데이터가 들어올 때마다 추론 단계에서 `NameError` 로그가 찍히고 있어요.**

진단 통합본의 "AttributeError 발생 → 흐름 정지"는 부정확. 실제로는 NameError + except 흡수.

### 위치

[fastapi-server/gas/services/gas_service.py:92-171](../../gas/services/gas_service.py#L92-L171)

### 현재 코드

```python
if len(_co_window) >= 30:
    try:
        entry = await _get_or_load("gas")
        if entry is None:
            AI_INFERENCE_FAILED_TOTAL.labels("gas_if", "model_not_loaded").inc()
            return await _save_gas_data(payload, individual_risks)  # ← 미정의 함수
        # P2 전 — IF 추론 실행 시간 측정
        _infer_start = time.time()
        row = _build_multi_feature_row(...)
        pred = int(entry.model.predict(row)[0])
        ...
    except Exception as e:
        AI_INFERENCE_FAILED_TOTAL.labels("gas_if", "inference_error").inc()
        logger.error(f"[AI 추론 실패] {e}")
```

### 수정 방향

1. `_save_gas_data` 호출 라인 자체를 **제거**.
2. 추론 블록(row 빌드 ~ asyncio.create_task) 전체를 `else:` 분기로 묶어 `entry is None`일 때는 추론을 스킵하고 함수 흐름이 자연스럽게 DRF 저장(line 173~) 단계로 내려가게 합니다.

### 수정 후 코드 (예시)

```python

            if pred == -1:

                ...
```

### 검증 방법

- `apps/ml/` 에 가스 모델 미등록 상태에서 `/gas/data` POST → 로그에 `NameError` 사라지고 `model_not_loaded` 메트릭만 증가

## ⑥ `ingress_ts` 위치 — 핸들러 첫 줄로 이동

### 현재 상태 (코드 안 보고 이해)

E2E 레이턴시(IoT 장비에서 데이터 도착 → 브라우저 전달까지)를 측정하려면 `ingress_ts`를 **요청 도착 직후**에 찍어야 합니다.
지금 가스 쪽은 AI 추론(슬라이딩 윈도우 30개 모이면 IF + ARIMA 계산)을 다 끝낸 후, DRF POST 직전에 `ingress_ts`를 찍고 있어요.
→ 측정값에서 AI 추론 시간(수십~수백 ms)이 빠짐 → **Grafana E2E 대시보드가 실제보다 짧게 표시**.

### 위치

[fastapi-server/gas/services/gas_service.py:61-72](../../gas/services/gas_service.py#L61-L72) (핸들러 첫 줄)
[fastapi-server/gas/services/gas_service.py:176](../../gas/services/gas_service.py#L176) (현재 ingress_ts 위치)

### 수정 방향

1. `process_gas_data()` 진입 직후(line 72 직후)에 `ingress_ts = time.time()` 변수로 캡처.
2. 기존 `SENSOR_LAST_RECEIVED.labels(...).set(time.time())`도 `set(ingress_ts)`로 일치시켜 한 번의 `time.time()` 호출을 공유.
3. `drf_payload["ingress_ts"] = time.time()` (line 176)을 `ingress_ts` 변수로 교체.

### 수정 후 코드 (예시)

```python
async def process_gas_data(payload: GasDataPayload) -> dict:
    """..."""
    # E2E 측정 시작점 — AI 추론(슬라이딩 윈도우)보다 먼저 잡아야 SLO 정확.
    ingress_ts = time.time()
    SENSOR_LAST_RECEIVED.labels("gas", payload.device_id).set(ingress_ts)
    # ... (기존 흐름)
    drf_payload = {
        ...
        "ingress_ts": ingress_ts,  # line 176 교체
        ...
    }
```

### 참고 — 전력 쪽 패턴

이번 PR에서 [power_router.py](../../power/routers/power_router.py)의 `recv_current` / `recv_voltage` / `recv_watt` 3개 엔드포인트에 동일 패턴을 이미 적용했어요. 같은 코드 스타일로 맞추면 됩니다.

### 검증 방법

- Prometheus 쿼리: `histogram_quantile(0.95, sum by (le) (rate(e2e_latency_seconds_bucket{sensor_type="gas"}[5m])))`
- 수정 전후 P95 값이 AI 추론 소요 시간(보통 30~80ms)만큼 증가하면 정상

## ⑦ AI 알람 `push_payload`에 `ingress_ts` 누락

### 현재 상태

룰 기반 알람(임계치 초과)은 `ingress_ts`를 전달해 `alarm_flush_loop`이 E2E를 측정하는데,
**AI 기반 알람의 `push_payload`에는 `ingress_ts` 필드 자체가 없어**서 측정이 누락됩니다.
→ Grafana에서 AI 알람의 E2E 속도를 전혀 파악 못 하는 상태.

### 위치

[fastapi-server/gas/services/gas_service.py:157-165](../../gas/services/gas_service.py#L157-L165)

### 현재 코드

```python
push_payload={
    "alarm_type": "gas_anomaly_ai",
    "risk_level": "danger",
    "source_label": "가스센서 AI 이상탐지",
    "summary": f"가스 이상 감지 (AI) | CO:{payload.co} H2S:{payload.h2s} CO2:{payload.co2}",
    "is_new_event": True,
    "gas_type": "co",
    "measured_value": payload.co,
},
```

### 수정 방향

위 ⑥에서 캡처한 `ingress_ts` 변수를 `push_payload`에 추가.

### 수정 후 코드 (예시)

```python
push_payload={
    "alarm_type": "gas_anomaly_ai",
    "risk_level": "danger",
    "source_label": "가스센서 AI 이상탐지",
    "summary": f"...",
    "is_new_event": True,
    "gas_type": "co",
    "measured_value": payload.co,
    "ingress_ts": ingress_ts,  # ← 추가 (⑥에서 캡처한 변수 재사용)
},
```

### 검증 방법

- `apps/ml/` 가스 모델 등록 + 이상 패턴 시뮬레이터 발화 → Grafana 알람 E2E 패널에서 `alarm_type="gas_anomaly_ai"` 시리즈 등장 확인

## 추가 발견 — 전력 AI 알람 `push_payload`도 동일 누락 (✅ 같은 PR 처리됨)

진단 통합본 ⑦은 가스만 짚었지만, 전력 AI 알람에도 같은 누락이 있어 **같은 PR에서 처리했습니다**:

- [power_service.py](../../power/services/power_service.py) — `process_anomaly_inference` 시그니처에 `ingress_ts: float | None = None` 추가, `push_payload`에 `"ingress_ts": ingress_ts` 키 추가
- [power_router.py](../../power/routers/power_router.py) `recv_watt` — `process_anomaly_inference(..., ingress_ts=ingress_ts)`로 전달

가스 ⑦ 작업 시 위 전력 패턴을 그대로 참고하면 됩니다 (한 가지 차이: 가스는 `push_payload`가 dict literal로 평면 구성, 전력은 `anomaly_meta` 중첩 dict 위에 평면 키).
