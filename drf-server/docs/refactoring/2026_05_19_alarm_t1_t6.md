## 1. 왜 본 작업인가

전력 AI 5축 정책 엔진 (Z-score / Change Point / Isolation Forest / ARIMA / 야간 가동) 까지 도입 완료된 시점, 운영자 UI 에는 여전히 ML 기술용어가 노출되고 있었음.

### Before — 운영자가 보던 토스트
```
┌────────────────────────────────────────────────────┐
│ ⚠ 위험                                              │
│ 압연기                                              │
│ [Z-score 이상 감지] 압연기 watt=3611.3             │
│   (score 0.0195, combined=predict_warn)            │
│ 19:07:02                                           │
└────────────────────────────────────────────────────┘
```

문제점:
- `score 0.0195, combined=predict_warn` — ML 내부 용어 노출. 운영자는 의미 모름
- `watt=3611.3` — 천단위 구분 없음
- 모든 AI 알람이 같은 "이상 감지" 텍스트라 알고리즘별 본질 구분 안 됨
- 토스트 / 모달 / 이벤트 패널 / 이력 페이지 텍스트 fallback chain 으로 불일치

### After
```
┌────────────────────────────────────────────────────┐
│ ⚠ 위험                                              │
│ 압연기                                              │
│ 압연기 통계 이상 수치 (3,611.3 W)                  │
│ 19:07:02                                           │
└────────────────────────────────────────────────────┘
```

해결:
- ML 용어 (`score`, `combined=`) 완전 제거
- 천단위 구분 (`3,611.3`)
- **algorithm_source 별 운영자 친화 워딩** — 칩 없이 텍스트로 알고리즘 본질 구분
- 4 위치 (토스트 / 모달 / 이벤트 패널 / 이력) `AlarmRecord.get_short_message()` 단일 진실 공급원
- 이벤트 패널 fallback: 영문 alarm_type 코드 노출 → 한국어 라벨

### algorithm_source 별 워딩 매핑 (운영자 검토 결과)

| algorithm_source | 워딩 | 근거 |
|---|---|---|
| `isolation_forest` | 이상 수치 탐지 | IF = 분포 outlier (수치 기반) |
| `arima` | 이상 패턴 탐지 | ARIMA = 시계열 예측 (패턴 기반) |
| `combined` | 이상 수치·패턴 동시 탐지 | IF+ARIMA 동시 발화 = 최고 신뢰도 |
| `zscore` | 통계 이상 수치 | sliding window 통계 (IF 와 구분) |
| `change_point` | 패턴 변화 탐지 | 명확한 변화점 (ARIMA 추세 이탈과 구분) |
| `night_abnormal` | 야간 이상 가동 | 운영 시간 외 baseline 초과 |
| (fallback) | AI 이상 탐지 | 미매핑 시 |
| 룰 (`power_overload`) | **정적 임계치 초과** | ML 동적 탐지와 구분 |

### 칩 도입 검토 후 제외 (2026-05-19 운영자 검토 결과)

초기 plan 은 algorithm_source (Z-score / 급변 / IF / ARIMA / IF+ARIMA / 야간 가동) 를 시각 칩으로 표시할 예정이었으나, 운영자 검토 시:

- "위험한 알람은 동일하게 대응" 이라는 운영 멘탈 모델에 칩이 추가 학습 부담 발생
- 화면 설계서에 칩이 정의되어 있지 않음
- 알고리즘 출처는 엔지니어 진단 영역이지 운영자 의사결정 영역 아님

→ **칩 도입 제외 결정.** 본문 텍스트 정리·인프라 일관성·fallback 라벨만 시연 전 적용. 5축 정책 엔진의 가치는 별도 시연 채널 (운영 데이터 누적 + Grafana 대시보드) 로 입증.

## 8. 4 레이어 dedup 설계 결정 (T3, 2026-05-19)

본 PR 에는 T3 (중복 방지·dedup 의미 통일) 도 포함됨. 4 레이어 dedup 메커니즘의 의도·범위·TTL 명시화 + EventAck 하이브리드 시그널 + RESOLVED dedup 분리.

### 8.1 4 레이어 dedup 정리

| # | 레이어 | 위치 | TTL | 의도 | 단위 |
|---|---|---|---|---|---|
| 1 | rate_limit | [power_service.py:64](../../fastapi-server/power/services/power_service.py#L64) | 60s | 같은 센서 알람 생성 빈도 제한 (operator UX) | sensor_identifier |
| 2 | push_dedup | [alarm_queue.py:40](../../fastapi-server/websocket/services/alarm_queue.py#L40) | **30s** | Celery retry burst 차단 (infrastructure) | fingerprint |
| 3 | repopup_cooldown | [config/settings.py:382](../../drf-server/config/settings.py#L382) | 60s (env) | 활성 Event 재팝업 cooldown (operator UX) | Event |
| 4 | client_dedup | [alarm-popup.js:60](../../drf-server/static/js/shared/alarm-popup.js#L60) | 60s | 브라우저 토스트 중복 차단 (operator UX) | event_id or composite |

#2 만 30s 인 이유: Celery retry max 3회 × 5s = 15s + buffer. 인프라 레이어라 운영자 UX 와 별개 의도.

### 8.2 TTL 통일 검토 → 보류 결정

30s 단일 통일 검토했으나 **60s 유지 결정** (2026-05-19):
- 폭주 회피 > 시연 가시성 (운영 중 토스트 폭주가 더 위험)
- 영향 파일 다수 (주석 갱신 ~8곳, 정책 충돌 검토 필요)
- 시연 시 다른 센서/시나리오 전환으로 시각 가시성 확보 가능

각 dedup 상수 위치의 주석에 "30s 하향 검토 후 60s 유지" 결정 명시.

### 8.3 EventAck vs dedup 분리 — 다중 관리자 하이브리드

| | EventAck | Dedup |
|---|---|---|
| 범위 | **사용자별** (event, user) 쌍 | 전역 시간 기반 (사용자 무관) |
| 의도 | "내가 확인했음" 운영 이력 기록 | 단순 중복 차단 |
| 영향 | 한 사용자 ack → 다른 사용자에겐 계속 팝업 (안전망) | TTL 내 모든 사용자 차단 |

서로 다른 의도라 **분리 유지**. 다만 다중 관리자 환경에서 "누가 처리 중" 시그널 부족 → 하이브리드 보강:

**push_payload 에 `event_ack_users: list[str]` 동봉 → 토스트/모달 본문에 "(N 확인 중)" 시각 표시**

- 0명: 미렌더 (단일 운영자 환경 영향 0)
- 1명: `(홍길동 확인 중)`
- 2명: `(홍길동, 김민수 확인 중)`
- 3명+: `(홍길동 외 2명 확인 중)`

→ dedup 분리 유지 (안전망) + 코디네이션 시그널 (작업 중복 회피).

### 8.4 RESOLVED dedup 분리

위험 알람이 60s 안에 자동 해소되는 경우, JS `_popupDedupKey` 가 같은 event_id 로 dedup 차단 → "위험 해소" 토스트 누락 risk.

해결: `event_resolved_at` 타임스탬프를 suffix 로 별도 key 처리.

```js
// alarm-popup.js _popupDedupKey()
const resolvedSuffix = data.event_resolved_at
  ? `:resolved:${data.event_resolved_at}` : '';
return `event:${eventId}${resolvedSuffix}`;
```

fastapi push_dedup ([alarm_queue.py:72](../../fastapi-server/websocket/services/alarm_queue.py#L72)) 은 이미 `event:{id}:resolved` suffix 분리되어 있어 백엔드 측은 OK.
